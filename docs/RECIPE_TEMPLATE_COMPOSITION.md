# Модуль композиции рецептов: шаблоны, фрагменты и конкретное оружие

## Точка входа по переработке рецептов

**Центр проработки (навигация и дорожная карта):** **`docs/CRAFT_LINE_RECIPE_TECHNIQUE_COMPOSITION.md`** — с него начинать обзор всей связки «рецепт → полоса → техники».

**Этот файл (`RECIPE_TEMPLATE_COMPOSITION.md`)** — углублённая спецификация **модуля сборки рецептов** из шаблонов и фрагментов (материализация в `WeaponRecipe`), плюс **приложение §14** (меч / кинжал / топор).

| Документ | Роль |
|----------|------|
| **`docs/CRAFT_LINE_RECIPE_TECHNIQUE_COMPOSITION.md`** | **Главный хаб:** Крафповая линия, хребет, насадки, разрыв с кодом, фазы внедрения, риски. |
| **`docs/RECIPE_TEMPLATE_COMPOSITION.md`** (здесь) | DSL шаблонов, фрагменты, `RecipeDefinition`, дельты, валидация. |
| **`docs/TZ_SWORD_RECIPE_AND_CRAFT_LINE.md`** | ТЗ контента пилота меча. |

Все три перечислены в **`docs/README.md`** (техническая документация).

---

**Статус:** проектная спецификация (согласована с **`docs/CRAFT_LINE_RECIPE_TECHNIQUE_COMPOSITION.md`**, пилот меча — **`docs/TZ_SWORD_RECIPE_AND_CRAFT_LINE.md`**).  
**Цель:** при большом числе рецептов не проектировать каждый процесс «с нуля», а **собирать** исполняемый рецепт из **повторно используемых частей**, подмешивая **уникальные параметры** конкретного оружия.  
**Источник правды в рантайме** по-прежнему должен оставаться **развёрнутый** `WeaponRecipe`-подобный объект (полный список `parts`, `stages` и т.д.), чтобы симулятор крафта, расчёт времени и Крафтовая линия не разъезжались.

## 1. Задачи и границы

### 1.1 Что решает модуль

- Единый **каркас процесса** для семейства оружия (например все одноручные мечи с четырьмя стандартными ролями частей).
- **Переиспользование** повторяющихся цепочек стадий (`proc_smelting` → `proc_rolling` для металлической части и т.д.).
- **Точечные отличия**: статы, уровень, редкость, имя, состав частей (число слоёв металла, опциональная часть), замена одной стадии, вставка квестовой вставки.
- Связь с **Крафтовой линией**: микроэтапы хребта могут вешаться на **шаблон** (общие для семейства) и **дельты** на уровне конкретного рецепта (см. раздел 8).

### 1.2 Что модуль не заменяет

- Выбор материалов игроком, план крафта, техники обработки и боевые приёмы — по-прежнему внешние слои.
- Реестр типов этапов (`StageTypeDefinition`) и `process-generator` — остаются каноном исполнения; шаблон только **порождает** `RecipeStageConfig[]`.
- Уникальный лорный рецепт «с нуля» для одного артефакта остаётся допустимым как **полностью явный** рецепт без шаблона (escape hatch).

### 1.3 Инвариант

После этапа **материализации** (развёртывания) система везде видит тот же контракт, что и сегодня: **`WeaponRecipe`** с полным **`stages`**. Любой код, который читает рецепты, **не обязан** знать про шаблоны.

## 2. Термины

| Термин | Определение |
|--------|-------------|
| **Шаблон процесса (RecipeProcessTemplate)** | Именованная схема: упорядоченный список **слотов стадий** и/или вставок **фрагментов**, параметризованных ролями частей (`blade`, `guard`, …). |
| **Фрагмент процесса (ProcessFragment)** | Короткая повторяемая цепочка стадий с плейсхолдерами (например «металл: плавка + прокат» для `{{part}}`). |
| **Роль части** | Логический слот в шаблоне: не обязан совпадать с `RecipePart.id` конкретного рецепта, но **маппится** на него при материализации. |
| **Определение рецепта (RecipeDefinition)** | Запись автора: ссылка на шаблон(ы), маппинг ролей → фактические `part.id`, дельты, уникальные поля (`baseStats`, `requiredLevel`, …). |
| **Материализация** | Детерминированная функция: `RecipeDefinition` → `WeaponRecipe` (полный `stages`, готовый к регистру рецептов). |
| **Дельта (RecipeOverride)** | Правка к результату шаблона: вставка, удаление, замена стадии, патч микроэтапов хребта, смена длительности. |
| **Ключ стадии (StageKey)** | Стабильный идентификатор **внутри материализованного рецепта** для привязки якорей и микроэтапов: например индекс + `stageType` + `target`, или явное поле `stageInstanceId` после материализации. |

---

## 3. Архитектура слоёв

```text
┌─────────────────────────────────────────────────────────────┐
│  Авторинг: FragmentRegistry + RecipeProcessTemplate        │
│              + RecipeDefinition (шаблон + роли + дельты)     │
└────────────────────────────┬────────────────────────────────┘
                             │ материализация (build / CI / старт)
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  Канон для игры: WeaponRecipe (parts, stages, baseStats, …) │
└────────────────────────────┬────────────────────────────────┘
                             │
         ┌───────────────────┼───────────────────┐
         ▼                   ▼                   ▼
   process-generator   Крафтовая линия    Энциклопедия / ТЗ
```

Слой шаблонов **не** обязан жить в клиентском бандле целиком: допустима **генерация** TS/JSON рецептов на этапе сборки репозитория, если так проще отлаживать диффы.

## 4. Модель данных (концептуально)

Ниже — логическая схема полей, а не обязательный срез текущих `craft-v2.ts`. Имена можно согласовать при внедрении.

### 4.1 ProcessFragment

- `id: string` — глобально уникален (`fragment_metal_smelt_roll_v1`).
- `stages: FragmentStage[]` — список шаблонных стадий.

**FragmentStage** (концептуально):

- `stageType: string` — как в `RecipeStageConfig`.
- `materialRole?: string` — плейсхолдер роли (`"primary_blade"`), подставляется в `material`.
- `targetRole?: string` — подставляется в `target`.
- Иные поля `RecipeStageConfig` при необходимости (`baseDurationOverride` и т.д.).

Подстановка: при материализации `primary_blade` → `blade` даёт обычную стадию `{ stageType, material: 'blade', target: 'blade' }`.

### 4.2 RecipeProcessTemplate

- `id: string` — `template_sword_standard_4part`.
- `version: number` — для миграций контента (при смене шаблона пересобрать производные рецепты).
- `partRoles: TemplatePartRole[]` — какие **роли** ожидает шаблон и какие требования к типам материалов (можно ссылаться на общую схему или дублировать упрощённо).
- `body: TemplateNode[]` — упорядоченное тело: либо одна стадия, либо вызов фрагмента с передачей ролей.

**TemplateNode** (варианты):

- `{ kind: 'stage', ...FragmentStage }` — одна стадия с ролями.
- `{ kind: 'fragment', fragmentId: string, bind: Record<string, string> }` — вставить фрагмент; `bind` сопоставляет роли фрагмента ролям шаблона (`{ part: 'primary_blade' }`).
- Опционально в будущем: `{ kind: 'foreach', roleList: string[], node: TemplateNode }` — развернуть цикл по нескольким металлическим частям (осторожно с порядком).

### 4.3 RecipeDefinition

- `recipeId: string` — итоговый `WeaponRecipe.id`.
- `templateId: string` + `templateVersion?: number` (pin).
- `roleToPartId: Record<string, string>` — маппинг: `primary_blade` → `blade`, `secondary_guard` → `guard`.
- `overrides?: RecipeOverride[]` — дельты после развёртывания шаблона (раздел 5).
- **Уникальные поля** — копируются в `WeaponRecipe` как есть: `name`, `type`, `description`, `parts` (полные `RecipePart[]` с именами и свойствами), `combatPart`, `baseStats`, `source`, `requiredLevel`, `cost`.

**Важно:** `parts` в конкретном рецепте остаётся источником правды для UI и калькулятора; шаблон оперирует **ролями**, которые **уже согласованы** с этими `parts` через `roleToPartId`. Валидация: каждая роль из шаблона имеет ключ в `roleToPartId`, каждый `part.id` участвует хотя бы в одном ожидаемом слоте (правило задаётся по классу шаблона).

### 4.4 RecipeOverride (дельты)

Детерминированный мини-язык правок **после** развёртывания в список `RecipeStageConfig` (но до финальной нормализации ключей):

| Операция | Назначение |
|----------|------------|
| `insertAfter` | Якорь: ключ стадии / индекс / роль+stageType — вставить одну или несколько стадий. |
| `remove` | Удалить стадии по фильтру (например все `target === 'pommel'` и `stageType === …`). |
| `replace` | Первая стадия, удовлетворяющая предикату, заменяется другой конфигурацией. |
| `patchDuration` | Изменить `baseDurationOverride` у предиката. |

Предикаты должны быть **устойчивыми** к добавлению стадий в шаблоне: предпочтительно опираться на **`stageInstanceId`**, генерируемый материализатором для каждой развёрнутой стадии (см. 4.5), а не только на индекс.

### 4.5 Стабильные ключи стадий при материализации

При материализации каждая строка `stages[i]` получает:

- нормальный `RecipeStageConfig` для движка;
- опционально **`stageInstanceId: string`** (если расширяем тип) — например `basic_longsword:form_forging:blade:7`, используемый в дельтах и якорях Крафтовой линии.

Если поле в `WeaponRecipe` добавлять нельзя на первом шаге, ключ хранится в **параллельном** массиве или в таблице «рецепт + индекс → ключ» в данных микроэтапов хребта (хуже для сопровождения). Долгосрочно поле на стадии предпочтительнее.

## 5. Семантика слияния и порядок применения

1. Развернуть шаблон в плоский список стадий (рекурсивно подставить фрагменты и роли).  
2. Присвоить `stageInstanceId` последовательно или по детерминированной формуле.  
3. Применить **overrides** в порядке объявления в `RecipeDefinition`.  
4. Валидация: нет дыр в обязательных этапах для данного класса оружия; все `stageType` существуют в библиотеке этапов.  
5. Сборка итогового `WeaponRecipe`: `parts` и прочие поля из определения; `stages` — результат шага 3.

Конфликт двух дельт на одной стадии — **ошибка авторинга** (падает валидация сборки), если не задан явный приоритет; в пилоте приоритет не вводить, только порядок массива.

---

## 6. Связь с частями оружия

Два подхода (можно сочетать):

**A. Полностью явные `parts` в RecipeDefinition**  
Автор перечисляет `RecipePart[]` как сегодня; шаблон только даёт процесс. Маппинг ролей → id обязателен. Подходит для сложных рецептов с нестандартными именами частей.

**B. Схема parts в шаблоне**  
Шаблон декларирует «ожидаемые» части с типами материалов и лимитами; RecipeDefinition заполняет только отличия (имена, `minQuantity`). Требует второго прохода генерации `parts`. Риск — рассинхрон с визуальными ожиданиями; вводить после стабилизации пилота А.

Рекомендация: **фаза 1** — подход **A**; **фаза 2** — опциональные схемы для массовых серийных рецептов.

## 7. Реестры и версионирование

- **Реестр фрагментов** — один файл или модуль на семейство (`fragments/metal.ts`, `fragments/wood.ts`).  
- **Реестр шаблонов** — отдельно от рецептов игрока.  
- **Версия шаблона** — при изменении тела шаблона увеличить `version`; RecipeDefinition может pin `templateVersion`, чтобы старый контент не менялся молча; при обновлении пинов — массовая пересборка и QA.

---

## 8. Крафтовая линия и микроэтапы хребта

Согласованность с **`CRAFT_LINE_RECIPE_TECHNIQUE_COMPOSITION.md`**:

- **Микроэтапы по умолчанию** могут висеть на **`RecipeProcessTemplate`** (общий хребет для всех мечей этого шаблона) с привязкой к `stageInstanceId` или к паре `stageType` + роль.  
- **RecipeDefinition** может добавлять **дельты микроэтапов** (те же операции: заменить подписи, веса, добавить шаг только для «огненного» варианта меча).  
- После материализации Крафтовая линия читает **итоговый** рецепт так же, как для ручного рецепта; источник микроэтапа в логах может помечаться `template` vs `recipe_delta` для отладки.

**Префиксы id микроэтапов:** для шаблонных — например `tpl_sword_std_…`; для оверлоев конкретного рецепта — `{recipeId}_macro_…`, чтобы не пересекаться с техниками (как в ТЗ по мечу).

---

## 9. Валидация и тестирование

Обязательные проверки на сборке или в CI:

1. Материализация каждого `RecipeDefinition` не бросает; результат удовлетворяет схеме `WeaponRecipe`.  
2. Все `stageType` из развёрнутых стадий присутствуют в библиотеке этапов.  
3. `combatPart` совпадает с одним из `parts[].id`.  
4. Для пилотного шаблона меча: **снимок** `stages` совпадает с эталоном или с контрольным хешем (регресс при правке шаблона).  
5. Уникальность `stageInstanceId` внутри рецепта.  
6. При включённой Крафтовой линии v2: тест «план + материализованный рецепт → порядок подписей» не хуже эталона пилота.

## 10. Авторский процесс (рекомендация)

1. Выбрать или создать **шаблон** для класса оружия.  
2. Вынести повтор в **фрагменты** (когда одна и та же пара стадий встречается ≥ 3 раз в разных шаблонах).  
3. Создать **RecipeDefinition**: маппинг ролей, уникальные статы, дельты при необходимости.  
4. Прогнать материализацию, проверить снимок `stages`, прогнать крафт в игре / интеграционный тест.  
5. Зафиксировать версию шаблона в определении перед релизом контента.

## 11. Стратегия внедрения (согласованно с фазами Крафтовой линии)

| Фаза | Действия по модулю композиции |
|------|--------------------------------|
| **До / параллельно фазе 0 ТЗ меча** | Зафиксировать, что `basic_sword` остаётся **эталоном** явного рецепта; черновик одного `RecipeProcessTemplate`, воспроизводящего тот же порядок стадий, только в ветке / PR для проверки снимка. |
| **После стабилизации пилотных микроэтапов** | Ввести материализатор + один шаблон = `basic_sword`; CI-сравнение снимка с эталоном. |
| **Масштабирование** | Новые мечи = новые `RecipeDefinition` на тот же шаблон; редкие уникумы = отдельный шаблон или явный рецепт. |
| **С фазой 5 Крафтовой линии** | Микроэтапы на уровне шаблона и дельты на уровне рецепта (раздел 8). |
| **После пилотных примеров рецептов / параллельно масштабированию** | Актуализировать skill **[weapon-recipe-authoring](../.cursor/skills/weapon-recipe-authoring/SKILL.md)** (или выпустить v0 раньше — только плоский путь и `allRecipes`, затем дополнить шаблонами). См. **§9** [CRAFT_LINE_RECIPE_TECHNIQUE_COMPOSITION.md](CRAFT_LINE_RECIPE_TECHNIQUE_COMPOSITION.md). |

## 12. Риски и ограничения

| Риск | Митигация |
|------|-----------|
| «Чёрный ящик» для дизайнера | Экспорт развёрнутых стадий в markdown/JSON в CI; кнопка «показать итоговый таймлайн» в devtools при необходимости. |
| Дельты ломают индексы | Опора на `stageInstanceId`, не на сырой индекс после вставок. |
| Раздувание абстракции | Порог вынесения в фрагмент: минимум три повтора; иначе копипаста в шаблоне терпима. |
| Две правды (шаблон vs развёрнутый JSON) | Одна правда в репозитории: либо только Definitions + материализация при сборке, либо только плоские рецепты + генерация из DSL — не хранить расхождений без аудита. |

## 13. Резюме

Модуль композиции — это **слой авторинга** между дизайнером и существующим **`WeaponRecipe`**: шаблоны и фрагменты задают процесс, определение рецепта — уникальность оружия и дельты. После материализации поведение игры **идентично** ручному рецепту, что сохраняет согласованность симуляции, Крафтовой линии и будущего персиста микроэтапов.

## 14. Приложение: три базовых типа до старта работ (меч, топор, кинжал)

Ниже — **конкретизация концепции** для приёмки шаблонов и материализатора: фрагменты, тела шаблонов, маппинг ролей и отличия рецептов. Опора — текущие данные **`basic_sword`** (`swords.ts`), **`basic_dagger`**, **`basic_axe`**, **`battle_axe`** (`melee.ts`). Идентификаторы фрагментов и шаблонов — **проектные** (финальные имена фиксируются при внедрении).

### 14.1 Роли частей (логика шаблона → `RecipePart.id`)

| Роль | Смысл | Типичный `part.id` |
|------|--------|---------------------|
| `edge` | Режущая / ударная металлическая масса (клинок меча, полотно топора, клинок кинжала) | `blade` |
| `cross` | Поперечная металлическая фурнитура у гарды | `guard` |
| `haft` | Древко / рукоять (дерево, иногда кожа у кинжала) | `grip` |
| `counter` | Навершие, противовес | `pommel` |

У **топора** в данных только `blade` + `grip`: роли **`edge`** и **`haft`**. У **кинжала** в таймлайне сейчас задействованы только **`edge`** и **`haft`**; гарда и навершие в `parts` есть, но **стадии на них не разворачиваются** — это осознанное упрощение базового кинжала; расширение (отдельные стадии на `guard`/`pommel`) — через **дельты** или новую версию шаблона.

### 14.2 Каталог фрагментов (развёртка в `RecipeStageConfig`)

Каждый фрагмент при материализации подставляет **`partId`** из маппинга роли.

| `ProcessFragment.id` | Содержимое (последовательность `stageType`, поля `material` / `target` = роль, после маппинга = `partId`) |
|----------------------|------------------------------------------------------------------------------------------------------------------|
| `frag_core_prep` | `prep_heating`, `prep_tools` |
| `frag_metal_smelt_roll` | `proc_smelting`, `proc_rolling` для одной роли (обе стадии: та же часть) |
| `frag_wood_saw` | `proc_sawing` для роли рукояти / древка |
| `frag_form_edge_forge` | `form_heating`, `form_forging` на роли `edge` |
| `frag_form_cross_shape` | `form_heating`, `form_shaping` на роли `cross` |
| `frag_form_counter_shape` | `form_heating`, `form_shaping` на роли `counter` |
| `frag_form_haft_carve` | `form_carving` на роли `haft` |
| `frag_form_axe_upset_punch` | `form_upsetting`, `form_punching` на роли `edge` (топор) |
| `frag_form_axe_battle_extra` | `form_heating`, `form_shaping` на роли `edge` (боевой топор — между проходами койки, см. `battle_axe`) |

### 14.6 Шаблон `template_axe_2role_battle` или дельты к базовому топору (секира / боевой топор)

Текущий **`battle_axe`** отличается от **`basic_axe`** **длиннее цепочкой формовки полотна**: после `form_upsetting` идут повторный `form_heating`, `form_shaping`, затем `form_punching` (см. `melee.ts`).

**Вариант A (рекомендуется для ясности):** отдельный шаблон `template_axe_2role_battle` — строки 1–4 как у базового, затем:

| № | Вызов | Примечание |
|---|--------|------------|
| 5 | `frag_form_edge_forge` | как в battle_axe |
| 6 | `frag_form_axe_upset_punch` | только `form_upsetting`; **или** разбить фрагмент топора на два: `frag_axe_upset`, `frag_axe_punch` для точного совпадения порядка |
| 7 | `frag_form_axe_battle_extra` | `form_heating`, `form_shaping` |
| 8 | отдельная стадия | `form_punching` — вынести в `frag_axe_punch_only` при рефакторинге фрагментов |

Для **строгого совпадения** с текущим `battle_axe` на этапе внедрения проще **свернуть** строки 5–9 battle_axe в **один фрагмент** `frag_form_axe_head_battle_v1` с точной последовательностью из файла, чем наращивать комбинаторику мелких кирпичиков.

**Вариант B:** одна база `template_axe_2role_basic` + `RecipeOverride.replace` блока формовки `edge` на развёрнутый фрагмент «battle» у определения `battle_axe`.

**RecipeDefinition (`battle_axe`):** те же роли `edge`/`haft`; отличия только `baseStats`, `parts` (количества), `requiredLevel`, `source`.

**Термин «секира» в дизайне** здесь отождествляем с **тяжёлым одноручным / двуручным топором** в данных проекта — опорный рецепт **`battle_axe`**; при появлении отдельного `poleaxe` понадобится **новый шаблон** (другая топология или древко из двух частей).

### 14.7 Перекрёстное переиспользование фрагментов

```text
                    frag_core_prep
                    frag_metal_smelt_roll
                    frag_wood_saw
                    frag_fin_common
                           ▲
        ┌──────────────────┼──────────────────┐
        │                  │                  │
   template_sword    template_dagger    template_axe_*
```

Фрагменты **`frag_form_axe_*`** **не** используются в мече и кинжале; **`frag_form_cross_shape`** / **`frag_form_counter_shape`** — только в полном мече. Это снижает риск неверной вставки стадии в чужой шаблон.

### 14.8 Чеклист перед кодированием материализатора

1. Зафиксировать **версию** каждого шаблона (`template_sword_1h_4role` v1, …).  
2. Прогнать **четыре** материализации → дифф с `basic_sword`, `basic_dagger`, `basic_axe`, `battle_axe` (или принять осознанное отличие одной стадии с записью в changelog).  
3. Для каждого шаблона — **таблица стадия → смысловая группа** Крафтовой линии (наследие ТЗ меча + аналогичные группы для топора/кинжала).  
4. Запретить в v1 **условные** фрагменты («если guard выбран»); только статическое тело + дельты вручную.  
5. После первых успешных материализаций обновить skill **[weapon-recipe-authoring](../.cursor/skills/weapon-recipe-authoring/SKILL.md)** (раздел шаблонов и CI).

После выполнения приложения 14 концепция «три типа оружия из модулей» **проверяема** без догадок: каждая строка шаблона имеет имя фрагмента и однозначную развёртку в стадии, совместимую с действующими рецептами в репозитории.
