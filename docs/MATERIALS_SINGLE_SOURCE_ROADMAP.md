## 1. Зачем этот документ

В коде одновременно существуют:

- несколько **источников узлов** `MaterialNode` (ручные массивы, `worldResourceNodes`, bridge-заглушки, квестовые узлы);
- **два контура данных** для материала в оружии: `Material` в [`src/data/materials/metals.ts`](../src/data/materials/metals.ts) и каталог **`MaterialNode`** в [`src/data/materials/library/`](../src/data/materials/library/);
- **двойной учёт** запасов: `resources: ResourceKey` и `materialStash: materialId` с таблицами маппинга, мостами и исключениями в [`src/lib/craft/inventory-check.ts`](../src/lib/craft/inventory-check.ts);
- **переработка**, зашитая в [`src/data/refining-recipes.ts`](../src/data/refining-recipes.ts), и **техники обработки** в [`src/data/material-processing-techniques.ts`](../src/data/material-processing-techniques.ts), связанные с рецептами и вставками этапов — без общей модели «атомарная операция → граф процесса»;
- **энциклопедия** опирается на объединённый `allMaterials`, а вспомогательные срезы (`allMetals`, `materialsByClass.metal`) дают **другие** подмножества.

**Цель проекта:** одна согласованная система — **один каталог `materialId`**, **один способ учёта**, **один слой описания преобразований**, жёстко **привязанный к техникам** (обработка и в перспективе весь таймлайн крафта собирается из операций в данных техник). Постоянные адаптеры и «навсегда» миграции сейва **не являются целью** (см. §6).

## 2. Принципы (неготируемые)

1. **Единый каталог**  
   Каждый игровой материал имеет ровно один **`materialId`** и один узел в **`materialRegistry`** (или производимый из одной схемы). ENC, экспедиции, лавка, алтарь, расчёты — только отсюда.

2. **Нет скрытых дублей экономической сущности**  
   Два id на один смысл (например «Медь» и «Медный слиток» как две карточки без модели стадии) **устраняются**: либо один материал с явными стадиями в данных техники/процесса, либо два id с **разными** ролями в графе техники (вход/выход операции), отображаемые в ENC как цепочка, а не как независимые «два слитка». *Переходный слой UX:* в списке энциклопедии для пары `id` (класс `metal`) + существующий в реестре `id_alloy` игроку показывается одна карточка — слиток; узел базы остаётся в каталоге для маппинга и крафта (см. **§3.4**).

3. **Обработка = техника (контейнер операций)**  
   Любое превращение сырья в пригодную для части оружия стадию описывается **техникой обработки** (или составной техникой): внутри — **упорядоченные атомарные операции** (время, расход, условия, возможные ветвления). Техника может быть **универсальной** (подходит любому материалу с тегами/ролями X) или **узкой** (`targetMaterialIds` / один металл). Связь «продуктовая линия руда → слиток» **не выносится в отдельное поле `line_key` на материале**; она **воспроизводится графом техники** (цепочка операций и явные `input`/`output` `materialId`). См. §5.

4. **Один слой учёта на целевой горизонте**  
   Счётчики только по **`materialId`** в едином складе (позиция **A2** в терминах [MATERIALS_UNIFICATION_AUDIT §3.0](MATERIALS_UNIFICATION_AUDIT.md)). Промежуточные `ResourceKey` как дубликаты каталога **снимаются**, а не поддерживаются вторым рядом таблиц.

5. **Крафт оружия (целевое)**  
   Игрок выбирает **дизайн**, **материалы по частям**, **набор техник работы** (обработка сырья + боевые техники ковки и т.д.). **Этапы таймлайна** не захардкожены одной статичной последовательностью в коде: они **собираются** из операций выбранных техник и плана частей (динамическая адаптивная система). Текущий MVP (фиксированные `stageType` + `craftStageInsertions`) — **временный мост** до этой модели; см. §5.3.

6. **Legacy удаляется, не закрепляется**  
   Временные мосты реестра (pick-сегменты, устаревшие ручные concat, дублирующие типы) **убираются к концу работ** (сегмент **5.3–5.4** снят — см. **§11**). Временные ветки допустимы **только** внутри коротких PR по этому roadmap; в финале — **ноль** обязательных адаптеров. См. §6.

## 3. Целевая архитектура (кратко)

### 3.1 Реестр материалов

| Компонент | Назначение |
|-----------|------------|
| **`materialRegistry`** | Единственный декларативный список всех `MaterialNode`; сборка из доменов (`metals/`, `ores/`, `world/`, …) без «второго массива вслепую». |
| **`materialById`** | Строго производный индекс от реестра; **уникальность** `id`. |
| **Каталог ↔ склад** | Любое начисление/списание ссылается на `materialId` из реестра; правила совместимости с зданиями/экспертизой — в данных или в общих валидаторах. |

### 3.2 Техники и операции (целевая модель)

| Компонент | Назначение |
|-----------|------------|
| **Техника обработки** | Контейнер: метаданные, разблокировка, **список операций** (порядок, ветки при необходимости), связь с **входными/выходными** `materialId`, длительность, расход доп. ресурсов, модификаторы итога/прогноза. |
| **Атомарная операция** | Минимальный шаг с явной семантикой (пример из геймдиза: «обжиг руды», «сыродутная плавка», «проковка крицы», «цементация»). Может маппиться на UI-этап и на правила расхода склада. |
| **Техника боевого крафта** | Остаётся отдельным типом данных ([`src/data/techniques`](../src/data/techniques.ts) / каталог техник); **совместимость** с техниками обработки задаётся явно (аналог текущих `incompatibleWithCraftTechniqueIds`). |
| **Связь с переплавкой** | На переходный период техника может ссылаться на `refiningRecipeId`; **цель** — либо сгенерировать рецепт горна из операций техники, либо хранить расход **только** в операциях, без параллельного дубля в `refining-recipes`. |

**Пример (из ТЗ геймдиза):** техника «Цементационная сталь» → операции: обжиг руды → сыродутная плавка → проковка крицы → цементация → **итоговый материал:** стальной слиток (низкоуглеродистый) как явный `output` последней операции или финального шага техники.

### 3.3 Склад (store)

- Персист: **единый** склад по `materialId` (имя поля в Zustand может сохраниться `materialStash` до рефакторинга).  
- Валюты — отдельное решение (поля `gold` / `soulEssence` **или** материалы с тегом `currency`); фиксируется в фазе 0.

### 3.4 Энциклопедия

- Список карточек — только из **`materialRegistry`** (источник данных — `allMaterials` / `materialById`; без отдельных «теневых» массивов — контроль **§10**).  
- **Металлы `id` + `id_alloy`:** в каталоге по-прежнему два узла (база и слиток); в **списке** экрана энциклопедия базовый металл **скрыт**, если в реестре есть одноимённый `materialId` с суффиксом `_alloy` — в UI остаётся одна строка на семейство (канон склада/префиксов — слиток). Реализация: [`encyclopedia-base-metal-alloy-display.ts`](../src/lib/materials/encyclopedia-base-metal-alloy-display.ts) (`isBaseMetalSupersededByAlloyCatalogEntry`, `resolveEncyclopediaFocusMaterialId`), экран [`encyclopedia-screen.tsx`](../src/components/screens/encyclopedia-screen.tsx), тест [`encyclopedia-base-metal-alloy-display.test.ts`](../src/lib/materials/encyclopedia-base-metal-alloy-display.test.ts). Навигация «открыть в ENC» с `iron` перенаправляется на `iron_alloy`, чтобы скролл попадал в видимую карточку.  
- Группировка: `identity.class`, теги роли (**руда / заготовка / слиток** и т.д. — политика тегов); **ссылка на техники, ведущие к материалу как к выходу** — в UI «как получить» ([`material-acquisition-hints.ts`](../src/lib/materials/material-acquisition-hints.ts), индекс [`material-processing-output-index.ts`](../src/lib/materials/material-processing-output-index.ts)).  
- Сортировка: tier → имя ([`encyclopedia-display-order.ts`](../src/lib/materials/encyclopedia-display-order.ts)); при необходимости позже — тип стадии → имя или `sortKey` в данных.

### 3.5 UI: шапка и «Торговая лавка» (устаревшие механики — убрать в рамках рефактора)

Этот блок **не отменяет** единый каталог и склад, но синхронизирует UX с принципом «канон материалов — энциклопедия и склад по `materialId`», без вводящих в заблуждение агрегатов и бессмысленной торговли.

#### 3.5.1 Верхняя панель (шапка проекта)

- **Проблема:** в верхней части интерфейса отображаются суммарные числа в духе «Дерево», «Камень», «Железо», «Уголь» (агрегация пулов `ResourceKey` / legacy-представление). Для игрока это **не даёт** полезной информации о реальном каталоге и смешивает разные `materialId`, сведённые в один ключ.
- **Решение:** **полностью убрать** этот блок из шапки. Реализация в коде ориентировочно: компонент **`ResourceBar`** (сейчас объявлен внутри [`game-layout.tsx`](../src/components/layout/game-layout.tsx)) (см. также [LEGACY_UI.md](LEGACY_UI.md)).
- **После удаления:** место в шапке можно занять позже **осмысленным** UI (например краткий статус кузницы, золото, активный квест, уведомления) — **вне скоупа** текущего ТЗ; отдельное дизайн-решение.

#### 3.5.2 Магазин («Торговая лавка»)

- **Продажа ресурсов кузнецом — не нужна.** Механика продажи (вкладка **«Продажа»**, списание ресурсов/сырья в обмен на золото и связанный UI/store) **удаляется целиком**: нет игрового смысла в постоянной «ликвидации» сырья через лавку при опоре на экспедиции, склад и каталог.
- **Покупка — убрать заглушки.** На вкладке **«Покупка»** удалить все предложения типа «Мешок», «Телега», «Большой мешок» и любые аналогичные **нефункциональные** позиции (чистка [`shop-screen`](../src/components/screens/shop-screen.tsx), [`material-shop.ts`](../src/data/material-shop.ts) и связанных хелперов от мёртвого ассортимента).
- **Стартовый ассортимент покупки (жёстко минимальный):** доступны **только** три позиции, согласованные с каноническими `materialId` каталога:
  - **железная руда** — `iron_ore`;
  - **уголь** — `coal` (или иной канонический id топлива, если переименуют в реестре — тогда одна строка в данных лавки);
  - **берёза** — `birch`.
- **Расширение ассортимента:** в данных лавки должна быть **простая схема** добавления новой позиции на продажу (например запись: `materialId`, цена, лимит стека, опционально уровень гильдии и т.д.). Чтобы предложение **появилось в UI**, у позиции должен быть флаг вида **`unlocked` / проверка разблокировки** (источник — store или отдельный реестр «доступные покупки»).
  - **Как именно разблокируются новые ресурсы** (квест, уровень, репутация, флаги сюжета) — **не цель этого roadmap**; достаточно **заложить расширяемую точку** и **комментарий в коде** у реестра/селектора списка покупок: «новые строки подключаются здесь; условие видимости — `isShopOfferUnlocked(offerId)` (или аналог)».
- **Связь с материалами:** каждая покупка начисляет **`materialId`** через существующий контур склада ([`addMaterialToStash`](../src/store/slices/resources-slice.ts), [`grantResourceKeyFromWorld`](../src/lib/craft/inventory-check.ts) — по мере перехода к A2 единый слой начисления); **не** плодить отдельные «лавочные» типы ресурсов без узла в `materialRegistry`.

**Ориентиры по коду (на момент ТЗ):** [`src/components/screens/shop-screen.tsx`](../src/components/screens/shop-screen.tsx), [`src/data/material-shop.ts`](../src/data/material-shop.ts), slice/экшены продажи в store — подлежат выносу или удалению вместе с вкладкой «Продажа».

### 3.6 Влияние на Craft v2, перековку, ремонт и постройку алтаря

Переработка материалов **не изолирована**: целевой **`materialId`** + единый слой списания задевают все системы, где есть расход со склада или ссылка на каталог. Ниже — **что именно** и **когда** по фазам **§7** пересматривать вместе с материалами (чтобы не было сюрпризов после волны **2.x** или **3.x**).

| Подсистема | Зависимость от материалов / склада | Главный риск при рефакторе | Фазы roadmap (ориентир) |

## 4. Схема данных материала (без `line_key`)

База — [`MaterialNode`](../src/types/materials/material-core.ts). Целевой **минимум полей на материале**:

| Поле | Описание |
|------|----------|
| `identity.id`, `name`, `class`, `origin`, `tags` | Как сейчас; теги включая **роль в процессах** (руда, fuel, …) по [MATERIAL_SEMANTIC_PROCESS_ROLES.md](MATERIAL_SEMANTIC_PROCESS_ROLES.md). |
| Физика / обработка / экономика | Как сейчас (или шаблон наследования для семейства руд). |

**Явно не вводим** поле уровня материала `line_key` для группировки «медная линия»: эту связь даёт **граф техники** (операция N: `copper_ore` → `copper_concentrate`; операция M: → `copper_alloy`). Если понадобится **кэш для ENC** («показать связанные стадии»), его строят **off-line** из реестра техник (производная структура), а не дублируют вручную на каждом узле.

**Связь материал ↔ техника:**  
- *Снизу вверх:* техника декларирует `targetCatalogMaterialIds` и/или паттерны тегов (универсальная техника).  
- *Сверху вниз:* по мере необходимости — обратный индекс `materialId → techniqueIds[]` для UX (генерация, не ручное поле в `MaterialNode`).

## 6. Политика по сейвам и миграциям

**Зафиксировано (требование заказчика):**

- **Постоянная миграция старых сейвов не является целью.** Игра без релиза; потерять старые сохранения допустимо.
- На время работ по этому roadmap **допускается** временный код переноса/очистки **только** для удобства разработчика (например поднять `STORE_VERSION` и сбросить несовместимые поля).
- **Финальное состояние репозитория:** весь legacy-код совместимости (`replaces_legacy_ids`, мосты stash↔resources, дубли типов) **удалён**; документация описывает **одну** модель. Нет «навсегда» dual-path в [`game-store-composed`](../src/store/game-store-composed.ts) persist middleware ради старых ключей.

Следствие: инварианты CI и ТЗ ориентируются на **чистоту кода**, а не на бессрочную поддержку миграций `resources.iron` → stash.

## 7. Фазы работ (скелет; критерии выхода)

Фазы **намеренно крупные**; реальная работа идёт **пакетами** (отдельные PR), чтобы каждый набор операций был обозримым и заканчивался зелёным CI. Пакеты именуются **«фаза X — пакет Y»** (или `X.Y` в worklog).

**Планирование и §8.5:** таблица **§7** отвечает на вопрос *что менять в данных/коде*; таблица **§8.5** — *какие обязательные тесты, опциональный CLI и рекомендуемый Cursor Skill* закрепить за тем же диапазоном пакетов. Это **один план**, а не «сначала фича, потом тесты»: строка **§8.5** для **0.1–0.2** задаёт минимум CI (Vitest-контракт + расширение сканеров), **0.5** — слот под `audit:materials`, Skill — опционально после стабилизации сканеров. При оценке спринта/PR проверять **обе** таблицы.

### 7.0 Дробление: правило «один пакет — один смысл»

| Критерий | Как резать |
|----------|------------|
| **Размер** | Один PR: типично **одна** из: данные, store, UI, контракт+сканеры, документация — без смешения трёх систем сразу. |
| **Контракт** | После пакета: **§8** без новых `error` в заявленном scope; сканеры пополнены, если появился новый тип ссылок на `materialId`; инструменты из **§8.5** для этой фазы (тесты обязательны, CLI/Skill — по таблице). |
| **Волна материалов** | Горизонтальный срез («все топлива», «базовые руды») — **отдельный пакет** внутри фазы 1–2 + строка в §11. |
| **Вертикальный e2e** | Пакет может быть «одна цепочка меди end-to-end» вместо «пол-игры за раз». |

Ниже у каждой фазы перечислены **рекомендуемые пакеты**; состав можно уточнять, но **не объединять** в монолит без крайней необходимости.

### 7.1 Рекомендуемый порядок внедрения и запреты

Чтобы не потерять **сотню узлов** каталога и не словить «половину A2» в проде:

| Правило | Смысл |
|--------|--------|
| **Никакого big-bang** | Не менять одновременно реестр, склад, горн, экспедиции и UI без промежуточных зелёных CI. |
| **Фаза 0 — только измерение** | Первый mergable результат: контрактные тесты + отчёт покрытия (**§8**), без смены игрового поведения (или с минимальным). |
| **Волны по доменам** | Переносить или чистить материалы **волнами** (например: топливо → базовые руды → мир `worldResourceNodes` → квестовые) — одна волна ≈ один ограниченный PR + зелёный **`material-catalog-contract`**. |
| **1b рано и отдельно** | Фаза **1b** (шапка, лавка) слабо связана с A2 — можно вынести **раньше** или **параллельно** с фазой 1, чтобы уменьшить когнитивную нагрузку на больших PR. |
| **Горизонтальные срезы только с тестом** | Любой срез «все материалы класса X» сопровождается обновлением **источников для сканера** (**§8.2**) и прогоном контракта. |
| **Вертикальные срезы предпочтительнее** | Один полный сценарий end-to-end (например «медная цепочка: награда → stash → техника → крафт») лучше довести до конца на подмножестве id, чем «везде чуть-чуть сломано». |
| **После каждого PR** | `npm run lint`, `npm run test`, `npm run type-check`, `npm run build` (как в [AGENTS.md](../AGENTS.md) / CI). |

**Критерии готовности одной волны:** контракт **§8** без новых `error`; существующие тесты цепочек ([`expedition-inventory-chain.test.ts`](../src/lib/materials/expedition-inventory-chain.test.ts), [`inventory-check.test.ts`](../src/lib/craft/inventory-check.test.ts)) зелёные; ручной smoke по сценарию волны (краткий чеклист в описании PR).

## 8. Покрытие каждого материала: контракт, утилита, skill

Задача: **планомерно убедиться**, что по каждому `materialId` не забыты ссылки, маппинг и смысл (или сознательное исключение). Ручной просмотр «всех 100+» не масштабируется — нужен **машинный контракт** и по желанию **человеческий** skill для добавления контента.

### 8.1 Два уровня истины

1. **Реестр** — множество `allMaterials` / будущий `materialRegistry` (`materialById`).  
2. **Потребители** — разбросаны по коду: маппинг [`inventory-check.ts`](../src/lib/craft/inventory-check.ts) (`CORE_MATERIAL_TO_RESOURCE`, мосты), [`refining-recipes.ts`](../src/data/refining-recipes.ts), [`material-processing-techniques.ts`](../src/data/material-processing-techniques.ts), [`material-shop.ts`](../src/data/material-shop.ts), конфиги алтаря ([`altar-phases-config.ts`](../src/data/altar/altar-phases-config.ts) и баланс материалов), таблицы экспедиций/наград, квестовые узлы, [`metalMaterials`](../src/data/materials/metals.ts) до слияния.

**Контракт** — один модуль теста или скрипт, который:

- собирает **множество A** = все `materialId` из реестра;
- собирает **множество B** = все строковые ссылки `materialId` из зарегистрированных **источников** (статический список файлов/паттернов в одном месте, чтобы не гадать);
- проверяет: **B ⊆ A** (нет «висячих» id вне каталога);
- опционально: **A \\ B** помечается как «только ENC / только лор» через явный **allowlist осознанных orphans** в том же тесте (каждая строка с комментарием в коде).

Дополнительно — отдельные правила по категориям (примеры):

- если материал в **`CORE_MATERIAL_TO_RESOURCE`** или участвует в **списании крафта v2**, он **обязан** иметь валидный путь расхода (реестр + тест `canCatalogMaterialSpendInForgeCraft` где применимо);
- если материал только в **добыче** — проверка моста [`world-resource-inventory-bridge.ts`](../src/lib/materials/world-resource-inventory-bridge.ts) / будущего единого правила;
- **квестовые артефакты** могут быть вне экономики горна — явный список исключений в контракте.

### 8.2 Артефакт «покрытие по материалу»

Рекомендуется генерировать **таблицу** (Markdown или JSON в репозитории **или** только в артефакте CI — решение команды) со строками:

`materialId | в реестре | маппинг склада | горн/рецепт | техника обработки | экспедиция | лавка | алтарь | исключение (да/нет)`

Столбцы заполняются **эвристикой** из сканеров; столбец «исключение» — только если id в **allowlist** с обоснованием в коде контракта.

Так видно **дыры**: материал есть в реестре, но ни один контент его не выдаёт; или наоборот — награда ссылается на несуществующий id (**error**).

**Снимок покрытия маппинга (волны 2.4d–2.4h + хвосты 3.x/5.x этого пакета):** **2.4d–2.4f** — см. строку **§11**. **2.4g** — канонические руды и базовые металлы в [`WORLD_RESOURCE_TO_RESOURCE_KEY`](../src/lib/materials/world-resource-inventory-bridge.ts). **2.4h** — сплавы (`steel`, `high_carbon_steel`, `*_alloy`, `bronze`), `processed_wood`, `processed_stone` перенесены из **`CORE_MATERIAL_TO_RESOURCE`** в тот же мост; **`CORE`** пустой; **`ALLOY_RECIPES`** в [`inventory-check`](../src/lib/craft/inventory-check.ts) без изменения роли. **3.x / кожа:** рецепт **`tanned_leather_tan`** (`stashInputsPerBatch`, выход `tanned_leather`), постройка **`tannery`** в типах рецепта. **5.3–5.4 (закрыто в пакете «roadmap next» + follow-up):** узлы моста перенесены в `library/metals`, `library/leathers`, `library/stones`, `library/woods` и **`material-registry-core`**; плейсхолдер **`registryBridgeMaterialNodes`** снят с манифеста ([`material-registry-manifest.ts`](../src/data/materials/library/material-registry-manifest.ts)). Контракт каталога и CORE∩WORLD-тесты остаются зелёными.

**Хвост dual-path (исполнение плана «Materials roadmap next»):** пул **`ResourceKey` `coal`** переведён на учёт только через `materialStash` (каталожные id угля и эквиваленты в мосте); `resources.coal` после загрузки обнуляется идемпотентной миграцией **`STORE_VERSION` 31**. Код: [`a2-stash-only-pools.ts`](../src/lib/craft/a2-stash-only-pools.ts), [`a2-dual-path-snapshot.ts`](../src/lib/craft/a2-dual-path-snapshot.ts) + Vitest [`a2-dual-path-snapshot.test.ts`](../src/lib/craft/a2-dual-path-snapshot.test.ts). Перековка **0.2:** расширены `catalogMaterialSpendIds` в [`reforge-techniques-registry.ts`](../src/data/reforge/reforge-techniques-registry.ts); ремонт — те же декларации для всех типов в [`repair-system.ts`](../src/data/repair-system.ts). **5.2:** Vitest — стабильность id в [`metals-runtime-merge.test.ts`](../src/data/materials/metals-runtime-merge.ts); техники **3.x** — этапы из `processingOperations` без дубля `craftStageInsertions` у меди, олова, тщательной и базовой плавки железа; **0.2** — сканер **`requiredMaterials`** боевых техник (только id из реестра); **P1.5c** — микрозадачи у техник обработки для ENC/UX.

### 8.3 Утилита vs Cursor Skill

| Инструмент | Назначение |
|------------|------------|
| **Vitest-контракт + при необходимости `scripts/*.mjs`** | **Обязательный минимум:** детерминированная проверка на каждый PR; падение CI = нельзя мержить с «забытым» материалом или битой ссылкой. Не заменяется «ручным чеклистом». |
| **Cursor Skill** | **Рекомендуется** для **операции «добавить/переименовать материал»**: пошаговый чеклист для агента/человека — обновить узел, прогнать контракт, обновить [`RESOURCE_TRANSFORMATION_MAP.md`](RESOURCE_TRANSFORMATION_MAP.md), волну в worklog. Можно расширить существующий визард [`.cursor/skills/material-definition-wizard/SKILL.md`](../.cursor/skills/material-definition-wizard/SKILL.md) **или** добавить отдельный skill вида `material-catalog-wave` со ссылкой на **этот документ §8**. Skill **не** заменяет тест: он снижает ошибки при редактировании данных. |

Итого: **риск «пропустили материал»** режется **CI**; **риск «сделали не в том порядке»** — §7.1 и волны; **риск «забыли обновить карту документации»** — skill + шаблон PR.

### 8.4 Синхронизация с уже существующими аудитами

Файлы наподобие [`MATERIALS_PHASE0_INVENTORY.md`](MATERIALS_PHASE0_INVENTORY.md) / JSON можно **не дублировать вручную**: отчёт §8.2 генерируется из кода; раз в спринт сверять глазами только **аномалии** (orphans, пустые колонки).

### 8.5 Тесты, утилита и Cursor Skill — **часть работы по фазам**

Инструменты не «откладываются на потом»: они **встроены в пакеты** §7 и при планировании учитываются **в той же строке**, что и фазовый пакет (см. пояснение в начале **§7**). Сводка:

## 9. Риски и меры (расширенно)

Цель раздела — **снять неожиданности до кода**: что может пойти не так, как это **увидеть рано**, что делать **до** мержа и **после** регресса.

### 9.1 Технические риски (данные, код, CI)

| Риск | Мера | Ранний сигнал |
|------|------|----------------|
| Слишком большой PR | Пакеты **§7.0**; один смысл на PR; **§7.1**. | Дифф > ~400 строк по `data/` + `store` без промежуточного контракта. |
| **Тихий пропуск материала** | **`material-catalog-contract`**; отчёт **§8.2**; allowlist только с комментарием в коде. | Новый `materialId` в JSON/TS вне реестра; предупреждения `reportCraftMaterialMappingGap` в консоли. |
| **Дубликат `materialId`** в concat | Тест уникальности при сборке `materialById`. | Контракт или runtime падает при дубле; при появлении — немедленный фикс, не «потом». |
| **Половинчатый A2** | Волны **2.2–2.3**; материал без пути списания → **error** в контракте до закрытия волны. | Игрок тратит из `stash`, но UI/заказ ещё смотрит на `resources`; тест цепочки красный. |
| Рассинхрон техник и каталога | После **3.x**: I/O операций ⊆ реестр; `targetCatalogMaterialIds` в сканере. | Красный контракт после правки `material-processing-techniques.ts`. |

| **Облако / Turso / API сейва** ([`cloud-save-feature.ts`](../src/lib/cloud-save-feature.ts), [`api/save`](../src/app/api/save/route.ts)) | При удалении полей или смене формата склада — чеклист расширения схемы **до** деплоя; без долгой совместимости (**§6**). | Ошибка десериализации в облаке; заранее: «облако выкл» или сброс dev-БД. |

| Флейки и **нестабильный рандом** в тестах | По [AGENTS.md](../AGENTS.md): мок `Math.random` где нужно; контракт **детерминирован**. | Плавающие падения CI — изолировать тест, убрать время/рандом из контракта. |

### 9.2 Процессные и организационные риски

| Риск | Мера | Ранний сигнал |
|------|------|----------------|
| **Scope creep** («ещё один маленький адаптер навсегда») | Явный запрет в **§6**; любой мост — с датой удаления в §11 или задаче; не мержить без плана пакета. | PR добавляет `if (legacy)` без тикета на снос. |

| Потеря контекста между пакетами | Каждый PR: **1 строка в §11** (пакет, волна); при смене правила — обновить **§10** или этот §9. | «Почему так сделано» не восстанавливается из git blame. |
| Документация расходится с кодом | После волны: точечно [RESOURCE_TRANSFORMATION_MAP.md](RESOURCE_TRANSFORMATION_MAP.md); верхний уровень — этот файл (**§11**). | Игрок/агент читает карту, в коде уже другие id. |
| Дублирование с [CRAFT_SYSTEM_ROADMAP.md](systems/CRAFT_SYSTEM_ROADMAP.md) | Граница: здесь — **каталог, склад, обработка, лавка**; там — детали боя ковки и ENC UX; одна **отсылка** в шапке, не две полные копии ТЗ. | Две противоречивые цифры порога экспертизы в разных файлах — сверка по `FORMULAS` / коду. |

### 9.4 Остаточные (принимаемые) риски

До закрытия **фазы 5** возможны:

- **Кратковременный** дубль учёта (A1+A2) в одном домене — только внутри явно названной волны **2.x** с красным контрактом за пределами волны.
- **Ручной** allowlist в контракте для квестовых/ENC-only id — с комментарием «почему не в экономике горна».
- **Не полное** покрытие экспедиционных наград сканером в **0.2** — допускается до подключения модуля; после подключения — **B ⊆ A** обязателен.

Фиксация в §11 при принятии остаточного риска.

### 9.5 Когда остановить пакет и откатить

- **Красный CI** после серии зелёных на той же ветке — bisect или откат коммита, не «дожимать» мержем.  
- **Контракт** показывает **> N** новых ссылок на несуществующие id (порог N задать команде, напр. 3) — не расширять PR, сначала починить реестр или сканер.  
- **Регресс gameplay**, не покрытый тестом — добавить минимальный тест или смоук-сценарий в описание PR **до** следующего пакета той же волны.
