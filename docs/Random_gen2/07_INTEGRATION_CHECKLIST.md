# 07. Чеклист интеграции и приёмки

## Функциональные сценарии

- [ ] Старт экспедиции при достаточном золоте: списание `supplies + deposit`, появление записи в `guild.activeExpeditions`.  
- [ ] Отказ старта: нет золота / нет слота по уровню гильдии / прочность оружия ≤ 10 / атака ниже `minWeaponAttack`.  
- [ ] События: после старта у активной экспедиции заполнен массив `events` с ожидаемым числом элементов и метками `triggeredAt`.  
- [ ] Завершение до `endsAt` и после `endsAt` (оба разрешены, если UX так задуман).  
- [ ] Успех: начисление комиссии золотом, War Soul через `addWarSoulToWeapon`, опыт игрока, запись в `history`, удаление из `activeExpeditions`.  
- [ ] Провал с потерей оружия: триггер `weaponLost`, создание `RecoveryQuest`, `removeWeapon`.  
- [ ] Крит: комиссия и war soul умножены согласно формулам.  
- [ ] Слава: поле `glory` гильдии увеличивается согласно формуле.  

## vNext: награды событий

- [ ] Эффект с `grant_resource` корректно увеличивает счётчик `resources`.  
- [ ] Эффект с материалом вызывает `addMaterialExpertise` (или согласованный новый API) и не превышает clamp 0–100.  
- [ ] Повторное открытие одного и того же события не дублирует награды (идемпотентность).  

## Сохранения

- [ ] После перезагрузки страницы `activeExpeditions` и `events` восстанавливаются из persist.  
- [ ] При **`NEXT_PUBLIC_CLOUD_SAVE_ENABLED=true`**: колонка `guild` в Turso содержит те же поля; загрузка через `/api/save` не ломает `merge` в store.  
- [ ] При выключенном облаке: регрессии нет — данные только в `localStorage`; прямой `GET /api/save` ожидаемо даёт `503` + `cloudSaveDisabled`.  

## Миграция

- [ ] Старый сейв без `adventurerExtended` обрабатывается (fallback или миграция).  
- [ ] Введённый `expeditionModuleVersion` мигрирует формат событий при необходимости.  

## Регресс хоста

- [ ] Другие экраны (`forge`, `resources`) не зависят от неинициализированного состояния после экспедиции.  
- [ ] Нет утечек ссылок на удалённое оружие в UI после потери.
