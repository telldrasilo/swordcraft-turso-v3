# План обновления SwordCraft до Next.js 16.1.x (минимизация рисков)

Документ описывает поэтапное обновление с **Next.js 15.x** на **Next.js 16.1.1** (или ближайший патч 16.1.x на момент выполнения), с учётом текущего стека: **React 19**, **App Router**, **next-auth v4**, **next-intl**, **Prisma + libSQL**, **Tailwind 4**, **Vitest**, CI на **Node 20**.

Цели:

- Обновить рантайм фреймворка без поломки продакшен-сборки и облачных сейвов.
- Свести «сюрпризы» к контролируемым шагам: отдельная ветка, фиксированные версии, полный прогон CI-эквивалента локально.
- Зафиксировать решения и откаты в **worklog** (ниже).

---

## 1. Исходное состояние (baseline)

Перед началом зафиксировать в worklog:

| Параметр | Где смотреть / что записать |
|----------|-----------------------------|
| Версии | `package.json`: `next`, `eslint-config-next`, `react`, `react-dom`, `next-auth`, `next-intl` |
| Node | Локально: `node -v`; в CI: `.github/workflows/ci.yml` (сейчас `20`) |
| Сборка | Вывод `npm run build` без ошибок |
| Линт / тесты | `npm run lint`, `npm run test` |
| Конфиг | `next.config.ts`, `tsconfig.json`, `middleware.ts` (если есть) |

Убедиться, что рабочее дерево чистое или изменения осознанно переносятся в ветку апгрейда.

---

## 2. Принципы снижения рисков

1. **Один крупный скачок за PR** — в одном изменении поднимаем `next` + `eslint-config-next` до одной мажорной линии; не смешивать с рефакторингом фич.
2. **Пины версий** — для апгрейд-PR зафиксировать точные версии (`16.1.1` и т.д.), не полагаться на широкие `^`, пока не стабилизируется билд.
3. **Ветка и PR** — все изменения в `chore/next-16-upgrade` (или аналог); merge только после зелёного CI и ручного смоук-теста UI.
4. **Кодмод опционально** — `npx @next/codemod@canary upgrade latest` ускоряет мелкие правки; если кодмод трогает слишком много нерелевантного, применить вручную по списку из официального гайда и чейнджлога 16.x.
5. **Откат = revert коммита(ов) или `git reset`** — не удалять ветку до успешного merge в main.
6. **Документация проекта** — после merge обновить **AGENTS.md** (там указана версия Next.js), чтобы агенты и люди не расходились с реальностью.

## 3. Матрица рисков и митигации

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Несовместимость **next-auth v4** с Next 16 | Средняя | После bump — полный логин/логаут, колбэки; проверить `NEXTAUTH_URL` / `AUTH_TRUST_HOST` (см. комментарии в `next.config.ts`). При блокере — смотреть issue/релизы next-auth или план миграции на Auth.js v5 отдельным эпиком (не в том же PR без необходимости). |
| **next-intl** / i18n middleware | Средняя | Прогон всех локалей, редиректы, статические страницы. |
| **Prisma + libSQL** в Route Handlers | Низкая–средняя | `npm run build`; при ошибках бандлера — `serverExternalPackages` / транзитивные пакеты (в16.1 улучшали поведение Turbopack для transitive externals — всё равно проверить). |
| Изменения в **`next.config`** (`experimental`) | Средняя | Сверить с документацией 16.x: `optimizePackageImports` и прочие флаги. |
| **ESLint** (`eslint-config-next`) | Низкая | Поднять версию конфига синхронно с `next`. |
| Регрессии **Turbopack** в dev | Средняя | Смоук `npm run dev` и при необходимости задокументировать обход (например, временный флаг) в worklog. |
| Поведение **кеша `.next`** | Низкая | Один раз `npm run clean:next` после апгрейда при странных 404/HMR (уже принято в проекте). |

## 4. Фазы работы

### Фаза 0 — Подготовка (без изменения зависимостей)

**Цель:** зафиксировать baseline и убрать шум.

- [ ] Создать ветку `chore/next-16-upgrade` от актуального `main`/`master`.
- [ ] Заполнить **worklog, п. 6.1 «До апгрейда»** (версии, хэш коммита).
- [ ] Локально: `npm ci` → `npm run lint` → `npm run test` → `npm run build` — все зелёные.
- [ ] Краткий смоук UI: главная, авторизация (если используется), один экран с тяжёлым клиентским состоянием (например, игра).

**Критерий выхода:** baseline записан, ветка чистая для bump.

---

### Фаза 1 — Обновление зависимостей (узкий diff)

**Цель:** поднять Next и сопутствующие пакеты минимальным набором.

Рекомендуемый порядок (в одном коммите или двух: deps → lockfile):

1. Обновить **`next`** до целевой версии (например `16.1.1`).
2. Обновить **`eslint-config-next`** до той же мажорной линии (`16.x`).
3. Проверить рекомендации релиза для **`react`** / **`react-dom`** — при необходимости выровнять до версий, указанных в релиз-ноутах Next 16.1 (у вас уже React 19 — часто достаточно минорного выравнивания).
4. Выполнить `npm install` (или `npm ci` после правки `package-lock.json` в PR).

**Не делать в этой фазе:** массовый рефакторинг, смена next-auth на другую мажорную версию без отдельного решения.

**Критерий выхода:** `package.json` + lock обновлены, проект хотя бы запускается (`next dev` или `build` — см. фазу 2).

---

### Фаза 2 — Сборка, типы, линт, тесты

**Цель:** устранить ошибки компиляции и правила ESLint.

- [ ] `npm run type-check`
- [ ] `npm run lint` — исправить новые правила/депрекейты, не отключая проверки без причины (в проекте `ignoreBuildErrors: false`).
- [ ] `npm run test` и `npm run test:coverage`
- [ ] `npm run build`

Типичные правки на этой фазе:

- Импорты/API, помеченные deprecated в 16.x.
- Настройки `next.config.ts` (переименование experimental-опций).
- Типы в `next-auth` / обёртках маршрутов.

**Критерий выхода:** все команды из CI проходят локально.

---

### Фаза 3 — Конфигурация и серверные границы

**Цель:** убедиться, что API routes, Prisma, libsql и внешние пакеты ведут себя предсказуемо.

- [ ] Проверить **Route Handlers** (`src/app/api/**`): сохранение, auth.
- [ ] При ошибках вида «module not found / can't resolve» для нативных или транзитивных зависимостей — настроить `serverExternalPackages` в `next.config.ts` по документации 16.x.
- [ ] Убедиться, что переменные окружения для Turso/Prisma не изменились в поведении (только регрессионный смоук).

**Критерий выхода:** API и облако/локальный сейв работают на dev-сборке; production build без предупреждений, блокирующих релиз.

---

### Фаза 4 — Ручной смоук и dev-UX

**Цель:** поймать то, что CI не ловит.

- [ ] `npm run dev` (и при необходимости `dev:clean` после странностей).
- [ ] Проверить сценарии: холодный старт, навигация, Zustand persist (localStorage), критичные экраны игры.
- [ ] Проверить i18n (next-intl): смена языка, вложенные пути.
- [ ] Зафиксировать в worklog любые аномалии Turbopack/HMR и способ воспроизведения.

**Критерий выхода:** список блокеров пуст или перенесён в отдельные задачи с явным приоритетом.

---

### Фаза 5 — PR, CI, merge и хвосты

**Цель:** безопасно влить изменения.

- [ ] Открыть PR с описанием: версии до/после, ссылка на этот документ, краткий итог worklog.
- [ ] Дождаться зелёного **GitHub Actions** (`lint`, `test`, `test:coverage`, `build`).
- [ ] Merge; тег/релиз-ноуты — по принятому в команде процессу.
- [ ] Обновить **AGENTS.md** (строка про Next.js 15 → 16).
- [ ] Worklog: секция **«После merge»** — дата, версии, ссылка на PR.

---

## 5. Откат (rollback)

1. **До merge:** удалить ветку или `git reset` к baseline; восстановить `package-lock.json` из main.
2. **После merge:** `git revert` merge-коммита; повторный PR с исправлением.
3. Задокументировать причину отката в worklog.
