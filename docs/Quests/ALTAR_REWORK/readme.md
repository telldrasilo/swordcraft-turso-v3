## README: Внедрение строительства алтаря и расширенного квеста

**Версия:** 1.1  
**Для:** разработчики, технические дизайнеры  
**Среда:** Cursor + TypeScript/Next.js/Zustand, возможное использование LLM для генерации кода  

**Канон по шагам квеста ↔ фазам стройки:** [DEVELOPMENT_HUB.md §3.5](./DEVELOPMENT_HUB.md#35-канон-шаг-квеста--фаза-строительства-и-события). Старые таблицы в этом readme (например «фаза I только на шаге 7») не использовать без сверки с §3.5.

---

### 1. Назначение и состав документов

Есть четыре итоговых документа, отвечающие за разработку нового модуля постройки алтаря. Все они согласованы и готовы к реализации. Ниже – их краткое назначение и взаимосвязь.

| Файл | Содержание | Для кого |
|------|------------|----------|
| `ALTAR_BUILDING_V2.md` | Геймдизайн строительства алтаря: 5 фаз, микроэтапы, техники, требования к материалам, UX, отмена, сохранение. | Геймдизайнеры, разработчики UI и store |
| `ALTAR_MATERIAL_BALANCE.md` | Где в коде балансировать количества материалов на старт фаз (`altar-phase-material-balance.ts`), связь фазы I с рецептом плавки. | Геймдизайнеры, баланс |
| `FORGOTTEN_FORGE_v2.md` | Полный сценарий квеста «Эхо забытой кузни» (шаги 0–15) с диалогами, вариантами ответов, техническими обозначениями. | Сценаристы, разработчики логики квеста |
| `CHAT.md` | ТЗ по доработке модуля общения: расширение store, диалоговая система, новые триггеры, интеграция с UI вкладки «Алтарь». | Разработчики store и UI |
| `SPECIFICATION.md` | Техническая спецификация: типы, интерфейсы store, события, хуки, примеры кода. | Все разработчики |

Все документы **не противоречат друг другу** и используют единую нумерацию шагов квеста (0–15, финал 18) и фаз строительства (I–V).

ВАЖНО: идёт разработка модуля алтаря на вкладке проекта "Зачарования". Существующий модуль Алтарь, который находится в Кузнице "Сборка узла зачарований" - нужно полностью отключить и не учитывать в разработке. Это старая версия, из неё для дальнейшей разработки НИЧЕГО брать не нужно. Спланировать полное удаление старого модуля строительства алтаря из проекта.

### 3. Рекомендации по работе с документами в Cursor (с использованием LLM)

1. **Поместите все четыре файла в контекст проекта** (например, в папку `docs/altar-quest/`). В Cursor можно добавить их в индекс для поиска.
2. **Для генерации кода** используйте следующий промпт:
   > «На основе документов ALTAR_BUILDING_V2.md, FORGOTTEN_FORGE_v2.md, CHAT.md, SPECIFICATION.md реализуй [конкретный модуль/компонент/функцию]. Следуй типам из SPECIFICATION.md. Учитывай логику переходов из FORGOTTEN_FORGE_v2.md.»
3. **При работе с store** сначала сгенерируйте типы и начальное состояние, затем по одному actions.
4. **Для UI** попросите LLM сгенерировать компоненты с использованием существующей дизайн-системы (если есть).
5. **После генерации каждого блока** запускайте TypeScript-компилятор и проверяйте интеграцию.

### 4. Возможные доработки и расширения (backlog)

- **Улучшение алтаря** (вкладка «Улучшение алтаря» после финала квеста): можно добавить новые фазы, требующие редких материалов и техник (например, `rune_engraving_advanced`).
- **Система рун** – отдельная механика, где `rune_engraving_basic` открывает доступ к нанесению рун на оружие.
- **Музыкальные артефакты** с использованием `frequency_tuning`.
- **Благословение оружия** через `spirit_blessing`.
- **Новые экспедиции** с уникальными наградами, привязанные к этим техникам.

### 5. Критерии готовности к релизу (приёмочные тесты)

1. Игрок может активировать квест в особых заданиях.
2. Проходит все экспедиции шагов 1–6, получает три артефакта.
3. Начинает фазу I строительства, завершает её.
4. Получает сообщение о фазах II, завершает фазу II.
5. Выполняет крафт оружия, затем три экспедиции за техниками (руны, обжиг, настройка частот).
6. Завершает фазу III, получает задание на благословение, выполняет экспедицию, получает `spirit_blessing`.
7. Завершает фазы IV и V.
8. После фазы V алтарь переключается в режим готового, квест завершён.
9. Все диалоги появляются в канале архивариуса, варианты ответов работают.
10. Панель цели на вкладке «Алтарь» корректно отображает текущую задачу.
11. Прогресс сохраняется при перезагрузке на любом шаге (включая активную фазу строительства).

### 6. Примечания по рискам

- **Синхронизация событий** – `src/lib/game-events.ts`; emit после commit в Zustand (см. крафт / стройку).
- **Списание материалов** – дважды проверьте, что при старте фазы материалы списываются атомарно и при отмене не возвращаются.
- **Техники** – некоторые техники (`clay_firing`, `frequency_tuning`, `spirit_blessing`) могут ещё не быть реализованы в других частях игры. Создайте их заглушки с минимальным описанием.
- **Квестовые экспедиции** – не забудьте добавить в данные экспедиций поля `questTag` и `requiredItems` для проверки ресурсов.
