## README: Внедрение строительства алтаря и расширенного квеста

**Версия:** 1.0  
**Для:** разработчики, технические дизайнеры  
**Среда:** Cursor + TypeScript/Next.js/Zustand, возможное использование LLM для генерации кода  

---

### 1. Назначение и состав документов

Есть четыре итоговых документа, отвечающие за разработку нового модуля постройки алтаря. Все они согласованы и готовы к реализации. Ниже – их краткое назначение и взаимосвязь.

| Файл | Содержание | Для кого |
|------|------------|----------|
| `ALTAR_BUILDING_V2.md` | Геймдизайн строительства алтаря: 5 фаз, микроэтапы, техники, требования к материалам, UX, отмена, сохранение. | Геймдизайнеры, разработчики UI и store |
| `FORGOTTEN_FORGE_v2.md` | Полный сценарий квеста «Эхо забытой кузни» (шаги 0–15) с диалогами, вариантами ответов, техническими обозначениями. | Сценаристы, разработчики логики квеста |
| `CHAT.md` | ТЗ по доработке модуля общения: расширение store, диалоговая система, новые триггеры, интеграция с UI вкладки «Алтарь». | Разработчики store и UI |
| `SPECIFICATION.md` | Техническая спецификация: типы, интерфейсы store, события, хуки, примеры кода. | Все разработчики |

Все документы **не противоречат друг другу** и используют единую нумерацию шагов квеста (0–15, финал 18) и фаз строительства (I–V).

ВАЖНО: идёт разработка модуля алтаря на вкладке проекта "Зачарования". Существующий модуль Алтарь, который находится в Кузнице "Сборка узла зачарований" - нужно полностью отключить и не учитывать в разработке. Это старая версия, из неё для дальнейшей разработки НИЧЕГО брать не нужно. Спланировать полное удаление старого модуля строительства алтаря из проекта.

### 2. Общий план внедрения (рекомендуемый порядок)

Разбиваем работу на **6 этапов**. Каждый этап заканчивается работоспособным инкрементом, который можно протестировать.

#### Этап 0. Подготовка окружения и типизация
- Скопировать все четыре документа в папку `docs/altar-quest/` для удобного доступа.
- На основе `SPECIFICATION.md` создать/расширить TypeScript-типы:
  - `QuestForgottenForgeState`, `AltarConstructionState`, `AltarPhase`, `AltarStage`, `AltarPhaseConfig`.
- Добавить в существующий store недостающие поля (см. `SPECIFICATION.md` п.1.1, 1.2). Пока без реализации действий – только типы и начальные значения.

**Чеклист этапа 0:**
- [ ] Типы определены и экспортируются.
- [ ] Store расширен новыми полями с корректными начальными значениями.
- [ ] Поле `materialStash` имеет поддержку флага `isQuestItem` (если ещё нет – добавить).

#### Этап 1. Система строительства алтаря (базовая механика)
- Реализовать `altarPhasesConfig` для всех пяти фаз (материалы, техники, микроэтапы) по данным из `ALTAR_BUILDING_V2.md`.
- Реализовать actions в store: `startAltarPhase`, `completeAltarPhase`, `cancelAltarPhase`, `updateAltarProgress`.
- Подключить тиковый обновлятор (setInterval/requestAnimationFrame) для активной фазы.
- Реализовать проверку `canStartPhase` и списание материалов `consumeMaterialsForPhase`.
- Создать базовый UI вкладки «Алтарь»:
  - Панель выбора фазы с дорожной картой, требованиями, кнопкой «Начать».
  - Экран строительства (прогресс-бар, таймер, лог).
  - Кнопка отмены с диалогом.

**Чеклист этапа 1:**
- [ ] Конфигурация фаз полностью соответствует таблицам из геймдизайна.
- [ ] Можно запустить фазу I, материалы списываются, прогресс идёт.
- [ ] После завершения фазы генерируется событие `altar:phaseCompleted`.
- [ ] Отмена фазы сбрасывает активное состояние, материалы не возвращаются.
- [ ] Сохранение/восстановление после перезагрузки работает.

#### Этап 2. Расширение квеста (флаги, триггеры, диалоговая система)
- На основе `CHAT.md` и `SPECIFICATION.md`:
  - Расширить store квеста полями `step` и `waitingForCraftAfterPhase2`.
  - Реализовать `setStep`, `setWaitingForCraft`.
  - Создать файл `src/data/quests/forgotten-forge-dialogue.ts` со структурой диалогов по шагам (тексты ответов архивариуса и варианты игрока). Можно начать с заглушек, потом заполнить из `FORGOTTEN_FORGE_v2.md`.
  - Реализовать функцию `selectArchivistChoice(choiceId)`, которая обновляет тред, выполняет эффекты и меняет шаг.
- Добавить подписку на события:
  - `expedition:completed` (уже есть, доработать с учётом `questTag`).
  - `craft:completed` (добавить генерацию события в системе крафта).
  - `altar:phaseCompleted` (уже генерируется на этапе 1).
- Написать хук `useForgottenForgeQuestEvents` и подключить в `_app.tsx` или layout.

**Чеклист этапа 2:**
- [ ] Store квеста корректно сохраняется и загружается.
- [ ] При завершении фазы I шаг меняется с 7 на 8 (по событию).
- [ ] При завершении фазы II устанавливается `waitingForCraftAfterPhase2 = true`.
- [ ] После успешного крафта оружия сбрасывается флаг и шаг переходит на 10.
- [ ] Экспедиции с разными `questTag` ведут к разным переходам и выдают техники.
- [ ] Все переходы по шагам соответствуют сценарию (проверить таблицу в `FORGOTTEN_FORGE_v2.md`).

#### Этап 3. Интеграция UI: панель цели и блокировка фаз
- Реализовать компонент `AltarQuestGoal` (как в `SPECIFICATION.md` п.5).
- Разместить его на вкладке «Алтарь» над дорожной картой.
- Реализовать логику блокировки кнопки «Начать фазу» в зависимости от текущего шага квеста (фаза I доступна только на шаге 7, фаза II – на шаге 8, фаза III – на шаге 13, фаза IV – на шаге 15, фаза V – на шаге 16). Это условие добавляется к уже существующей проверке материалов и техник.
- Добавить подсказки при наведении на заблокированную фазу: «Сначала выполните задание архивариуса».

**Чеклист этапа 3:**
- [ ] Панель цели отображается только на шагах 7–16 и показывает правильный текст (включая динамические подсказки про ресурсы для экспедиций).
- [ ] Кнопка запуска фазы недоступна, если шаг квеста не соответствует ожидаемой фазе.
- [ ] Подсказка появляется при наведении на неактивную кнопку.

#### Этап 4. Выдача предметов и техник, интеграция с инвентарём
- В нужных шагах (см. `FORGOTTEN_FORGE_v2.md`) добавить вызовы `addMaterial` для трёх артефактов и `unlockTechnique` для четырёх техник.
- Убедиться, что артефакты помечены `isQuestItem: true` и не списываются при старте фазы III (проверка только на наличие).
- Добавить в систему экспедиций возможность указывать `requiredItems` (например, `shadow_leather x2` для экспедиции на частоты) и автоматически списывать их при отправке.
- Реализовать в UI цели подсказки, если требуемых ресурсов нет.

**Чеклист этапа 4:**
- [ ] После шага 3 в инвентаре появляется `resonator_matrix`, после шага 5 – `focusing_chalice`, после шага 6 – `lunar_tuning_fork`.
- [ ] Артефакты не тратятся при строительстве.
- [ ] После экспедиций с `questTag` выдаются соответствующие техники.
- [ ] Экспедиция на `frequency_tuning` не позволяет отправиться без `shadow_leather x2`.

#### Этап 5. Завершающий финал и переключение вкладки «Алтарь»
- При достижении шага 17 (завершение фазы V) установить `altarBuiltInForge = true`.
- Убрать панель цели.
- Переключить вкладку «Алтарь» на режим готового алтаря: показать подвкладки «Зачарование» (основной модуль) и «Улучшение алтаря» (заглушка с тизером из диалога).
- Убедиться, что повторные диалоги с архивариусом по этому квесту не запускаются.

**Чеклист этапа 5:**
- [ ] После завершения фазы V квест финализируется, флаг `altarBuiltInForge = true`.
- [ ] Интерфейс алтаря меняется на готовый.
- [ ] Квест не активируется повторно.

#### Этап 6. Тестирование, отладка, баланс
- Пройти весь квест от шага 0 до финала, проверяя все переходы, выдачу техник, списание материалов.
- Проверить сохранение/восстановление на каждом шаге (перезагрузка страницы).
- Проверить краевые случаи: отмена фазы, попытка начать не ту фазу, отсутствие ресурсов для экспедиции.
- Сбалансировать длительности микроэтапов и количества материалов (начальные числа из геймдизайна – ориентир).
- Добавить консольную команду для сброса квеста (например, `window.resetForgottenForgeQuest()` для отладки).

**Чеклист этапа 6:**
- [ ] Все 16 шагов проходятся без зависаний.
- [ ] Сохранение работает после каждого значимого действия.
- [ ] Нет возможности эксплуатировать отмену фаз для дублирования материалов.
- [ ] Баланс длительностей фаз укладывается в 1–3 часа на фазу (субъективно).
- [ ] Логирование переходов в консоль помогает отладке.

### 3. Рекомендации по работе с документами в Cursor (с использованием LLM)

1. **Поместите все четыре файла в контекст проекта** (например, в папку `docs/altar-quest/`). В Cursor можно добавить их в индекс для поиска.
2. **Для генерации кода** используйте следующий промпт:
   > «На основе документов ALTAR_BUILDING_V2.md, FORGOTTEN_FORGE_v2.md, CHAT.md, SPECIFICATION.md реализуй [конкретный модуль/компонент/функцию]. Следуй типам из SPECIFICATION.md. Учитывай логику переходов из FORGOTTEN_FORGE_v2.md.»
3. **При работе с store** сначала сгенерируйте типы и начальное состояние, затем по одному actions.
4. **Для UI** попросите LLM сгенерировать компоненты с использованием существующей дизайн-системы (если есть).
5. **После генерации каждого блока** запускайте TypeScript-компилятор и проверяйте интеграцию.

### 4. Возможные доработки и расширения (backlog)

- **Улучшение алтаря** (вкладка «Улучшение алтаря» после финала квеста): можно добавить новые фазы, требующие редких материалов и техник (например, `rune_engraving_advanced`).
- **Система рун** – отдельная механика, где `rune_engraving_basic` открывает доступ к нанесению рун на оружие.

- **Благословение оружия** через `spirit_blessing`.
- **Новые экспедиции** с уникальными наградами, привязанные к этим техникам.

### 5. Критерии готовности к релизу (приёмочные тесты)

1. Игрок может активировать квест в особых заданиях.
2. Проходит все экспедиции шагов 1–6, получает три артефакта.
3. Начинает фазу I строительства, завершает её.
4. Получает сообщение о фазах II, завершает фазу II.
5. Выполняет крафт оружия, затем три экспедиции за техниками (руны, обжиг, настройка частот).
6. Завершает фазу III, получает задание на благословение, выполняет экспедицию, получает `spirit_blessing`.
7. Завершает фазы IV и V.
8. После фазы V алтарь переключается в режим готового, квест завершён.
9. Все диалоги появляются в канале архивариуса, варианты ответов работают.
10. Панель цели на вкладке «Алтарь» корректно отображает текущую задачу.
11. Прогресс сохраняется при перезагрузке на любом шаге (включая активную фазу строительства).

### 6. Примечания по рискам

- **Синхронизация событий** – убедитесь, что `eventBus` работает глобально и все системы генерируют события в одном потоке (без задержек).
- **Списание материалов** – дважды проверьте, что при старте фазы материалы списываются атомарно и при отмене не возвращаются.
- **Техники** – некоторые техники (`clay_firing`, `frequency_tuning`, `spirit_blessing`) могут ещё не быть реализованы в других частях игры. Создайте их заглушки с минимальным описанием.
- **Квестовые экспедиции** – не забудьте добавить в данные экспедиций поля `questTag` и `requiredItems` для проверки ресурсов.
