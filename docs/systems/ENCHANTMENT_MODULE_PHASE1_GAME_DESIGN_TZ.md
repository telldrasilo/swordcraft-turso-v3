## 1. Продукт: SwordCraft (контекст)

### 1.1. Что это за игра

**SwordCraft** — браузерная игра-симулятор **кузнеца**: игрок собирает оружие из частей и материалов, управляет ресурсами и рабочими, ведёт **гильдию** и отправляет **искателей** в **экспедиции** с готовым оружием. После миссий оружие может **повредиться**; в кузнице игрок **чинит** повреждения техниками и материалами. Параллельно развивается экономика (золото, материалы, заказы NPC).

**Зачарования** в новом каноне — не отдельная абстрактная «магазинная» мини-игра, а слой над **памятью клинка** (шрамы после ремонтов), **душой войны** (опыт оружия в миссиях) и **потенциалом души** (качество сборки при крафте). Старый экран «Алтарь» с жертвоприношениями и «школами» **снят**; новый модуль строится по отдельному канону (см. §3).

### 1.2. Технологии (ограничения UX)

Клиент: **Next.js** (App Router), **React**, **TypeScript**, **Zustand** (состояние), **Tailwind CSS**, **shadcn/ui** (Radix). Типичный интерфейс: боковая навигация по разделам игры, карточки, модальные окна, тёмная «средневековая» эстетика UI. Реализация не требует нативных приложений; ориентир — **веб**, в том числе узкие экраны.

### 1.3. Основной игровой контур (релевантный фазе 1)

| Этап | Суть для дизайна |
|------|------------------|
| Крафт | Сборка оружия из частей; качество, характеристики, **потенциал души** (`soulPotential`) как свойство сборки. |
| Экспедиция | Искатель с оружием игрока; награды, в т.ч. **душа войны** на экземпляр; возможен урон по оружию → **видимые теги повреждений**. |
| Ремонт | Снятие видимых тегов техниками; расход материалов/золота; после успешного ремонта накапливаются **шрамы** (отдельная система, фаза 1 может к ним не подключаться). |
| Гильдия | **Уровень гильдии** открывает **типы локаций** экспедиций (tier 1…4). Это привязано к **гейту** экрана зачарований (см. §4). |

### 1.4. Где живёт игрок в UI (навигация)

В типичной раскладке есть разделы вроде: кузница (крафт, инвентарь, ремонт, **перековка** — отдельная вкладка после «Ремонт»), гильдия (экспедиции, искатели), ресурсы, энциклопедия, магазин. Пункт меню **«Зачарования»** (ранее «Алтарь») **сохранён**; сейчас после «фазы 0» там **заглушка** «модуль в разработке». В фазе 1 нужно заменить заглушку на **осмысленный замок** до условий; механика перековки — на **вкладке «Перековка»** в кузнице (не подменю ремонта).

## 2. Канон нового модуля (выдержки для фазы 1)

Источник правды по продукту: концепция «Пробуждение Души войны». Ниже — только то, что **обязательно учитывать** в фазе 1; полное дерево перков, эссенция, алтарь-постройка — **позже**.

### 2.1. Термины

| Термин | Смысл |
|--------|--------|
| **Душа войны (War Soul)** | Накопленный на **экземпляре оружия** «опыт» миссий. Определяет **тир души** (0–10). Сама по себе не даёт активных умений в бою игрока-кузнеца. |
| **Пре-зачарование** | **До** постройки алтаря **нет эссенции**. Раннее усиление — в **ремонте/перековке**: расход **очков души войны** + **техника** (из прогрессии), **слабый** бонус или примитивный эффект **без** полноценного древа. Знакомство с идеей «развития клинка». |
| **Перековка** | Игровое название того же слоя в кузнице (§5.1): работа над готовым клинком за **душу войны**, отдельно от снятия повреждений. |
| **Экран зачарований (модуль)** | До выполнения условий — **закрыт замком**. Условие уровня прогрессии: **нет доступа к локациям 2-го тира** экспедиций → экран недоступен по смыслу (детали гейта — §4). |
| **Древо перков** | Появляется **только после первого пробуждения** на оружии (отдельная миссия, **не** фаза 1). До пробуждения на экземпляре **нет** дерева — только пре-зачарование в ремонте, если дизайн разрешает. |
| **Алтарь зачарований** | Строительство + особая миссия; открывает эссенцию, распыление, полноценный модуль. Это **фаза 2** реализации, не фаза 1. |

### 2.2. Цепочка прогрессии (фрагмент)

1. Крафт → экспедиции → душа войны (+ позже шрамы через ремонт).  
2. **До алтаря:** пре-зачарование в ремонте — душа войны + техники, слабые усиления.  
3. Тир локаций 2 → позже первое **пробуждение** и древо (не фаза 1).

### 2.3. Что явно **не** входит в фазу 1

- Эссенция души, осколки перков, распыление оружия.  
- Постройка алтаря, особая миссия открытия алтаря.  
- Генерация перков из шрамов, три варианта на шаг, ветвления древа.  
- Миграция старых сохранений игроков (проект в разработке).

### 2.4. Связь с «шрамами» и стихиями

В полном каноне перки и роллы завязаны на **шрамы** (память клинка) и **оси стихий**. Для **фазы 1** допустимо **не** усложнять пре-зачарование шрамами: достаточно **души войны + техника + простой эффект**, если команда не готова связывать слой с платформой стихий. Если дизайн захочет опору на шрамы — это отдельное решение с оценкой объёма; в ТЗ заложить как **опцию**.

## 3. Состояние проекта после «фазы 0» (факт для дизайна)

- Старый UI алтаря удалён; экран **«Зачарования»** — текстовая заглушка с отсылкой к документации.  
- В типах данных заложены **заготовки** под древо (пробуждения, шаги) — **без** игровой логики.  
- Старые действия store старого алтаря могут ещё существовать в коде **без UI**; новый контент их **не использует**.  
- Игроки в разработке допускают **сброс сейва** при смене схемы.

Для внешнего дизайнера: **не опираться** на старый алтарь как референс экрана.

## 4. Гейт доступа к экрану «Зачарования»

### 4.1. Продуктовое правило (канон)

Модуль **не считается открытым**, пока у игрока **нет доступа к локациям экспедиций 2-го тира**.

### 4.2. Как это сопоставить с прогрессией (техническая опора для геймдизайна)

В реализации SwordCraft доступ к tier локациям завязан на **уровень гильдии**. В текущих данных ориентир такой: при низком уровне гильдии доступны только локации **tier 1**; по мере роста открываются **tier 2**, затем 3 и 4. **Порог для tier 2** нужно трактовать как **единое игровое правило** (например «гильдия 3-го уровня» — цифра для согласования с балансом; финальное число фиксирует геймдизайн после плейтеста).

**Задача дизайна:** сформулировать для игрока **понятное условие** (не только «tier 2», но и **как этого достичь**: уровень гильдии, квест, здание — если вводится доп. условие, оно должно быть согласовано с остальной прогрессией).

### 4.3. Состояния экрана «Зачарования»

| Состояние | Описание для UX |
|-----------|------------------|
| **Замок** | Игрок видит, что раздел существует, но контент недоступен; объяснение **зачем** открывать (мотивация), **что** получит позже (без спойлеров фазы 2 или с контролируемым тизером). |
| **После гейта, до фазы 2 кода** | Может отображаться **промежуточный** экран: «Экономика древа и алтарь — в следующем обновлении» или тизер визуала древа **без** интерактива — **решение продукта**; зафиксировать в макетах. |

## 5. Пре-зачарование, перековка и техники (ядро геймдизайна фазы 1)

### 5.1. Перековка и отличие от ремонта

В каноне **пре-зачарование** и **перековка** — это один ранний слой усиления оружия **до** алтаря и древа: работа кузнеца над **уже готовым** экземпляром, не меняющая рецепт сборки частей, но оставляющая **след в данных экземпляра** (слабый бонус / флаг / модификатор).

| Аспект | **Ремонт** (текущая игра) | **Перековка / пре-зачарование** (фаза 1) |
|--------|---------------------------|------------------------------------------|
| **Цель** | Убрать **видимые** повреждения (теги), вернуть работоспособность клинка. | **Усилить** экземпляр без нового крафта: «настроить дух металла», закрепить слабый эффект. |
| **Основной ресурс** | Материалы склада, золото (по плану ремонта), время этапов. | **Душа войны** на этом оружии — **обязательный** расходуемый ресурс фазы 1. |
| **Роль техник** | Техники **починки**: привязка к тегам урона (`clearsTagIds`), этапы, стоимость материалов. | Отдельный контур выбора техники (см. §5.5–5.6): не путать с «снятием тега», если только дизайн не введёт **гибрид** (напр. перековка доступна после успешного ремонта конкретным методом). |
| **Место в UI** | Карточка ремонта, выбор техник починки, осмотр. | Отдельная вкладка кузницы **«Перековка»** сразу после **«Ремонт»** (тот же клинок на верстаке `repairBenchWeaponId`). |

**Задачи для копирайтинга:** заголовок вкладки — **«Перековка»**; короткая **лор-строка**, почему тратится именно душа войны (опыт клинка как «топливо» символической перековки).

### 5.2. Минимальный геймплейный контракт (из канона и инженерного плана)

| Элемент | Требование |
|---------|------------|
| **Ресурс** | Списание **души войны** с **конкретного экземпляра** оружия (не ниже нуля; верхняя граница — `maxWarSoul`, если используется). |
| **Инструмент** | Выбор **техники перековки** — из согласованного набора (см. §5.5–5.6); это **не** обязательно те же записи, что техники починки 1:1. |
| **Эффект** | **Простой** модификатор: числовой бонус к уже существующим полям (атака, прочность, износ, мета экспедиции и т.д.) или отдельный флаг «базовое усиление» — **решение геймдизайна**. |
| **Ограничения** | Без эссенции; без шагов древа; возможен **лимит** применений на экземпляр или кулдаун — **решение геймдизайна**. |

### 5.3. Душа войны: тиры (числовой ориентир для баланса)

Тир души определяется по накопленным очкам на экземпляре. Примерная таблица из проектного брифа (цифры могут уточняться балансом):

| tier | Название (пример RU) | Диапазон warSoul (мин–макс) | Примеры бонусов тира (% к успеху / золоту / приросту души / криту) |
|------|----------------------|------------------------------|-------------------------------------------------------------------|
| 0 | Искра | 0 – 500 | 0 / 0 / 0 / 0 |
| 1 | Тлеющая | 500 – 2 000 | 2 / 3 / 2 / 0 |
| 2 | Горящая | 2 000 – 8 000 | 4 / 5 / 3 / 1 |
| … | … | … | … |
| 10 | Великая Душа Войны | 250 000 – ∞ | 25 / 30 / 25 / 10 |

Пре-зачарование **логично связывать** с тиром или с «сырыми» очками — на усмотрение геймдизайна (например стоимость в warSoul зависит от тира, чтобы не ломать раннюю игру).

**Взаимосвязь с тиром (дизайн-риск):** трата `warSoul` **снижает** накопленную душу и может **понизить тир** души войны на экземпляре. Нужно явно решить: (а) перековка списывает душу и тир падает — стратегический выбор; (б) списание идёт из отдельного «пула перековки» (требует нового поля — не канон фазы 1); (в) списание фиксировано мало и не сдвигает тир в типичных диапазонах — баланс таблицей.

### 5.4. Потенциал души (`soulPotential`)

При крафте на оружии задаётся множитель **эффективности** получения души за миссию (ориентир диапазона **0.85…3.0**, по умолчанию **1.0** для старых сейвов). Для пре-зачарования можно: **не трогать** в фазе 1, **или** использовать как модификатор **силы эффекта** перековки (логично по смыслу «качества резонанса» клинка) — зафиксировать в таблице баланса.

### 5.5. Две системы «техник» в SwordCraft и место перековки

В продукте уже сосуществуют **разные** сущности с похожим словом «техника». Для ТЗ важно не смешивать их в одной таблице баланса.

| Система | Назначение | Типичные поля / правила |
|---------|------------|-------------------------|
| **Техники обработки (крафт)** | Этапы обработки материалов и частей при **создании** оружия; влияние на характеристики и синергии. | Уровень кузницы, материалы для разблокировки, источник (старт / гильдия / экспедиция), экспертиза по материалам. |
| **Техники починки** | Снятие **видимых** тегов повреждений на экземпляре; этапы с длительностью; стоимость материалов и золота. | Привязка к тегам урона (`clearsTagIds`), стадии `preparation` / `work` / `finishing`. **Душа войны** при починке сейчас не является базовой валютой шага (могут быть другие правила износа — см. код ремонта). |

**Перековка (фаза 1)** по канону опирается на **душу войны + технику**. Возможные модели **связи** с существующими данными (выбрать одну или комбинацию — зафиксировать в дизайн-доке):

1. **Новый подреестр «техники перековки»** — отдельные id, тексты, стоимость в `warSoul`, опционально золото/материалы; **нет** привязки к `clearsTagIds` (это не починка). Минимальный риск путаницы в коде и для игрока.  
2. **Параллель с техниками починки по имени/теме** — например «Правка кромки» как ремонт трещин кромки и как **отдельная** операция перековки с другим эффектом; в данных — **разные id** или флаг `usage: 'repair' | 'reforge'`, чтобы UI не смешивал расчёты.  
3. **Опора на техники крафта** — перековка использует id из **обработки**, если игрок **разблокировал** технику для кузницы; смысл: «применить мастерство обработки к готовому клинку». Тогда условия разблокировки те же, что у крафта + доп. порог по гильдии/душe.  
4. **Гибрид** — часть перековок открывается экспертизой по материалам, часть — рангом гильдии, часть — уникальные рецепты перековки.

**Обязательный выход дизайна:** матрица «техника перековки → источник разблокировки → стоимость warSoul (+ опц. материалы) → эффект».

### 5.6. Требования к новым техникам перековки

Ниже — чеклист для **новых** записей (если не переиспользуются только существующие id с новым контекстом).

| Требование | Описание |
|------------|----------|
| **Идентичность** | Уникальный `id`, не пересекающийся с ремонтными/крафтовыми без явного маппинга в данных. |
| **Нарратив** | 1–2 предложения: что делается с клинком **метафорически** (настройка резонанса, прожиг канала и т.д.), без противоречия лору «душа = опыт миссий». |
| **Разблокировка** | Явные условия: уровень гильдии, уровень кузницы, экспертиза по материалу X, завершённый рецепт, флаг «первая перековка на аккаунте» — на выбор, но **документировано**. |
| **Стоимость в душе войны** | Минимальное и максимальное ожидаемое списание `warSoul`; согласование с таблицей тиров §5.3 (чтобы игрок не «обнулял» клинок одной кнопкой без осознанности). |
| **Вторичные ресурсы** | Нужны ли золото / слитки / травы **дополнительно** к душе — если да, обосновать (экономика склада vs эксклюзивность души). |
| **Эффект** | Тип модификатора, числа, стек с другими баффами, лимит применений на экземпляр. |
| **UX** | Короткое имя для кнопки + длинное описание в подсказке; иконка или цвет категории (физика / стихия / универсал) — по стилю игры. |
| **Совместимость** | Не блокирует введение древа и эссенции во **фазе 2** (нет жёсткого «вечного» преимущества, ломающего баланс дерева). |

**Повторное использование существующих техник починки** допустимо только при явном правиле в данных (например «после успешного ремонта этой техникой можно применить перековку с тем же сюжетом») — иначе игрок увидит один и тот же id в двух разных контекстах с разной ценой и запутается.

### 5.7. Ресурсы: стек, приоритеты и взаимосвязи

| Ресурс | Роль в перековке фазы 1 | Заметки |
|--------|-------------------------|---------|
| **Душа войны (`warSoul`)** | **Основной** оплачиваемый ресурс; снимается с **экземпляра**. Источник пополнения — преимущественно **экспедиции** (формула награды уже завязана на `soulPotential` и тир души). | Связь с прогрессией: трата снижает запас на клинке → влияет на тир души (см. §5.3). |
| **Золото** | Опционально: доплата за «инструменты мастера» или налог мастерской — **решение баланса**. | В ремонте золото уже используется в планах; не дублировать без смысла. |
| **Материалы склада** | Опционально: расход слитков/древесины как **второй** слой стоимости рядом с душой. | Усиливает связь с экономикой базы; усложняет UI (два ряда стоимости). |
| **Экспертиза материалов / техник** | Может быть **гейтом** («перековка доступна, если экспертиза по стали ≥ N») без доп. траты. | Согласовать с системой экспертизы кузницы. |
| **Эссенция / осколки** | **Не используются** в фазе 1 по канону. | Не показывать в UI перековки. |

**Приоритет списания (для единообразия UI и кода):** рекомендуется зафиксировать порядок: сначала проверка **души войны** и списание, затем — если дизайн ввёл — материалы и золото; либо **одновременная** проверка всех с отказом «не хватает: X».  

**Взаимосвязь с техниками починки по ресурсам:** ремонт тратит материалы/время; перековка — **в первую очередь душу**. Одна и та же сессия карточки оружия может включать **сначала** ремонт (если есть теги), **затем** перековку — порядок и блокировки («нельзя перековать пока есть незакрытый критический тег») задаёт геймдизайн.

## 6. Задачи для геймдизайна (что выдать артефактами)

### 6.1. Гейт экрана «Зачарования»

1. **Текст условия открытия** (короткий + развёрнутый вариант для подсказки).  
2. **Тон коммуникации**: суровый кузнечный / мистический / нейтральный — выбрать и применить ко всем строкам раздела.  
3. **Тизер будущего**: что обещаем игроку (древо, алтарь) без ложных обещаний по срокам.  
4. **Edge cases**: что показывать, если игрок «читерит» прогресс (если возможно) или откатился сейв — по желанию одна строка FAQ для саппорта.  

### 6.2. Перековка и пре-зачарование

1. **Название фичи** для UI (рабочие варианты 3–5 + финал): не путать с «ремонтом» и «зачарованием» старого алтаря; согласовать с §5.1.  
2. **Триггер в UX**: отдельная секция на карточке ремонта / отдельный шаг мастера / вкладка — выбрать и обосновать; учесть порядок с ремонтом (§5.7).  
3. **Экономика одного применения:**  
   - фиксированная цена в warSoul или формула; влияние на **тир души** при списании (§5.3);  
   - влияет ли качество оружия / тир души / редкость;  
   - вторичные ресурсы: золото и материалы — да/нет (§5.7);  
   - можно ли комбинировать с обычным ремонтом в один «визит» или строгая последовательность.  
4. **Матрица техник:** по модели из §5.5 (новый подреестр / параллель с ремонтом / опора на крафт / гибрид) — заполнить колонки для каждой техники перековки.  
5. **Каталог эффектов фазы 1:** 3–10 вариантов (достаточно для разнообразия), с типом эффекта, числами, ограничениями, привязкой к технике (1:1 или пул); требования к новым записям — §5.6.  
6. **Прогрессия:** что открывается с уровнем гильдии / экспертизой / рецептами / разблокировкой крафтовых техник.  
7. **Анти-эксплойт:** нельзя бесконечно накачивать одно оружие без экспедиций — если душа только с миссий, описать ожидаемый темп.  

### 6.3. Согласование с будущими фазами

Кратко описать, **как** пре-зачарование соотносится с будущим древом (стакается / перекрывается / конвертируется), чтобы не закрыть дорогу дизайну фазы 2–3.

## 7. Задачи для UX/UI

### 7.1. Экран «Зачарования» (замок и пост-гейт)

- **Wireframe** мобильной и десктопной ширины: иерархия заголовка, иконки замка, CTA (если есть — например «в гильдию»), вторичные ссылки.  
- **Визуальная метафора** замка: иллюстрация vs иконка — согласовать с арт-направлением игры.  
- **Состояния:** только замок; замок + баннер «скоро»; разблокировано но пусто (заглушка фазы 2).  
- **Доступность:** контраст, читаемость текста на тёмном фоне, длина строк для локализации (RU + запас под EN).

### 7.2. Вкладка «Перековка» (кузница)

- Отдельная вкладка **после** «Ремонт»; не внутри `RepairCard`. Визуально отделить **ремонт** (теги урона) и **перековку** (душа войны + техника перековки).  
- **Форма выбора:** техника перековки (список/грид), стоимость **warSoul** (+ опционально золото/материалы по §5.7), превью эффекта, кнопка подтверждения, ошибки (недостаточно души, техника заблокирована, не выполнены условия разблокировки).  
- **Обратная связь после применения:** тост, анимация, обновление чисел на карточке оружия.  
- **Согласованность** с существующими компонентами (карточки, бейджи, прогресс-бары прочности).  
- **Микрокопирайтинг:** подсказки «что такое душа войны» в один тап (глоссарий/tooltip).

### 7.3. Доставляемые артефакты (рекомендуемый минимум)

| Артефакт | Назначение |
|----------|------------|
| User flow (схема) | Открытие кузницы → выбор оружия → ремонт → пре-зачарование → результат. |
| Wireframes | 2–4 ключевых экрана/состояния. |
| UI-kit заметки | Отступы, типографика, цвета статусов (успех/ошибка/замок). |
| Таблица копирайтинга | Все строки UI в одном месте для передачи в разработку. |
| Баланс-таблица | Эффекты, стоимости, лимиты — для импорта в `src/data/` позже. |

## 8. Критерии приёмки дизайна (фаза 1)

Ниже — **для внешней согласования**, не для CI.

- [ ] Условие гейта понятно **новому игроку** без чтения вики.  
- [ ] Перековка не путается с обычным ремонтом и не выглядит как старый алтарь; техники перековки отличимы от починки и от крафта (§5.5).  
- [ ] Экономика пре-зачарования согласована с темпом получения warSoul из экспедиций (порядок величин).  
- [ ] Все обязательные состояния UI (замок, ошибка, успех, блокировка техники) описаны.  
- [ ] Есть план совместимости с фазой 2 (алтарь, эссенция) без противоречий.  
- [ ] Макеты/копирайт передаваемы разработке без устных дополнений.

## 9. Риски и открытые вопросы

1. **Гейт только по tier 2** или дополнительные условия (квест, здание) — влияет на тексты и туториал.

4. **Монетизация/донат** — в документе не предполагается; при появлении ограничений — отдельное решение.

## 10. Глоссарий (кратко)

| Термин | Пояснение |
|--------|-----------|
| Экземпляр оружия | Конкретный меч в инвентаре с уникальным id. |
| Перековка / пре-зачарование (фаза 1) | Усиление готового экземпляра за **душу войны** и **технику перековки**; не снимает теги урона само по себе. |
| Ремонт | Снятие видимых тегов повреждений техниками **починки**; материалы/золото/этапы, не путать с перековкой. |
| Техника обработки (крафт) | Техника этапов **создания** частей оружия; разблокировки из прогрессии кузницы/гильдии. |
| Техника починки | Запись в реестре ремонта: какие теги снимает, стадии, материалы. |
| Техника перековки | Запись в дизайне фазы 1: эффект перековки + стоимость в **warSoul** (+ опц. другие ресурсы); задаётся в матрице §5.5–5.6. |
| Душа войны (`warSoul`) | Накопленные очки на экземпляре; тиры §5.3; основной ресурс перековки. |
| Tier локации | Уровень сложности/контента экспедиции (1–4). |
| Уровень гильдии | Прогрессия гильдии; открывает доступ к локациям. |
| Видимый тег повреждения | Что игрок чинит в ремонте; не путать со шрамом. |
| Шрам | Память клинка после успешного ремонта; для полного модуля зачарований — вес для перков. |
