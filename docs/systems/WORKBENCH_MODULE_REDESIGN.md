# Верстак: концепция v2 (единая вкладка, очередь ремонт + перековка, без «изъятия» оружия)

Документ описывает целевой UX и модель данных **на уровне спецификации** (реализация в коде — отдельными задачами).

**Текущее состояние кода (апрель 2026):** в кузнице одна вкладка **«Верстак»** (`forgeMainTab: 'bench'`) с подвкладками **«Ремонт»** / **«Перековка»** (`forgeBenchSubTab`). Единая очередь **`workbenchQueue`**; общий таймер и полоса — [`workbench-shell.tsx`](../../src/components/forge/workbench-shell.tsx), [`workbench-bar-baseline.ts`](../../src/lib/workbench/workbench-bar-baseline.ts). Поле **`repairBenchWeaponIds`** пока сохраняется для совместимости сейвов и отдельных UX-путей ремонта.

**Разрыв с целевым UX (зафиксировать в следующих итерациях):**

1. **Единый инвентарь.** Сейчас **ремонт** и **перековка** по-прежнему отрисовывают **разные** экраны (`RepairSection` / `ReforgeSection`) со **своими** списками/фильтрами оружия; игрок не видит **одну** общую колонку «все клинки» и переключение только режима работы (планирование ремонта vs перековки). **Целевое состояние (§2, §4):** один источник списка — `weaponInventory.weapons` (или общий презентационный компонент), один скролл/поиск; подвкладка меняет только панель действий и допустимые операции, а не дублирует инвентарь.
2. **Место подвкладок «Ремонт / Перековка».** Сейчас в [`forge-screen.tsx`](../../src/components/screens/forge-screen.tsx) переключатель подвкладок стоит **в одной строке** с кнопкой основной вкладки «Верстак» (рядом с Крафт / Инвентарь). **Целевое состояние:** в верхней строке кузницы — только выбор **«Верстак»** как целого режима; сегмент **Ремонт | Перековка** переносится **внутрь** интерфейса верстака (шапка карточки / `WorkbenchShell` / первая зона контента), чтобы визуально иерархия была «кузница → верстак → режим работы», а не три уровня в одном `flex` ряду.

**Оглавление:** §1–4 — цели и сценарии; §5 — полоса прогресса и панель этапов; §6 — зафиксированные решения; §7 — затрагиваемый код; §8 — риски; §9 — worklog.

## 1. Цель продукта

Игрок открывает **вкладку «Верстак»** в кузнице и **не перекладывает** клинки из инвентаря на отдельный слот:

- выбирает **любое** оружие из общего списка;
- понимает: «сейчас чиню шрамы / усиляю дух / пробуждаю наследие / всё запланировал подряд»;
- собирает **одну очередь работ** (ремонт и перековка), запускает сессию, видит **общую полосу прогресса** по всему пакету и **детальную панель этапов** текущего пункта;
- удерживает ментальную модель: **один кузнец у одного верстака** — в каждый момент времени выполняется **не больше одной** операции на всю кузницу (§6.2).

Инвентарь остаётся единственным источником правды: оружие не «изымается» в отдельную сущность. Ограничения на экспедиции и заказы — в §6.1.

## 3. Модель «верстак без изъятия»

### 3.1. Отказ от семантики «на верстаке = недоступно вне кузницы»

- **Убрать** использование `repairBenchWeaponIds` как списка изъятых id (или оставить поле пустым после миграции).
- Опционально: `benchSelectedWeaponId: string | null` для развёрнутой карточки в UI.

### 3.2. Единая очередь `workbenchQueue` (рабочее имя)

По смыслу заменяет `repairQueuePlan`, элементы строго типизированы:

```ts
type WorkbenchQueueItem =
  | { kind: 'repair'; queueItemId: string; weaponId: string; techniqueIds: string[]; executionOpts?: ...; status; queuedAt; ... }
  | { kind: 'reforge_buff'; queueItemId: string; weaponId: string; techniqueId: string; status; queuedAt; ... }
  | { kind: 'reforge_awaken'; queueItemId: string; weaponId: string; techniqueId: string; status; queuedAt; ... }
```

**Статусы:** `planned` | `running` | `done` | `error` (как в текущем ремонте).

**Порядок исполнения:** строго **в порядке добавления** в очередь (§6.3). Несколько пунктов с одним `weaponId` подряд допустимы; уникальный идентификатор позиции — `queueItemId`.

### 3.3. Ремонт в очереди

- Сохранить многоэтапный таймер: [`RepairTechniqueStageRunState`](../../src/store/slices/craft-slice.ts), `beginRepairTechniqueStageRun`, [`useWeaponRepairStageRun`](../../src/hooks/use-weapon-repair-stage-run.ts).
- План этапов и длительности — как сейчас: [`buildWeaponRepairPlan`](../../src/lib/weapon-damage/build-repair-plan.ts), [`repair-stage-timing`](../../src/lib/weapon-damage/repair-stage-timing.ts).
- Старт пункта `repair` **не** зависит от `repairBenchWeaponIds`.

### 3.4. Перековка в очереди — процесс во времени

**Решение:** по UX и длительности перековка **приравнивается к ремонту**: у техники задаётся **план этапов** (вплоть до одного этапа) с суммарной длительностью; после завершения последнего этапа один раз вызывается [`applyReforgeTechniquePure`](../../src/lib/reforge/apply.ts) (или эквивалент в store) на **актуальном** состоянии оружия.

- **Данные:** расширить [`reforge-techniques-registry.ts`](../../src/data/reforge/reforge-techniques-registry.ts) или вынести планы в `reforge-stage-plans.ts`: стадии `{ id, name, durationMs | durationSec, category? }` по аналогии с ремонтом, либо сгенерированный план из N равных тактов.
- **Рантайм:** обобщённый `WorkbenchTechniqueStageRunState` или отдельное `reforgeTechniqueStageRun` с теми же идеями, что у ремонта (`weaponId`, набор техник / одна техника, этап, `startedAt`, `source: 'queue'`).
- **UI:** тот же паттерн, что [`RepairStageProgressPanel`](../../src/components/ui/repair-stage-progress-panel.tsx); целевое имя компонента, например `BenchStageProgressPanel`, с пропсами для подписей «Ремонт» / «Перековка».

**Пробуждение:** предпросмотр шанса и списания ДВ (§6.4); окончательный бросок — при `apply` после последнего этапа.

## 4. UX-поток

1. **Кузница → Верстак** открывает оболочку верстака; **внутри неё** игрок переключает **Ремонт / Перековка**. Один и тот же **список оружия** (сплошной инвентарь клинков); карточка клинка → **«В очередь»** / опционально **«Сделать сейчас»** в зависимости от активной подвкладки.
2. **Панель очереди:** отображается порядок пунктов; **удаление** — только для позиций в статусе `planned` и в рамках §6.5; кнопка **«Начать работу»** запускает сессию.
3. **Во время сессии** (§5): сверху — **сегментированная полоса** по всему пакету; ниже — **панель этапов** текущего пункта.
4. **Карточка оружия:** краткий список запланированных для него шагов (опционально).

**Формулировка для игрока:** верстак — место **последовательного** выполнения работ **одним мастером**, а не «ячейка, куда пропадает меч».

### 4.1. Диалог «оружие в очереди → экспедиция» (§6.1)

- **Заголовок** — понятный игроку, без имён полей из кода.
- **Текст:** клинок запланирован в очереди верстака; при желании — число **запланированных** пунктов с этим `weaponId`, без перегрузки цифрами.
- **Действия:** основная кнопка **«Убрать из очереди и выбрать»**, вторичная **«Оставить в очереди»**. Клавиша `Escape` и клик вне окна (если предусмотрен) **не** меняют очередь.
- После подтверждения основного действия удаляются **все** позиции со статусом `planned` и выбранным `weaponId` (§6.1); далее оружие можно назначить на экспедицию как обычно.

## 5. Полоса суммарного прогресса и панель этапов

### 5.1. Назначение

Одна полоса отражает **весь запущенный пакет** очереди: сколько **общего времени** приходится на каждый **пункт** очереди (не на «оружие» абстрактно: один и тот же клинок дважды в очереди даёт **два сегмента**).

**Пример:** три пункта «ремонт» (три разных клинка или один и тот же трижды — для геометрии полосы неважно) и один пункт «перековка» — **четыре сегмента**. Доля ширины сегмента с индексом *i* = (оценка времени пункта *i*) / (сумма оценок по всем пунктам на момент старта сессии).

### 5.2. Расчёт долей

- **`repair`:** сумма длительностей этапов по плану + учёт финального шага, если он учитывается в UX-таймлайне.
- **`reforge_*`:** сумма длительностей этапов из плана перековки (§3.4).
- **Цвета:** различимые сегменты (по индексу или хешу `weaponId`); **активный** сегмент визуально выделен.
- **Заполнение:** для завершённых пунктов сегменты «закрыты»; для текущего — пропорционально прогрессу этапов внутри пункта; общий fill полосы = взвешенная сумма.

### 5.3. Подсказка при наведении

У каждого сегмента — **подсказка** (на MVP — атрибут `title`): полное имя оружия, тип операции (ремонт / усиление / пробуждение), **номер пункта** в очереди (1…N). Длинные имена в списке можно обрезать с «…», в подсказке показывать целиком.

### 5.4. Размещение панели этапов

**Под** суммарной полосой — обобщённая **[`RepairStageProgressPanel`](../../src/components/ui/repair-stage-progress-panel.tsx)** (ремонт или перековка): текущий пункт, прогресс по этапам, список шагов. Реализация должна жить **только внутри** контента верстака (вместе с [`WorkbenchShell`](../../src/components/forge/workbench-shell.tsx)), а не в отдельной «странице» без общей оболочки; при **едином инвентаре** панель очереди и этапов одинаково доступна с обеих подвкладок **Ремонт** и **Перековка** при активной сессии.

**Доступность:** контраст полосы и панели — не ниже привычных для UI кузницы; на сенсорных экранах не полагаться только на hover — tap по сегменту или отдельная легенда под полосой.

## 6. Зафиксированные продуктовые решения

| № | Тема | Решение |
|---|------|---------|
| **6.1** | **Экспедиции и очередь** | **Сессия запущена** (после «Начать работу», пока не обработаны все пункты текущего пакета): если у оружия есть пункты в статусе **`planned` или `running`** в этой сессии, клинок в экспедицию **нельзя** — в списке отображается, строка **неактивна** с короткой причиной. Пункты со статусом **`done`** для этого `weaponId` блокировку не продлевают — клинок снова можно отправить в поход, если нет других ограничений (прочность, состояние ремонта и т. п.). **Сессия не запущена**, оружие только в очереди (все пункты ещё `planned`): в планировании экспедиции показываются **значок и подпись «в очереди»**. При **выборе** такого клинка — **диалог** (текст и кнопки — §4.1). **Подтверждение:** из очереди удаляются **все** позиции со статусом **`planned`** и с выбранным **`weaponId`**, чтобы не осталось «висящих» шагов для клинка, который уходит в поход. **Отмена** диалога — очередь **без изменений**. **Заказы NPC:** те же правила, что и для экспедиций (блок при `planned`/`running` в активной сессии верстака; диалог и снятие всех `planned` по id при принудительном выборе, если такой сценарий есть в UI заказа). |
| **6.2** | **Параллелизм** | Ровно **одна** активная цепочка этапов на всю кузницу: «топор → меч → копьё», без второго параллельного таймера. |
| **6.3** | **Порядок работ** | Строго **как добавлено в очередь**. Обязателен аудит игровых правил — §8. |
| **6.4** | **Пробуждение и предпросмотр** | Перед финальным применением — **предпросмотр** шанса и пояснение списания ДВ; данные от **актуальной** ДВ на клинке на момент показа. Финальный ролл — при `apply` после последнего этапа. |
| **6.5** | **Отмена** | Пока у позиции очереди для данного оружия статус **`running`**, **отменить эту работу нельзя**. Позиции по **другим** `weaponId` в статусе `planned` **можно удалить**. Полный сброс текущего таймера «в ноль» **не предусмотрен** (как в текущей очереди ремонта), пока нет отдельного решения. |
| **6.6** | **Длительность перековки** | Как у ремонта: данные этапов + время, реализация в рамках модуля (§3.4). |
| **6.7** | **Миграция сейвов** | Устаревшие `repairBenchWeaponIds`: **сброс** в пустой массив при загрузке, перенос очереди по правилам W2 (§9). |
| **6.8** | **Туториал** | **Два шага** в [`tutorial.ts`](../../src/data/tutorial.ts): (1) вкладка «Верстак», очередь и запуск работы; (2) второй подрежим — «Перековка» или «Ремонт» (тексты задаются после финализации макета). |

## 8. Риски и проверки логики (к §6.3)

Чеклист перед мержем логики очереди:

1. **Между пунктами:** после перековки меняются статы и ДВ; следующий «ремонт» пересобирает план от **актуального** оружия на **старте** пункта, не от снимка при добавлении в очередь.
2. **Перековка → ремонт:** изменение максимальной прочности и пересчёт текущей прочности — план ремонта остаётся валидным либо даётся явная ошибка выполнения.
3. **Ремонт → перековка:** после ремонта меняются шрамы/теги — проверка лимитов стаков и пробуждения на старте пункта перековки.
4. **Несколько перековок подряд на одном клинке:** стоимость и допустимость второго `buff` проверять на **старте** пункта; при невозможности — понятное сообщение об ошибке или запрет добавления при планировании (зафиксировать при реализации).
5. **Полоса прогресса:** до старта сессии пересчитывать доли при удалении `planned`; после «Начать работу» — **зафиксировать** baseline долей и суммарного времени, чтобы не «прыгала» геометрия при удалении чужих `planned` (§6.5).
6. **Диалог §6.1:** удаление только `planned` для id; пункты в `running` для этого id в активной сессии по определению не дают выбрать клинок в экспедицию — конфликта «удалить и всё равно в походе» нет.
