# Платформа осей урона: память оружия для зачарований **v1.0**

**Статус:** **источник знаний** (канон дизайна и терминов по осям урона, шрамам, стихиям и связке с экспедициями). Практическое внедрение ведётся по рабочему документу [ELEMENTAL_PLATFORM_IMPLEMENTATION.md](./ELEMENTAL_PLATFORM_IMPLEMENTATION.md).

Игра в разработке; **обратная совместимость старых сохранений не требуется**.

**Канонические артефакты в этом файле:**

- **§3.5 — чеклист покрытия:** актуальный перечень «ось / тэг повреждения → события → локации»; по мере появления шаблонов событий и правок локаций таблицу **обновлять здесь** (и дублировать смысл в коде).
- **§3.6 — словарь отображения стихий:** канон имён осей для UI (RU); реализация в коде должна **соответствовать** этой таблице (один источник смысла — здесь; в коде — экспорт/константа без расхождений строк).

## 1. Логика цепочки (единая для физики и стихий)

Каждая строка в таблицах §2–§3 проходит одну и ту же цепочку:

1. Оружие получает **видимое повреждение** в экспедиции (тэг: `physical_*` или `elemental_*`). **Источник — только случайные события миссии**, в которых сработал эффект урона по оружию (`damage_weapon` и маппинг на тэги); не начислять повреждения «в конце миссии» отдельно от событий. В рантайме снимки событий копятся в ходе миссии, а запись в `activeDamageTags` на экземпляр выполняется при **завершении** экспедиции из этих снимков — семантически тэг всё равно привязан к **конкретному прошедшему событию**, а не к итоговому экрану.
2. Игрок-кузнец чинит и **снимает** видимый тэг в кузнице (техники ремонта, склад).
3. После успешной починки на оружии фиксируется **шрам** — память клинка для зачарований и распределения весов (см. §1.1).
4. Ключи шрамов использует модуль зачарований (когда будет подключён).

**Идентификаторы шрамов:**

- **Стихии:** после снятия `elemental_*` в память идёт **короткий id оси** (`fire`, `water`, `frost`, …) — тот же id, что в колонке «Шрам» у таблицы §3.
- **Физика:** после снятия `physical_*` в память идёт **id шрама** с префиксом `scar_*` (`scar_impact_brutal`, …) — колонка «Шрам» в таблице §2.

Термин **«скрытый тег»** в документации не используем: везде **шрам**. Не путать с `CraftedWeaponV2.hiddenTags` (поиск оружия для заказов NPC).

### 1.1. Шрамы: накопление, вес, топ-3 + топ-3

Все **шрамы** после починки — это **не** активные поломки, а **накопительная память** экземпляра.

- **Вес:** при каждом успешном снятии соответствующего видимого повреждения счётчик по этому ключу шрама **увеличивается на 1**. Пример: четыре раза сняли морозный тэг → в данных по оси `frost` вес **4** (в UI можно отображать как «frost ×4» или эквивалент).
- **Две независимые группы:**
  - до **3** шрамов с **наибольшим весом** среди **физических** ключей (`scar_*`);
  - до **3** шрамов с **наибольшим весом** среди **стихийных** ключей (`fire`, `water`, …).
- **Усечение:** если после очередного ремонта в группе получилось больше трёх различных ключей, **остаются только три ключа с максимальным весом**; остальные из этой группы отбрасываются (или не хранятся — по реализации, но правило отбора одно).
- Один и тот же ключ в группе не дублируется: один слот — один id, значение — **вес** (целое число ≥ 0).

Реализация на стороне данных: например `weaponLegacy.scarWeights` как `Record<string, number>` с последующей нормализацией в топ-3 физических и топ-3 стихийных по правилам выше; либо два поля `physicalScars` / `elementalScars` с ограниченным размером. Точная форма — при внедрении в `CraftedWeaponV2` / `WeaponLegacy`, без дублирования с `activeDamageTags`.

### 3.1. Стандартизация материалов для техник починки (под текущий проект)

Таблицы §2–§3 задают **смысл** операции; в коде — `RepairTechniqueDefinition` с **`materials: Record<string, number>`** (`ResourceKey`). Реализация: `src/types/weapon-repair.ts`, `src/data/weapon-damage/repair-techniques-registry.ts`.

**Правила:**

1. Не вводить расходники без записи в экономике; при необходимости — `docs/data/MATERIALS_ADDING.md`.
2. Сохранять **роль** текущих техник (`edge_truing`, `haft_tightening`, …); обновлять `clearsTagIds` под канонические `physical_*` / `elemental_*`, тексты этапов — под колонки «Техника починки».
3. Базовый пул: `ironIngot`, `steelIngot`, `mithrilIngot`, `coal`, `wood`; далее — остальные слитки, `planks`, `stone` / `stoneBlocks`, `soulEssence`, `gold` и т.д. из `ResourceKey`.
4. «Алхимия» в лоре при переносе в реестр раскладывается на конкретные ключи (один канон на игру для спорных случаев).

**Объём данных:** строк в §2–§3 больше, чем техник в реестре сейчас — нужно **расширить** `REPAIR_TECHNIQUE_REGISTRY` (и при необходимости G1 в `damage-tag-registry`): по одной или нескольким техникам на тэг повреждения, с материалами из пула выше.

### 3.2. Стихии и локации

Стихийные теги `elemental_*` не должны попадать на оружие из миссии, если стихия **не разрешена** локацией.

**Данные:**

- На **`Location`** — поле **`presentElements`**: массив коротких id осей (`fire`, `frost`, …), согласованных с биомом.
- **`ExpeditionTemplate.moduleLocationId`** уже ведёт к локации; список для UI: `getLocationById(moduleLocationId)?.presentElements`.
- Опционально **`presentElementsOverride`** на шаблоне экспедиции для сюжетных исключений.

**UI:** строка **«Встречающиеся стихии:»** на карточке экспедиции — из `presentElements` (или override).

#### 3.2.1. Сборщик тегов после миссии (обязательная доработка кода)

Функция **`buildActiveDamageTagsFromMissionSnapshots`** (`src/lib/expedition-post-mission-damage.ts`) должна получать контекст локации, например:

- **`locationId`** активной миссии (тот же, что у экспедиции / `ExpeditionTemplate.moduleLocationId`), и/или
- готовый **`presentElements: string[]`** (предпочтительно резолвить один раз выше по стеку).

**Алгоритм фильтрации:** для каждого кандидата `tagId` из `EVENT_TEMPLATE_TO_DAMAGE_TAGS`:

- если `tagId` **не** начинается с `elemental_` — оставить как есть (физика);
- если `elemental_*` — оставить **только если** соответствующая **ось стихии** из таблицы §3 входит в `presentElements` локации (нужна мапа **`elemental_*` → ось**, например `elemental_frost_bite` → `frost`).

Иначе тэг **не добавлять** в `activeDamageTags` (событие могло отработать в нарративе, но без стихийного повреждения клинка — по продуктовому решению; либо не выпускать такие события из пула локации на уровне генератора — совместимо с фильтром).

Вызов из **`guild-expedition-cross-slice`** при завершении экспедиции должен передавать `locationId` / `presentElements` в сборщик.

### 3.3. Случайные события: контент под все стихии и физику по локациям

Текущего набора шаблонов и строк в **`event-template-to-damage-tags`** недостаточно, чтобы покрыть **все** оси из §2–§3. Нужна целевая доработка модуля экспедиций.

**Стихии:** добавить (или переработать) события так, чтобы для **каждой** оси из колонки «Шрам» §3 существовал хотя бы один шаблон с `damage_weapon` и маппингом на соответствующий `elemental_*`, пригодный для локаций с этой стихией в `presentElements`. Раскидать по **биомам**; ориентир по тирам и лору см. §3.5 и §3.8.

**Физика:** для **каждого** `physical_*` из §2 — хотя бы одно событие, которое может выдать этот тэг, и **привязка к уместным локациям** (обвал/удар — удар и трещина; звери — вырыв металла и прокол; болото не обязано выдавать «удар по обуху», если нет контента — подобрать нарратив). Цель: игрок видит разнообразие повреждений в зависимости от места, а не один и тот же набор везде.

**Дисциплина данных:** каждая такая строка в `EVENT_TEMPLATE_TO_DAMAGE_TAGS` согласована с **`presentElements`** (для стихий) и с пулом событий локации (для физики). Дубли «общих» событий без локации минимизировать; фильтр §3.2.1 остаётся страховкой.

**Код / данные:** `src/modules/expeditions/data/events/`, реестры событий, `event-template-to-damage-tags.ts`; при необходимости — типы шаблонов событий и валидация «тэг ↔ локация».

### 3.4. Экспедиции: отказ от выбора вариантов в случайных событиях и лог событий

Кузнец **не участвует** в экспедиции лично; в бою и при срабатывании случайных событий действует **искатель** с оружием игрока. Соответственно **игрок не выбирает ветки** в случайных событиях миссии — исходы определяются **автоматически** (модель искателя: статы, черты, скрытый бросок, таблица последствий — детали в отдельном ТЗ по модулю экспедиций). Убрать UI выбора в событиях с `choices`, если такие есть для миссий гильдии.

**Снимки и урон по оружию:** ветка `choices` **детерминирована** тем же сидом, что превью лута (`expeditionStartedAtMs` миссии + порядок события). Сборщик тегов [`buildActiveDamageTagsFromMissionSnapshots`](../../src/lib/expedition-post-mission-damage.ts) использует **итоговые** эффекты ветки (см. `getEffectsForEventResolution`); отдельное поле «выбранная ветка» в снимке не требуется. Ранее: только основные `effects`; это снято для соответствия авторазрешению.

**Лог случайных событий** (таймлайн миссии): текущий вид слишком **разноцветный** и **тяжеловесный**. Нужен редизайн: **одна колонка** хронологии, **лаконичная** типографика, меньше конкурирующих акцентов, единая нейтральная основа + дискретное выделение типа события, компактные строки, без «толстых» карточек на каждое событие.

**Авторазрешение событий:** пока достаточно **текущего детерминизма** (сид/модуль); скрытые броски и расширенная модель искателя — возможная доработка позже.

**Лор и теги:** если текст события описывает повреждение клинка, нужно **подтянуть существующий** тэг из §2–§3 и эффект `damage_weapon` в данных; **новые типы повреждений не вводить** без обновления канонических таблиц.

Связанные области кода (ориентиры): компоненты лога экспедиции / `ExpeditionEventLog`, хост событий `expedition-module-events-host`, UI активной экспедиции в гильдии.

### 3.7. Тесты (Vitest)

Файл-ориентир: `src/lib/expedition-post-mission-damage.test.ts` (рядом с `expedition-post-mission-damage.ts`).

### 3.8. Тиры локаций и редкие стихии (по лору текущих данных)

- **`space`, `blood`, `skverna`** — опора на **tier 4**: `depths_of_the_world` (пространство, глубина, скверна), `dragon_scars` (кровь арены, bloodstone в ресурсах, драконья бойня).
- **Ранняя экзотика (tier 3):** по лору имеет смысл частично вводить **`arcane`**, **`nature`**, **`darkness`** раньше эндгейма — прежде всего **`whispering_forest`** (магия, древа, тени/шёпот); **`ash_wastes`** и **`frost_iron_ridge`** усиливают **`fire`** / **`frost`** / **`air`** без ожидания tier 4.
- **Редкие стихии без отдельной «локации-святилища»:** для осей вроде **`light`** (и иных узких сцен) **допускается** появление только через **редкие события** (низкий вес в пуле, сюжетные крючки) в уже существующих локациях. В **будущем** можно добавить выделенную локацию или усилить `presentElements`; до этого момента канон — «редкое событие», а не обязательность постоянной стихии на карте.
- Точный список `presentElements` по каждой локации — отдельная правка `Location` в коде; эта таблица — ориентир для контента.

## 4. Примечание по внедрению в код

- Реестр видимых тегов (`damage-tag-registry`) — канонические id из §2–§3.

- Материалы техник — §3.1.

- Контент событий и маппинг по локациям — §3.3.
- UX экспедиций: автоматическое разрешение событий без выбора игрока, редизайн лога — §3.4.
- Чеклист покрытия, словарь стихий, тесты — §3.5–§3.7; тиры — §3.8.
- Заказы: `hiddenTags` на оружии **не** смешивать с шрамами.
