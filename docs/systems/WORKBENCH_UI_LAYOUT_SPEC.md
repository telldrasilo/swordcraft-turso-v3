# Спецификация: раскладка UI верстака (редизайн по макету)

Документ фиксирует целевую структуру экрана **«Верстак»** (`forgeMainTab === 'bench'`) по пользовательскому макету: три колонки в среднем ряду, компактный инвентарь слева, перенос ремонта/перековки вправо, очередь снизу, удаление лишних блоков. Реализация — изменения в `WorkbenchScreen`, связанных компонентах и при необходимости в store (`forgeBenchSubTab`).

**Источник:** макет со скриншота (аннотации 1–4, пункты 1–6 в ТЗ). Текущее состояние кода на момент спецификации: `workbench-screen.tsx` (подвкладки Ремонт/Перековка сверху + сетка «список + детали | очередь»), `repair-section.tsx` (карточка кузнеца с бонусами), `forge-screen.tsx` (две иконки у вкладки «Верстак»).

---

## 1. Целевая сетка (desktop, `lg+`)

Сверху вниз:

| Уровень | Содержимое |
|--------|-------------|
| **A** | Общая полоса фильтров инвентаря — **на всю ширину** контента верстака (как единая шапка над сеткой). |
| **B** | Три колонки в одном ряду (`grid`), выравнивание `items-start`, вертикальный скролл у колонок 1 и 3 при переполнении. |
| **C** | Блок **очереди верстака** на полную ширину: сегментированная полоса (`WorkbenchQueueSegmentedBar` + легенда) и/или **пустое состояние с подсказкой** при отсутствии сегментов — по той же логике, что **`showWorkbenchBar`** сейчас; затем **`WorkbenchQueuePanel`**. |

### 1.1. Колонка 1 — компактный список оружия (аннотация **2**)

- **Источник данных:** тот же набор клинков, что и на верстаке сейчас: `weaponInventory.weapons` после применения **тех же фильтров**, что и у `WorkbenchWeaponList` (`useInventoryFilter`), чтобы список оставался согласованным с инвентарём/верстаком.
- **Представление (desktop):** вертикальный прокручиваемый список **уменьшенных ячеек** (новый компонент, условно `WorkbenchWeaponStripCell` / полоса в `WorkbenchCompactWeaponRail`).
- **Содержимое мини-ячейки (текущая реализация v1):** метрики и тип (см. `workbench-compact-weapon-rail.tsx`).
  - **Мощь:** `powerScore`, при отсутствии — fallback на `stats.attack` (как в `inventory-section`).
  - **Прочность:** `currentDurability ?? stats.durability` и `stats.maxDurability` (компактно, напр. `64/65` и/или ультра-короткий бар).
  - **Душа войны:** **`warSoul`** (ДВ), подпись согласовать с существующими копирайтами.
  - **Тип оружия:** **`CraftedWeaponV2.type`** + человекочитаемая подпись через реестр (`WEAPON_NAMES` и т.п.).
- **Целевой облик v2** (структурированная мини-карточка: имя, полоса прочности, бейджи, тултип, визуал качества) — **§11**.
- **Выбор:** клик / тап по ячейке выставляет **`workbenchSelectedWeaponId`** и обновляет центральную карточку.
- **Первый заход:** **не** выбирать клинок автоматически — центральная колонка остаётся в **пустом состоянии**, пока пользователь не нажмёт на ячейку (§6).
- **Ширина колонки:** доля полосы — **≈ четверть** условной ширины колонки с полной карточкой (колонка 2).
  - **Рекомендуемая сетка (desktop):** `grid-template-columns: 1fr 4fr 8fr` при общем `min-w-0` у колонок. Отношение **1 : 4** между колонкой 1 и 2; колонка 3 — **8fr** (2× относительно средней).
  - **Нижняя граница читаемости:** для колонки 1 задать **`min-width`** порядка **`4.25rem`–`5rem`** на узких вьюпортах; при конфликте приоритет — читаемость цифр и типа.
  - **Верхняя граница:** колонка 1 — **узкая боковая полоса**, не сравнимая по ширине с колонкой 2.

### 1.2. Колонка 2 — полная карточка выбранного клинка (центр макета)

- **Содержимое:** **`WeaponInventoryCard`** с `context="workbench"` при выбранном `workbenchSelectedWeaponId`; `workbenchMode` синхронизировать с активной вкладкой ремонт/перековка в колонке 3.
- **Пустое состояние:** пока пользователь **ни разу не выбрал** клинок в компактном списке — **заглушка** («Выберите клинок в списке слева» / согласованный копирайт). **Автовыбора первого элемента нет.**
- **Ширина:** ориентир `minmax(17.5rem, 1fr)` внутри доли **`4fr`** общей сетки §1.1.

### 1.3. Колонка 3 — ремонт и перековка на вкладках (аннотация **1**)

- **Содержимое:** контейнер **Tabs** «Ремонт» / «Перековка»:
  - **Ремонт** → **`RepairSection`** (без карточки мастера — §2.2);
  - **Перековка** → **`ReforgeSection`**.
- **Ширина:** в сетке **`1fr 4fr 8fr`** — колонка **8fr** (≈ 2× колонки 2).
- **Store:** переключение вкладок → **`setForgeMainTab('repair')` / `setForgeMainTab('reforge')`** (синхрон **`forgeBenchSubTab`**). Верхнего дублирующего переключателя нет (§2.1).
- **Туториалы:** атрибуты **`data-tutorial`** перенести с удаляёмых кнопок на **вкладки** этой колонки: например **`workbench-repair-subtab`** на триггер «Ремонт», **`reforge-tab`** на «Перековка» (§6, `tutorial.ts`).

### 1.4. Нижний блок — очередь (п. **5**)

- **`WorkbenchQueuePanel`** и **сегментированная полоса** — под средним рядом, **на всю ширину**.
- **Пустая очередь / полоса не показывается:** сохранить текущее поведение **`showWorkbenchBar`**: **не скрывать** зону целиком без подсказки — показывать **пустое состояние с подсказкой** (как сейчас принято для отсутствия активной полосы), плюс панель очереди в привычном виде.

## 3. Поведение на мобильных (`md` и меньше`)

Один **вертикальный стек** (порядок как логика desktop):

1. Фильтры на всю ширину.
2. **Компактный список в видимости «карусели»:** **горизонтальный** прокручиваемый ряд ячеек (`overflow-x-auto`, по желанию **`scroll-snap`**). Те же поля, что §1.1 (мощь, прочность, ДВ, тип). Клик по ячейке — выбор `workbenchSelectedWeaponId`.
3. Полная карточка выбранного клинка; при отсутствии выбора — пустое состояние (§1.2).
4. Вкладки «Ремонт» / «Перековка» с тем же контентом.
5. Очередь внизу (полоса + панель, пустые состояния §1.4).

**Переключателя «Оружие / Очередь» нет** — поле **`workbenchUiMode`** удаляется из store (§6, фаза 2).

**a11y (минимум):** фокусируемый контейнер карусели, прокрутка с клавиатуры; отдельные стрелки/индикаторы — **опционально** позже, не блокер релиза макета.

## 4. Зависимости от store и типов

| Элемент | Зависимость |
|--------|-------------|
| Компактные ячейки | `weaponInventory.weapons`, `useInventoryFilter`, `CraftedWeaponV2` (`powerScore`, `stats`, `warSoul`, `type`, `recipeId`) |
| Выбор клинка | `workbenchSelectedWeaponId`, `setWorkbenchSelectedWeaponId` |
| Режим ремонт/перековка | `forgeBenchSubTab`, `setForgeMainTab('repair'|'reforge')` |
| Очередь | `useWorkbenchQueueRuntime`, `showWorkbenchBar`, `WorkbenchQueuePanel`, `WorkbenchShell` / рефактор размещения |

После редизайна **`workbenchUiMode`** и **`setWorkbenchUiMode`** **не** используются (удаление из `craft-slice` и связанных мест, если grep чист).

## 5. Критерии приёмки (чеклист)

1. Desktop: **`1fr 4fr 8fr`**, компактная колонка узкая; центр — карточка только после явного выбора.
2. Мини-ячейка v1: **мощь, прочность, ДВ, тип**; **v2** — см. §11 (имя + полоса + бейджи).
3. Вкладка «Верстак» в шапке кузницы — **одна** иконка.
4. Карточки мастера в **`RepairSection`** **нет**.
5. Очередь **внизу** на всю ширину; при «пустой» полосе — **подсказка**, согласованная с **`showWorkbenchBar`**.
6. Нет верхнего ряда «Ремонт/Перековка»; переключение — только вкладки колонки 3 + синхронизация **`forgeBenchSubTab`**.
7. `navigateToForgeTab('repair'|'reforge')` открывает верстак с нужной вкладкой.
8. Мобилка: **вертикальный стек** + **горизонтальная карусель** компактных ячеек; **нет** `workbenchUiMode` в UI и store.
9. **`data-tutorial`**: цели для шагов туториала существуют на вкладках колонки 3 и при необходимости на контейнере компактного списка; **`tutorial.ts`** обновлён под новые селекторы.

## 6. Зафиксированные продуктовые решения

| Тема | Решение |
|------|---------|
| **Первый заход** | Центральная карточка **пустая**, пока пользователь не нажмёт на ячейку в компактном списке. |
| **Мини-ячейка** | **v1:** цифры + тип (§1.1). **v2:** трёхрядная карточка + бейджи + тултип (§11). |
| **Мобилка — компактный список** | **Горизонтальная карусель** (скролл; при необходимости `scroll-snap`). |
| **`workbenchUiMode`** | **Удалить** из `craft-slice`, `game-store-composed` (`resetGame` и др.), **`workbench-screen`** и любых упоминаний после grep по репозиторию. В **persist** поле и так не входит — миграция persist не требуется. |
| **Туториалы** | Перенести **`data-tutorial`** с удаляемых узлов на **вкладки колонки 3** и при необходимости на **контейнер компактного списка**; синхронизировать **`src/data/tutorial.ts`** (`target` селекторы). |
| **Пустая полоса очереди** | **Пустое состояние с подсказкой**, как по логике **`showWorkbenchBar`** (не прятать блок без объяснения). |

## 11. План: редизайн мини-карточки в колонке 1 (v2)

Раздел фиксирует **следующий итерацию** левой колонки верстака: визуальная иерархия, компактность (**ориентир** ширины **~9–10rem / 140–160px**, высота **~80–100px** на ячейку), без растровой графики — текст, эмодзи из уже принятого набора, CSS. Идеи взяты из продуктового черновика (скрин/описание UX); ниже — что **напрямую стыкуется с кодовой базой SwordCraft**, а что нужно **уточнить или упростить**.

### 11.1. Цели UX

1. **Чёткая сетка:** фиксированная минимальная ширина ячейки, стабильные отступы, три логические зоны (шапка / прочность / бейджи).
2. **Приоритет информации:** сразу видно **имя** (сокращение визуально), **мощь**, **прочность** (полоса + цифры), затем **состояние** (очередь, повреждения, эффекты из данных).
3. **Согласованность с игрой:** те же сигналы, что на `WeaponInventoryCard` и в превью очереди (качество, повреждения, очередь), без отдельной «второй правды».

### 11.2. Макет строк (целевая структура)

| Зона | Содержимое |
|------|------------|
| **Верх** | Слева **`WeaponIcon`** (`forge-utils.tsx` — уже тип → эмодзи **⚔️🗡️…**). По центру **одна строка имени:** `fullName` с `truncate` и `title={fullName}` на всё имя (**не** жёсткая обрезка «10 символов» — на кириллице и с emoji длина в символах обманывает). Справа **мощь** крупно (`tabular-nums`, `font-semibold` / `text-base` по сетке). |
| **Середина** | **Полоска прочности** (ширина = доля `current/max`), рядом или под полоской текст **`64/65`**. Цвет полосы: **единая шкала с проектом** — сейчас на карточке пороги **>50% зелёный / >25% жёлтый / иначе красный** (`weapon-inventory-card.tsx`). Если хотим пороги **70% / 30%** как в черновике — вынести в **один хелпер** (напр. `durabilityBarTone(percent)`) и использовать и в карточке, и в полоске верстака, чтобы не расходилось. |
| **Низ** | До **3** визуальных маркеров + **`+N`**. Источники только из данных (см. §11.3). |

**Выбор:** по-прежнему клик по ячейке → `workbenchSelectedWeaponId`; фокус/кольцо выбранного — как сейчас (amber ring).

### 11.3. Что из черновика хорошо ложится в проект

| Идея | Как привязать к коду |
|------|----------------------|
| Эмодзи типа | **`WeaponIcon`** — не дублировать таблицу тип→emoji; при появлении новых `weapon.type` расширять `iconMap` там же. |
| ⌛ в очереди | **`weaponHasActiveWorkbenchWork`** / та же логика, что у компактных индикаторов в `WeaponInventoryCard`. |
| ⚠️ повреждено | **`getActiveDamageTagLabels`** или фактия **`activeDamageTags`**; при необходимости «порог прочности < max» как на карточке (`needsRepairShortcut`). |
| Рамка / фон по «редкости» | **`qualityGrade`** + те же оттенки, что **`iconBgByQuality`** / акцент рамки у превью очереди — визуальная линия кузницы. |
| Тултип на десктопе | Radix **`Tooltip`**; содержимое — компактный блок полей (**имя, тип, мощь, прочность, ДВ, теги**), по возможности **вынести общий фрагмент** с `WeaponQueuePreviewCard` или мини-компонент `WeaponBenchStripTooltip`, чтобы не копировать разметку. |
| Моноширинные цифры | **`tabular-nums`** уже используется; полный `font-mono` — опционально и только для строк с цифрами. |

### 11.4. Что доработать / упростить относительно черновика

1. **Бейджи «🔥 огонь / ❄️ лёд / …»** — в игре вред и эффекты описаны **реестрами** (`damage-tag-registry`, другое). **Не** вводить произвольный набор emoji без маппинга из id. **v2 минимум:** показывать **⚠️** (есть теги урона или низкая прочность), **⌛** (очередь), при необходимости **1–2** символа из **нормализованных подписей** `getActiveDamageTagLabels` (текст/аббревиатура) или **фаза v2.1:** таблица `damageTagId → emoji` в **одном** модуле рядом с UI (согласовать с `docs/data` при расширении).
2. **«✨ зачаровано»** — нужен **однозначный критерий** в типах/данных (наличие перековки / слотов / баффов). Пока критерия нет в одном поле — **опустить в v2** или отдельная подзадача после аудита `CraftedWeaponV2`.
3. **Душа войны (ДВ)** — в §1.1 для v1 ДВ была в ячейке; в трёхрядном макете места мало. **Рекомендация:** **ДВ в тултипе** и/или **микротекст** одной цифрой в углу шапки (`·12` рядом с мощью), если продукт настаивает на видимости без hover.
4. **Доп. фильтры / поиск / виртуализация** над колонкой — **вне v2 ячейки:** сортировка уже в **`inventory-filter-slice`** / `InventoryFilterBar` в шапке верстака; поле поиска и virtual scroll при **>50** клинков — **отдельные задачи**, чтобы не раздувать скоуп.
5. **Долгое нажатие на мобилке** под тултип — низкий приоритет: **тап уже выбирает** и открывает центральную карточку. При необходимости позже: `Sheet` с тем же контентом, что тултип.

### 11.5. Влияние на сетку колонки 1

Текущие **`minmax(4.25rem,1fr)`** и карусель **`min-w-[4.5rem]`** уже **уже**, чем целевые **140–160px**. В рамках v2:

- задать ячейке **`min-w-[9rem]`**–**`max-w-[10rem]`** (или эквивалент в `rem`);
- **поднять `min-width` первой колонки** grid в `workbench-screen.tsx`, чтобы текст имени и мощь не ломали вёрстку;
- пересмотреть **доли `1fr 4fr 8fr`** после UX-пробы (допустимо слегка сузить центр при широких экранах).
