# Постройка алтаря (фаза 2 модуля зачарований) — план внедрения по фазам

**Канон спецификации:** [ENCHANTMENT_MODULE_PHASE2_ALTAR_CONSTRUCTION.md](./ENCHANTMENT_MODULE_PHASE2_ALTAR_CONSTRUCTION.md) — этапы по квесту, материалы §4.2–4.3, флаги store, P1/P2 крафта, интендант.

**Назначение этого файла:** разбить внедрение на **последовательные фазы** с критериями готовности, типичными затрагиваемыми зонами кода и местом для **worklog**. Числа баланса и точный список стадий — в данных и в канон-доке; здесь — порядок работ и зависимости.

**Связанные документы:** [FORGOTTEN_FORGE.md](../Quests/FORGOTTEN_FORGE.md), [ENCHANTMENT_MODULE_PHASE1.md](./ENCHANTMENT_MODULE_PHASE1.md), [GUILD_INTENDANT_AND_SPECIAL_QUESTS_PLAN.md](./GUILD_INTENDANT_AND_SPECIAL_QUESTS_PLAN.md), [FORGE_SYSTEM.md](./FORGE_SYSTEM.md), [docs/03_STATE_MANAGEMENT.md](../03_STATE_MANAGEMENT.md).

### Принятые решения продукта

| Решение | Выбор | Следствие для внедрения |
|--------|--------|-------------------------|
| Канон §4.1 | **Вариант B** | Не заводить три отдельных `materialId` артефактов в `materialStash`. Постройка опирается на **`altarUnlockedByForgottenForgeQuest`** + квест; стадии списывают **§4.2–4.3**; имена резонатора/чаши/камертона остаются в лоре и UI прогресса. |
| Канон §5.1 | **P1** | Один **`recipeId`** в форме **`WeaponRecipe`** с фиктивными частями узла (например `blueprint_assembly`, `fastening`, `soul_wiring`), **`generateCraftStages`**, при завершении — без **`CraftedWeaponV2`**, только **`setAltarBuiltInForge(true)`**. P2 (отдельный мета-проект в store) **не** используется, пока P1 не упрётся в инварианты. |

### Правила

1. **Не дублировать** таблицы материалов из канона — ссылаться на §4.2–4.3 phase2-дока.
2. Закрывая подзадачу, затрагивающую **persist / облако / новые поля store**, обновить **`src/lib/cloud-save-feature.ts`** и при необходимости **[ENCHANTMENT_MODULE_DEVELOPER_BRIEF.md](./ENCHANTMENT_MODULE_DEVELOPER_BRIEF.md)** в том же PR или следующим.

4. Worklog ведётся **сверху вниз** в разделе [Worklog](#worklog): новые записи — **в начало** блока.

## Фаза 1 — Рецепт постройки (данные)

**Цель:** завести **рецепт P1** сборки алтаря с макрофазами I → II → III (эпилог), в рамках **§4.1 вариант B** (без артефактов в stash).

| Подзадача | Описание | Канон |
|-----------|----------|--------|
| 1.1 | **Уже принято:** вариант **B** — без трёх отдельных `materialId` артефактов; при смене продуктового решения — обновить канон §4.1 и эту таблицу. | §4.1 |
| 1.2 | **`recipeId`** и **`WeaponRecipe`** с фиктивными частями узла под **`generateCraftStages`** (P1). Задел P2 не делать до явной блокировки P1. | §5.1 |
| 1.3 | Таблица **стадий** по §5.2: для каждой фазы — список материалов (§4.3 для I–II; **§4.2 + при необходимости опционалы** для III). Количества — баланс. | §4.2–4.3, §5.2 |
| 1.4 | Проверка **`inventory-check`** / переплавки: игрок может реально собрать затраты на tier 1–2 без новых «цепочек» вне канона. | §4.3 |

**Критерий завершения фазы:** данные согласованы с каноном; рецепт читается в коде (импорт из `src/data/` или выбранного реестра); нет обязательных новых `materialId` для реагентов §4.2.

## Фаза 2 — Интеграция с крафтом v2 (без выдачи оружия)

**Цель:** пройти стадии **generateCraftStages** / планировщик → списание **`materialStash`** → по успеху — **`setAltarBuiltInForge(true)`**, без создания **`CraftedWeaponV2`**.

| Подзадача | Описание | Канон |
|-----------|----------|--------|
| 2.1 | Развести **ветку завершения**: окончание «проекта алтаря» не добавляет оружие в инвентарь; вызывает только флаг и при необходимости очистку временного состояния крафта. | §5.1 |
| 2.2 | Подключить **экспертизу материалов** и прогноз по тем же правилам, что крафт оружия (см. `use-craft-v2`, `FORMULAS.md` §6.13), если это входит в выбранный UX. | §5.2 |
| 2.3 | Юнит-тесты на **чистые функции**: проверка стадий, списания материалов по плану, отсутствие дублирования квестовой «очистки чаши» в кузнице (логика квеста не трогается). | §2.3 |

**Критерий:** тесты зелёные; после завершения сборки `altarBuiltInForge === true` и экран зачарований ведёт себя по канону §3.

**Цель:** закрыть гейты стадий **сначала** существующими техниками из §6.2 канона; при нехватке — новые id по §6.3 с **вторым применением**; продажа за репутацию — только через расширение интенданта по §7.

| Подзадача | Описание | Канон |
|-----------|----------|--------|
| 4.1 | Маппинг стадий → **`basic_forging`**, **`folded_steel`**, **`spirit_blessing`** и т.д.; не смешивать с ремонтом/перековкой. | §6.2–6.3 |
| 4.2 | Если нужны новые техники крафта: завести id, **не** дублировать kind интенданта для ремонта; при продаже — **`craft_technique`**, **`unlockedCraftTechniqueIds`**, `purchaseIntendantOffer`, persist + облако. | §6.3, §7 |

**Критерий:** стадии проходимы без софтлока; интендант расширен только если реально продаются техники крафта.
