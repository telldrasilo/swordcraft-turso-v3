# Модуль зачарований — фаза 0: фундамент и зачистка legacy

Связанные документы:

- [ENCHANTMENT_AWAKENING_CONCEPT.md](ENCHANTMENT_AWAKENING_CONCEPT.md) — канон системы (пробуждения, древо, слияние, план фаз 0–5).
- [ENCHANTMENT_MODULE_DEVELOPER_BRIEF.md](ENCHANTMENT_MODULE_DEVELOPER_BRIEF.md) — краткий контракт для разработки UI и данных.

## 3. Этап 0.2 — удаление или изоляция legacy

**Задача:** убрать из пользовательского пути то, что противоречит [ENCHANTMENT_AWAKENING_CONCEPT.md](ENCHANTMENT_AWAKENING_CONCEPT.md), либо явно пометить как временный shim.

### 3.1. Принципы

1. **Не ломать сборку и тесты** — каждый коммит проходит `npm run lint`, `npm run test`, `npm run type-check` (или эквивалент CI).
2. **Удаление крупными блоками** с удалением мёртвых импортов предпочтительнее долгого «полумёртвого» кода.
3. Если функционал ещё нужен для отладки — вынести за **флаг разработчика** или закомментировать с ссылкой на задачу, а не оставлять полуподключённым UI.

### 3.2. Типичные действия

- Удалить маршрут/страницу старого алтаря или заменить на заглушку с текстом «модуль в разработке» (если навигация должна остаться).
- Убрать `altar` из `GameScreen` или перенаправить на новый экран-заглушку.
- Сократить surface API store: неэкспортируемые внутренние хелперы; публичные методы — только после ревью (см. brief).

### 3.3. Облако и персистентность

- При изменении полей сейва: обновить **`src/lib/cloud-save-feature.ts`** (чеклист), **`save-payload-schema`** и тесты схемы.
- **Миграция старых сейвов не делается** — допустим сброс локального состояния разработчика; задокументировать в worklog при смене схемы.

**Критерий готовности:** приложение собирается; старый сценарий «алтарь как сейчас в проде» либо удалён, либо явно изолирован; навигация не ведёт в 404.

## 4. Этап 0.3 — черновая схема данных

**Задача:** зафиксировать в типах минимальную структуру под будущее древо, не реализуя всю игровую логику.

### 4.1. Минимальный набор полей (ориентир)

Документ канона описывает: `awakeningLevel` 0…3, шаги древа, ветвления после 2-го/3-го пробуждения, эссенция/осколки на уровне игрока. На фазе 0 достаточно:

| Сущность | Поля (черновик) | Примечание |
|----------|-----------------|------------|
| Экземпляр оружия | `awakeningLevel`, массив шагов (`branchId`, `globalStepIndex`, `perkId`, ранг перка) | Пустые массивы до первого пробуждения |
| Игрок / склад | задел под `soulEssence`, типы осколков | Могут быть 0 и не использоваться до фазы 2 |
| Флаги ветвления | `secondBranchMode`, `thirdBranchMode` | Enum в `src/types` |

Имена полей привести к одному стилю с `docs/04_TYPES_SYSTEM.md`; при добавлении — **обновить** этот файл документации типов.

### 4.2. Нормализация сейва

- Если есть `normalize` / `migrate` в `save-craft-normalize` или аналогах — добавить **только** приведение к новой форме (без сложной миграции с прод-данных).
- Unit-тест на «загрузка старого JSON → нормализация → валидная структура» — по мере появления полей.

**Критерий готовности:** типы компилируются; новые поля не ломают существующие тесты; схема save обновлена там, где персистятся новые ключи.

## 6. Правила разработки (модуля зачарований)

1. **Канон** — [ENCHANTMENT_AWAKENING_CONCEPT.md](ENCHANTMENT_AWAKENING_CONCEPT.md); расхождения только после явного обновления документа.
2. **Типы** — сначала `src/types/` и `docs/04_TYPES_SYSTEM.md`, затем store и UI.
3. **Минимальные диффы** — без рефакторинга несвязанных систем; один PR/ветка — одна крупная тема (например «удаление экрана алтаря»).
4. **Тесты** — Vitest, `src/**/*.test.ts`; для чистой логики — unit-тесты рядом с модулем; не раздувать интеграционные тесты всего store без необходимости.
5. **CI** — перед пушем: `npm run lint`, `npm run test`, `npm run type-check` (как в [AGENTS.md](../../AGENTS.md)).
6. **Сохранения** — любое новое поле: чеклист `cloud-save-feature.ts`, схема payload, при необходимости Prisma/API; **без** обязательной миграции с legacy-сейвов до релиза.
7. **Документация** — при смене публичного контракта обновлять [ENCHANTMENT_MODULE_DEVELOPER_BRIEF.md](ENCHANTMENT_MODULE_DEVELOPER_BRIEF.md) и строку в worklog.
8. **Legacy** — удалённый код не оставлять закомментированным «на всякий случай» без ссылки на issue и даты удаления.
