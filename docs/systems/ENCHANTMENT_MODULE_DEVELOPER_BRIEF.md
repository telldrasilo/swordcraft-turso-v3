# SwordCraft — автономный бриф модуля зачарований (новая база)

Документ **самодостаточен**: внешние файлы проекта не требуются для понимания контекста. Модуль зачарований может разрабатываться отдельно; при интеграции в игру хост передаёт описанные ниже данные и события.

**Граница ответственности:** здесь зафиксированы концепция SwordCraft, игровая петля, канон **осей урона**, **шрамов**, **души войны** и **потенциала души** — опора для нового зачарования. Старый экран «Алтарь» (жертвоприношения, валюта эссенции в текущем виде, каталог «школ») **не** является источником правды и будет заменён.

---

## 1. Продукт и стек (кратко)

**SwordCraft** — браузерная игра-симулятор кузнеца: крафт оружия, ресурсы и рабочие, гильдия с экспедициями, заказы NPC, ремонт оружия после миссий. Зачарование — слой над **памятью клинка** (шрамы по физике и стихиям) и **душой войны**, а не отдельная мини-игра про валюту алтаря.

Типичный стек хоста: Next.js (App Router), React, TypeScript, Zustand, Tailwind, компоненты UI; сохранения в браузере и опционально в облаке. Для автономного модуля достаточно контракта данных (JSON) и событий домена.

## 2. Игровая петля (макро)

| Контур | Суть |
|--------|------|
| **Ресурсы и рабочие** | Добыча, здания, стамина, производство материалов. |
| **Кузница** | Сборка оружия из частей (клинок, гарда, рукоять, навершие), техники обработки, расчёт характеристик, при необходимости поля **слотов/силы зачарования** на экземпляре. |
| **Гильдия и экспедиции** | Искатель ходит в миссию с оружием игрока; случайные события могут нанести **урон по оружию**; в конце миссии на экземпляр записываются **видимые тэги повреждений**. |
| **Ремонт** | В кузнеце с видимых тэгов снимают конкретную поломку техникой починки (расход материалов). |
| **Платформа осей и шрамов** | После **успешного** снятия видимого тэга накапливается **шрам** (память клинка): отдельно по **физике** (`scar_*`) и по **стихийным осям** (`fire`, `water`, …). |
| **Заказы NPC** | Требования к оружию; служебные теги для подбора заказа — **не** то же самое, что шрамы (см. §10). |
| **Зачарования (новый модуль)** | Правила наложения/усиления магии на экземпляр с опорой на **веса шрамов**, **оси**, **warSoul**, **soulPotential** и при необходимости расчётные слоты/силу с крафта. |

Крафт создаёт экземпляр оружия; экспедиции и ремонт меняют видимые тэги и шрамы. Модуль зачарований читает память клинка и прогресс души.

## 3. Цели модуля зачарований

1. Второй контур прогрессии поверх кузницы — через оси, совместимость и стоимость, завязанные на **историю** клинка (шрамы), а не на абстрактный магазин.
2. Усиление готового оружия без полного перекрафта.
3. Согласованность с экспедициями (какие стихии и повреждения реально встречались) и с экономикой ремонта.
4. Единый канон **осей стихий** из §7 — не смешивать с устаревшими «школами» старого каталога без явного маппинга в дизайне.

---

## 4. Цепочка памяти клинка

Одна логическая цепочка для любой строки таблиц §6–§7:

1. В миссии оружие получает **видимое повреждение** — строковый тэг `physical_*` или `elemental_*`. Источник — события миссии, в которых сработал урон по оружию; итоговый набор тэгов фиксируется **при завершении** экспедиции из снимков событий (семантически тэг привязан к прошедшему событию).
2. Игрок в кузнеце выполняет ремонт и **снимает** соответствующий видимый тэг (подходящая техника и материалы).
3. После **успешной** починки в память клинка добавляется **шрам**: для физики — id вида `scar_*`; для стихии — **короткий id оси** (`fire`, `frost`, …).
4. Видимые тэги и шрамы — разные слои: тэги — «что сломано сейчас»; шрамы — «что клинок пережил и помнит» для зачарований и весов.

**Термин «шрам»** — только накопительная память. **Не путать** со служебными тегами заказов на экземпляре (§10).

---

## 5. Стихии на локации и фильтр тэгов

У каждой **локации** миссии задаётся список **встречающихся стихий** `presentElements: string[]` — короткие id осей из §7 (`fire`, `water`, …).

**Правило фильтрации при сборке итоговых повреждений после миссии:**

- Для каждого кандидата в видимый тэг:
  - если тэг **не** начинается с `elemental_` — оставить (физика не фильтруется по стихиям локации);
  - если тэг `elemental_*` — оставить **только если** ось стихии для этого тэга (маппинг «тэг → ось» совпадает с колонкой §7) **входит** в `presentElements` данной локации;
  - иначе тэг **не** добавлять на оружие (событие могло быть в нарративе, но без стихийного повреждения клинка — продуктовое решение).

Так игрок не накапливает, например, морозный след на клинке в локации без мороза в `presentElements`.

## 6. Механика шрамов (канон)

- Шрамы — **не** активные поломки, а **накопительная память** экземпляра.
- **Вес:** каждый раз, когда успешно снят видимый тэг, соответствующий шрам получает **+1** к весу по своему ключу.
- **Две независимые группы:**
  - до **3** ключей с **наибольшим весом** среди **физических** шрамов (ключи `scar_*`);
  - до **3** ключей с **наибольшим весом** среди **стихийных** шрамов (ключи — id осей: `fire`, `water`, …).
- **Усечение:** в каждой группе хранятся только три ключа с максимальным весом; четвёртый и далее отбрасываются. Один ключ не дублируется: одна запись — один id, значение — целое **вес** ≥ 0.

**Представление данных (целевое):** например `scarWeights: Record<string, number>` с последующей нормализацией в топ-3 физических и топ-3 стихийных; либо два отображения `physicalScars` / `elementalScars` ограниченного размера. Дублировать с массивом активных видимых тэгов не нужно.

Модуль зачарований использует веса и ключи для доступности, стоимости или силы эффектов по осям и типам шрамов.

## 9. Душа войны и потенциал души

### 9.1. Назначение

**Душа войны** — накопленный на экземпляре оружия показатель боевого опыта клинка; растёт за успешные экспедиции (и другие источники, если появятся в дизайне). От неё зависят **тиры**, дающие бонусы к успеху миссии, золоту, приросту души и шансу крита.

**Потенциал души (`soulPotential`)** — множитель **эффективности** получения души войны за миссию: насколько хорошо сборка материалов и техник «проводит» душу. Целевой диапазон множителя: **от 1.00 до 3.00**. Считается при крафте из частей клинка, синергий материалов и техник; если поля нет у старого сохранения — считать **1.0**.

Целевая продуктовая линия: **не** опираться в UX на огромную «ёмкость пула» души как главную метрику; ключевую понятность даёт **потенциал** и текущий **тир**; ёмкость пула при наличии в коде хоста может оставаться техническим полем.

### 9.2. Формула потенциала (структура)

```text
rawPotential = basePotential + materialScore + synergyScore + techniqueScore
soulPotential = clamp(rawPotential, SOUL_POTENTIAL_MIN, SOUL_POTENTIAL_MAX)
```

Рекомендуемые стартовые константы:

- `basePotential = 1.00`
- `SOUL_POTENTIAL_MIN = 0.85`
- `SOUL_POTENTIAL_MAX = 3.00`
- `SYNERGY_CAP = +0.60` — потолок суммарного вклада синергий до clamp

**materialScore:** по частям оружия `blade`, `guard`, `grip`, `pommel`:

```text
partContribution = normalizeSoul(material.weaponEffects.soulCapacity) * partWeight
materialScore = sum(partContribution) * MATERIAL_SCORE_SCALE
```

- `normalizeSoul(...)` — нормализация `soulCapacity` материала в диапазон 0…1 по каталогу материалов хоста.
- Веса частей (стартовый ориентир): клинок **0.40**, гарда **0.25**, рукоять **0.20**, навершие **0.15**.
- `MATERIAL_SCORE_SCALE` подбирается так, чтобы материальный вклад давал в сумме порядка **+0.90** к rawPotential (калибровка баланса).

**techniqueScore (простой вариант):** сумма бонусов проводимости техник × коэффициент; ограничить диапазоном **−0.10 … +0.30**.

**synergyScore:** сумма бонусов сработавших правил синергии (см. §9.4), затем ограничение до **`SYNERGY_CAP`** по сумме.

### 9.3. Награда души за экспедицию

После базовой награды души войны из калькулятора миссии:

```text
effectiveWarSoul = floor(baseWarSoul * soulPotential * tierMult * critMult)
```

- `baseWarSoul` — базовая награда из логики экспедиции;
- `tierMult = 1 + warSoulBonus / 100`, где `warSoulBonus` — процент бонуса **текущего тира** души войны;
- `critMult` — множитель крита при успешном крите (как в текущей логике хоста, напр. ×1.5).

### 9.4. Синергии материалов (правила в данных)

Синергии задаются **данными**, не жёстким кодом ветвлений. Формат правила:

```text
WarSoulSynergyRule:
  id: string
  name: string
  requirements: {
    blade?: string[]
    guard?: string[]
    grip?: string[]
    pommel?: string[]
    categoriesAll?: ('metal' | 'alloy' | 'wood' | 'leather' | 'stone')[]
    atLeastLegendaryParts?: number
  }
  bonus: number   // аддитивно к rawPotential
  note: string
```

Приоритет проверки: **точные id материалов** → **категории** → **fallback по части**. Может сработать несколько правил; итоговый `synergyScore` режется `SYNERGY_CAP`.

### 9.6. Связь с модулем зачарований

- **warSoul** и **тир** — мера **зрелости** клинка;
- **soulPotential** — мера **качества сборки** как магического резонатора;
- правила зачарований (стоимость, сила эффекта, открытие слотов) логично вешать на комбинацию **шрамов + душа/потенциал**, а не на отдельную валюту старого алтаря.

---

## 10. Заказы NPC и служебные теги

На экземпляре оружия могут быть **служебные теги для подбора заказа** (например скрытые признаки редкости или свойств для матчинга NPC). Это **отдельный** механизм от шрамов: шрамы — память клинка после ремонтов; служебные теги — контракты контента заказов. Модуль зачарований не должен смешивать эти массивы в одну семантику.

---

## 11. Контракт входных данных для модуля зачарований

Ниже — логическая модель того, что хост игры **может передавать** автономному модулю (или что модуль читает из состояния). Имена полей — ориентир; при интеграции допускается адаптер.

### 11.1. Экземпляр оружия (минимум для зачарований)

| Поле | Тип / смысл |
|------|-------------|
| `id` | Уникальный id экземпляра |
| `quality` | Числовое качество крафта |
| `tier` | Редкость материала (число или строка common…mythic) |
| `warSoul` | Накопленная душа войны |
| `warSoulTier` | Вычисляемый тир 0…10 по §9.5 |
| `soulPotential` | Множитель 0.85…3.0 (или 1.0 по умолчанию) |
| `activeDamageTags` | Массив строк — **видимые** повреждения до ремонта |
| `scarWeights` или пара `physicalScars` / `elementalScars` | Накопленные шрамы после топ-3 нормализации; ключи из §6–§8 |
| `stats.enchantSlots`, `stats.enchantPower` | Опционально: ограничения с крафта |
| `enchantments` | Текущие наложенные зачарования (если модуль их дополняет) |

### 11.2. Игрок и экономика (гейты)

| Поле | Смысл |
|------|--------|
| `player.level`, `player.fame` | Требования доступа к чарам |
| Ресурсы (золото, душа, реагенты) | По дизайну нового модуля — не фиксировать здесь конкретные валюты |

### 11.3. Контекст миссии

| Поле | Смысл |
|------|--------|
| `presentElements` | Какие стихии разрешены на локации; влияет на то, какие стихийные тэги могли появиться на клинке (§5) |

### 11.4. События домена (для реактивной логики)

- «Экспедиция завершена» — обновились видимые тэги.
- «Ремонт успешен» — снят тэг X → увеличен вес шрама Y, нормализация топ-3.
- «Зачарование наложено» — изменена запись в `enchantments`.

### 11.5. Перековка (фаза 1, хост SwordCraft)

Контракт см. [`ENCHANTMENT_MODULE_PHASE1.md`](ENCHANTMENT_MODULE_PHASE1.md) §3. Кратко для интеграции:

| Поле / поведение | Смысл |
|------------------|--------|
| `weaponReforge` на экземпляре | Состояние баффов и пробуждения шрамов |
| `buffStat` | Случайная прибавка % в диапазоне из реестра; фиксированная стоимость `warSoul` за применение |
| `awakenScar` | Списание **всей** текущей `warSoul` на попытку; шанс по формуле данных; при успехе — один шрам помечен, **`scarAwakeningCompleted`** = true (фаза 1 — не более одного успеха на экземпляр) |
| Верстак | Тот же `repairBenchWeaponId`, что и ремонт |
| Разблокировка строк реестра | `minGuildLevel` (гильдия); опционально `sourceCraftTechniqueId` — техника обработки материалов доступна, если `unlockedByDefault` в её данных **или** id входит в эффективный список планировщика от **`player.level`** и **`unlockedMaterialProcessingTechniqueIds`** (как в крафте v2), см. [`planner-unlocked-techniques.ts`](../../src/lib/craft/planner-unlocked-techniques.ts) |
| Гейт экрана «Зачарования» | К перековке **не** применяется (перековка в кузнице независима) |

## 12. Legacy (не использовать как финал)

Старый UX алтаря, жертвоприношение за эссенцию в прежнем виде, каталог «школ» с шестью фиксированными школами — **наследие**. Новый модуль может полностью заменить данные и экраны. На этапе разработки **миграция сейвов не обязательна** — допустим сброс при смене схемы; к релизу при необходимости отдельная задача.

## 13. Чеклист автономной разработки

1. Реализовать доменную модель зачарований, опираясь на **оси §8** и **шрамы §6–§7**, без смешения со служебными тегами заказов (§10).
2. Зафиксировать правила: как ось/вес шрама открывает или усиливает эффект; при необходимости маппинг «старый контент → новые оси».
3. Подключить **warSoul**, **тир** (§9.5) и **soulPotential** (§9.2–9.3) к стоимости и силе эффектов.
4. Определить хранение шрамов (§6) и события после ремонта (§11.4).
5. Спроектировать API/JSON между хостом и модулем по §11.
6. После стабилизации контракта — интеграционные тесты на граничные случаи (нет стихии на локации → нет стихийного шрама; топ-3 усечение; отсутствие `soulPotential` → 1.0).
