- Доступность локаций по уровню гильдии: `getAvailableLocations(guildLevel)` в `src/modules/expeditions/data/locations/index.ts`:

## 3. Дизайн: вкладка «Интендант»

### 3.1 Валюта

**Решение:** покупки у интенданта оплачиваются только **очками репутации текущего ранга** (в коде: `guild.reputation` — прогресс в ранге, не накопительный total; см. `guild-reputation-tier.ts`). Золото игрока (`resources.gold`) для других систем (в т.ч. глобальный магазин ресурсов, экономика вне гильдии) и **не** используется как валюта витрины интенданта.

**Реализация:** в данных каждой позиции каталога — целочисленная стоимость в «очках репутации» (или списание из пула репутации по правилам баланса: см. существующую прогрессию уровня гильдии). Единая точка списания в store, например `spendGuildReputationForIntendant(amount)` с проверкой достаточности и границ ассортимента по `guild.level`.

**UX:** в шапке вкладки «Интендант» отображать текущую репутацию и (при необходимости) прогресс до следующего ранга; при нехватке — понятное сообщение, без предложения «докинуть золотом».

### 3.2 Ассортимент и ранг

- Каталог строк: объединить:
  - рецепты оружия (из `recipe-shop`, исключая позиции, остающиеся бесплатными по §4.3);
  - рецепты переработки из кузницы (так как вкладка кузницы исчезает — иначе покупка потеряется);
  - **идентификаторы техник починки** (`repair_technique`, спец-набор из реестра) — **сделано**;
  - **идентификаторы техник перековки** — **сделано** (`reforge_technique`).
- Привязка к рангу: минимальный `guild.level` (и при необходимости порог репутации) на каждую строку каталога. Единый источник правды: `src/data/guild/intendant-catalog.ts` (новый файл) или расширение существующего модуля данных.

### 3.3 UI

- Новая вкладка в `GuildScreen.tsx` между `adventurers` и `stats` (сейчас сетка `grid-cols-4` — станет **5 колонок** или **горизонтальный скролл** на мобильных).
- Компоненты: `IntendantSection` / карточки товара; фильтры: рецепты оружия, переработка, техники ремонта, **перековка**.
- Действия store: либо расширить существующие (`unlockRecipe`, множества разблокировок техник), либо ввести `purchaseIntendantOffer(offerId)` с внутренним маппингом.

### 3.4 Персистентность и облако

- Любые новые поля: `src/store/game-store-composed.ts` (частичное состояние), `STORE_VERSION`, `src/lib/cloud-save-feature.ts`, Prisma/маршрут `src/app/api/save/route.ts` при включённом облаке.
- Миграция для старых сейвов: игроки, купившие рецепты в кузнице, должны **сохранить** разблокировки (`unlockedRecipes` уже хранит факты покупки).

## 4. Перенос контента из кузницы

### 4.1 Рецепты оружия: «кроме меча»

Нужно **зафиксировать определение**:

- **Строгий вариант:** бесплатно только семейство мечей (`basic_sword`, и, при желании, покупные формы `long_sword` остаются в интенданте / наградах). Все **остальные формы** (топор, кинжал, …) только через интендант или другие источники (заказ, экспедиция).
- **Мягкий вариант:** бесплатен только `basic_sword`; остальные `basic_*` переносятся в интендант с низкой ценой на низком ранге.

Текущий `initialUnlockedRecipes` необходимо **пересмотреть** в пользу выбранного варианта, иначе G4 не выполняется.

### 4.2 Рецепты переработки

`refiningRecipePrices` сейчас в том же `ShopSection`. После удаления вкладки кузницы их нужно **либо** перенести в интенданта, **либо** в глобальный экран «Магазин» (ресурсы). Для целостности «всё гильдейское знание в одном месте» предпочтительно **интендант** (отдельная подвкладка «Переработка»).

### 4.3 Техники починки

Реализовано: §5 + `intendant-catalog` (`repair_technique`), store `unlockedRepairTechniqueIds`, v19.

### 4.4 Техники перековки *(реализовано)*

- Базовые (`reforgeTier: 'basic'`) — две техники баффа без покупки.
- Спец (`reforgeTier: 'specialized'`) — покупка у интенданта **или** прежняя разблокировка через `sourceCraftTechniqueId` + планировщик кузни.
- `isReforgeTechniqueUnlocked`, `unlockedReforgeTechniqueIds`, UI и интендант — см. код и таблицу статуса.

## 5. Базовые vs узкоспециализированные техники починки

### 5.1 Концепция

| Класс | Доступность | Эффект | Шрамы / последствия |
|--------|-------------|--------|----------------------|
| **Базовые** | Старт игры или низкий ранг гильдии без покупки | Снятие тегов повреждений с **меньшей** эффективностью, дольше этапы / выше расход | При «успешном» снятии тега — **вероятность** начислить шрам памяти клинка (или иной негативный след по канону повреждений) **&lt; 100%**; при провале — тег частично остаётся |
| **Узкоспециализированные** (текущий реестр как ориентир) | Покупка у интенданта | Полноценное снятие по текущим правилам при успехе | Как сейчас (если сейчас 100% снятие тега при успехе — сохранить для этого класса) |

*Уточнение по коду (v19):* различие «базовые vs спец» реализовано как **доступность** (интендант), **фильтр UI/автоподбора** и **вероятностное начисление шрамов** при плане только из базовых техник; отдельная настройка длительности этапов/кубиков только для базовых в данных **не введена** (при необходимости — последующий баланс-проход).

### 5.2 Изменения данных *(факт реализации)*

- Один реестр `REPAIR_TECHNIQUE_REGISTRY` + поле **`repairTier`**: `'basic' | 'specialized'` (см. `src/types/weapon-repair.ts`).
- Разбивка на два файла не потребовалась. Отдельные флаги `scarOnSuccessProbability` на технике не вводились: для базового **плана целиком** шрамы по снятым тегам проходят фильтр вероятности (см. §5.3).

### 5.3 Изменения логики *(факт реализации)*

- Снятие тегов и наследие: `repair-cross-slice.ts` → при успехе, если план **только из базовых** техник (`repairPlanUsesOnlyBasicTechniques`), в `incrementScarWeightsFromClearedTags` попадает **подмножество** снятых тегов (берётся по одному броску на тег с шансом `REPAIR_BASIC_SCAR_ON_SUCCESS_CHANCE`); иначе — все снятые теги (как раньше для «тяжёлого» ремонта со спец-техниками).
- **Юнит-тесты:** `build-repair-plan.test.ts` (unlock / фильтр / only-basic); полный сценарий шрамов с моком RNG можно добавить отдельно при регрессиях.

### 5.4 UX *(факт реализации)*

- В `repair-card.tsx`: бейджи «Базовая» / «Спец.» / «интендант» для заблокированных спец; автоподбор техник использует только разблокированные спец.

Многоступенчатое задание на открытие алтаря должно:

- Иметь **чёткие шаги** (лор/геймдизайн: сбор квестовых предметов, успешные вылазки в tier-2 локации, постройка — когда будет модель базы).
- Отображаться в UI гильдии рядом с обычными экспедициями, но не смешивать списки без пометок.

### 7.2 UI: две субвкладки в «Экспедиции» *(реализовано в коде)*

- В `ExpeditionsSection`: `Tabs` — **«Доступные экспедиции»** / **«Особые задания»**; во второй — `ForgottenForgeQuestCard` и связка с выбором миссий (подсказка при активном квесте).
- Расширение списка «особых заданий» (несколько цепочек, общий прогресс store) — при необходимости отдельной задачей.

**Пересечение с «Доступные экспедиции»:** шаг квеста может требовать **отправить экспедицию** в конкретную локацию или с особым флагом; реализация — либо фильтр шаблонов экспедиций под квест, либо отдельный `questStep.startExpeditionTemplateId`. Важно переиспользовать **тот же** поток старта миссии (`startExpeditionFull` / аналог), чтобы не дублировать экономику и расчёт.

### 7.3 Данные и store

- Новые сущности (черновик):
  - `GuildSpecialQuestDefinition` — id, шаги, условия разблокировки (`minGuildLevel`, флаг «tier2 доступен»), награды.
  - `GuildSpecialQuestProgress` — `questId`, `currentStepIndex`, `completed`, `flags` (например собранные предметы).
- Размещение: `src/data/guild/special-quests.ts`, состояние в `guild-slice` или отдельный slice с селекторами.

### 7.4 Триггер при tier-2

- При первом выполнении условия «доступны локации tier 2» — автоматически выдать квест **или** показать его в списке (идемпотентно, один раз на сейв).
- Согласовать с `getAvailableLocations` и `canAccessEnchantmentAltarScreen`, чтобы не было ситуации «алтарь открыт в UI, а квест ещё не выдан».

### 7.5 Тексты для копирайта

Литературная проработка многоступенчатого квеста на алтарь и базовых техник починки вынесена в отдельное ТЗ (самодостаточный файл для автора): [COPYWRITING_TZ_ALTAR_QUEST_AND_BASIC_REPAIR.md](COPYWRITING_TZ_ALTAR_QUEST_AND_BASIC_REPAIR.md).
