## 1. Цели фазы 1

### 1.1. Экран «Зачарования» (навигация)

Игрок **не получает доступ к полноценному модулю** (эссенция, древо, миссии пробуждения древа) до выполнения **трёх условий**:

1. **Гильдия / локации 2-го тира** — как в концепции ([ENCHANTMENT_AWAKENING_CONCEPT.md](ENCHANTMENT_AWAKENING_CONCEPT.md), часть VII): реализация через `getAvailableLocations` / `canAccessEnchantmentAltarScreen`. Пока нет tier-2 — экран в режиме **замка** с CTA в гильдию.
2. **Сюжет: чертёж и артефакты** — завершение квеста «Эхо забытой кузни» (`altarUnlockedByForgottenForgeQuest`). Пока tier-2 уже есть, но квест не пройден — отдельное состояние с CTA к особым заданиям в гильдии.
3. **Узел собран в кузнице** — `altarBuiltInForge` (постройка на вкладке «Алтарь»; см. фазу 2). Пока квест завершён, но узел не собран — экран меню ведёт в кузницу на вкладку «Алтарь», без плейсхолдера полного модуля.

Плейсхолдер контента (древо/эссенция в разработке) показывается только когда **все три** условия выполнены.

### 1.2. Перековка (кузница)

Вводится ранняя механика **перековки** над готовым экземпляром: расход **души войны** с оружия, выбор **техники перековки** из отдельного реестра, постоянный эффект на экземпляр. **Эссенции нет**; **древо перков** из миссий пробуждения — не фаза 1.

**Важно:** перековка **не привязана** к гейту экрана «Зачарования». Она доступна **всегда**, когда выполнены условия по оружию, душе и разблокировке техник (независимо от того, открыт ли пункт меню «Зачарования»).

## 3. Спецификация перековки (геймдизайн)

### 3.1. Определение

**Перековка** — действие в кузнице над **готовым** экземпляром: тратит **душу войны** с этого оружия ради **постоянного** улучшения. Не снимает видимые повреждения и не использует эссенцию.

### 3.2. Два равноправных типа (разные техники перековки)

| Тип | Суть | Ресурс | Риск | Лимит на оружие (фаза 1) |
|-----|------|--------|------|---------------------------|
| **Пробуждение шрама** | Активирует один из **непробуждённых** шрамов → постоянный бонус (перк/модификатор по правилам шрама). | Тратится **вся** текущая душа войны на экземпляре (до 0 или до порога тира — решает баланс). | **Шанс неудачи:** душа списана, шрам не пробуждён. | **Один успешный** пробуждение на экземпляр за всё время; неудачи не расходуют «слот», успех — блокирует повтор техник пробуждения на этом оружии. |
| **Усиление атаки или прочности** | Постоянный прирост к атаке или к **максимальной** прочности (случайное значение в диапазоне). | **Фиксированная** стоимость в warSoul (лестница по номеру применения). | Нет провала; есть **разброс** прироста. | Многократно, пока хватает души и не достигнут **потолок** улучшений (число применений и/или суммарный бонус — баланс). |

Обе механики в коде — **отдельные техники перековки** с полем `reforgeType: 'awakenScar' | 'buffStat'` (или эквивалент).

### 3.3. Пробуждение шрама — логика

**Условия:**

- На оружии есть хотя бы один **непробуждённый** шрам.
- `warSoul > 0`.
- Нет флага успешного пробуждения перековки на этом экземпляре (`hasAwakenedScar` / аналог).
- Техника разблокирована.

**Поток:**

1. Игрок выбирает технику с `reforgeType === 'awakenScar'`.
2. **Потенциальная стоимость** для UI шанса = все текущие очки души до списания.
3. Показать **шанс успеха** (зависит от тратимых очков, тира души, опционально `soulPotential`; формула — данные, не жёстко в UI).
4. По подтверждению: списать душу по правилам баланса → бросок успеха → при успехе: один случайный непробуждённый шрам помечается пробуждённым, на оружие вешается **постоянный** бонус по таблице «шрам → эффект»; при неудаче — шрамы без изменения, душа уже потрачена.
5. После **успеха** — флаг на экземпляре; **остальные** шрамы остаются непробуждёнными в фазе 1 (расширение на несколько пробуждений — фазы 2–3).

**Не путать:** «пробуждение шрама» в перековке ≠ **миссии пробуждения** древа (часть I канона) — разные системы; в UI и типах имена должны различаться.

### 3.4. Усиление статов — логика

**Условия:**

- `warSoul` ≥ стоимости выбранной ступени техники.
- Техника разблокирована.
- Не исчерпан лимит улучшений (счётчик/потолок — данные).

**Поток:** списать фиксированную душу → случайная прибавка в заданном диапазоне к атаке **или** к max прочности → сохранить на экземпляре. Повторять до лимита.

### 3.5. Общие правила

- После траты души **пересчитать тир души** по таблице порогов.
- Кулдаунов реального времени **нет** — только ресурс и лимиты.
- Перековка **доступна при любых видимых повреждениях** (решение геймдизайна: не блокировать из‑за тегов ремонта). Рекомендуемый порядок в UI: сначала **Ремонт**, затем вкладка **Перековка**.
- Блокировки применения: нет шрамов (для awaken), не хватает души, техника залочена, достигнут лимит (для buff), уже успешное пробуждение шрама (для awaken).

## 4. Реестр техник перековки (гибридная модель)

- Отдельный реестр в данных: **не** смешивать с `REPAIR_TECHNIQUE_REGISTRY` и с массивом техник **крафта** как с одной таблицей.
- Пример полей записи: `id`, `name`, `reforgeType`, условия разблокировки (гильдия, изученная техника крафта `sourceCraftTechniqueId`, рецепт экспедиции и т.д.), стоимость/лестница warSoul, диапазоны эффектов (для buff), нарративные стадии (для UI), ссылка на иконку.
- Примеры id и структуры (контент может расширяться):

| ID (пример) | Тип | Идея разблокировки |
|-------------|-----|---------------------|
| `reforged_001` | Пробуждение шрама | Гильдия + техника крафта (закалка) |
| `reforged_002` | Усиление прочности | Техника крафта «закалка в масле», лестница стоимости |
| `reforged_003` | Усиление атаки | Гильдия + правка кромки |
| `reforged_004` | Пробуждение шрама (альт.) | Рецепт tier 2 + другая закалка |
| `reforged_005` | Усиление атаки (дорого) | Гильдия + дамасский узор |

Полный перечень строк копирайта и ключей — в геймдизайн-пакете; ориентир ключей: `ui.reforge.*`, `ui.enchantment.gate.*`, `ui.enchantment.placeholder.*`.

## 5. Этапы реализации (инженерные)

### 5.1. Этап 1.1 — гейт экрана «Зачарования»

**Условия (последовательно):**

1. **Tier-2:** нет доступа к локациям **2-го тира** — `canAccessEnchantmentAltarScreen(guildLevel)` (`getAvailableLocations`, порог в `locations/index.ts`).
2. **Квест:** tier-2 уже есть, но не завершён сюжетный квест — `altarUnlockedByForgottenForgeQuest === false` (квест «Эхо забытой кузни», см. [FORGOTTEN_FORGE.md](../Quests/FORGOTTEN_FORGE.md)).
3. **Узел:** квест завершён, но `altarBuiltInForge === false` — CTA в кузницу на вкладку «Алтарь» (сборка по фазе 2).

**UI:**

1. **Замок (нет tier-2):** «Алтарь зачарований закрыт», уровень гильдии, CTA «В гильдию».
2. **Замок (tier-2 есть, квест не завершён):** отдельный текст и CTA к особым заданиям / гильдии.
3. **Квест есть, узел не собран:** «Соберите узел в кузнице», CTA на вкладку «Алтарь».
4. **Tier-2 + квест + `altarBuiltInForge`, фаза 2 контента нет:** заголовок зачарований, текст что древо и эссенция в разработке, CTA «В кузницу — перековка»; перековка доступна во вкладке кузницы независимо от этого экрана.

**Задачи:** хелперы `canAccessEnchantmentAltarScreen`, `canUseEnchantmentAltarContent` + тесты; `AltarScreen` под четыре состояния (замок tier / замок квест / сборка узла / плейсхолдер).

### 5.2. Этап 1.2 — вкладка «Перековка» и логика

1. **Типы и сейв:** поля экземпляра под шрам пробуждения перековки, счётчики buff, модификаторы; `save-payload-schema`, `cloud-save-feature.ts`, `04_TYPES_SYSTEM.md`.

5. **UI:** вкладка **Перековка** в `forge-screen.tsx`, `ReforgeSection`, список техник, состояния (замок техники, мало души, шанс, лимит, тосты успех/провал).
6. **Тесты:** краевые случаи (0 души, нет шрамов, повтор после успеха, лестница buff).

### 5.3. Этап 1.3 — прогрессия и контент

### 11.2. Термины и границы (таблица терминов концепции)

| Требование концепции | Проверка |
|------------------------|----------|
| Пре-зачарование = **перековка** на вкладке **«Перековка»** после «Ремонт», без эссенции и без древа | Реализовано: `ForgeMainTab === 'reforge'`, нет эссенции в потоке перековки |
| Перековка **не зависит** от гейта пункта меню «Зачарования» | Реализовано: доступ по верстаку и душе/данным, без проверки `canAccessEnchantmentAltarScreen` |
| **Пробуждение шрама** (перековка) ≠ **миссия пробуждения** древа | В коде: `scarAwakeningCompleted` / `awakenScar`; древо — `awakeningLevel` / шаги дерева (фаза 2+) |
| Гейт экрана «Зачарования» (меню) | **Необходимо:** tier-2 (`canAccessEnchantmentAltarScreen`), квест (`altarUnlockedByForgottenForgeQuest`), узел (`altarBuiltInForge`). Перековка по-прежнему **не** зависит от этого гейта. |
| До алтаря эссенции нет; древо только после первой **миссии** пробуждения (другая система) | Перековка не начисляет эссенцию и не открывает древо |

### 11.3. Расхождения, требующие внимания

- В концепции §3.3 и **часть VII** местами исторически фигурировало «пре-зачарование в ремонте» — каноническое размещение **вкладка «Перековка»**; формулировки в [ENCHANTMENT_AWAKENING_CONCEPT.md](ENCHANTMENT_AWAKENING_CONCEPT.md) приведены к этой схеме.
- **Часть VII** формулирует закрытие модуля до tier-2; **второй гейт** (квест) добавлен в продукте для сюжетного открытия пункта меню — см. §1.1 и §5.1 этого файла; концепция обновлена в части VII и части VIII.
- Полная **таблица эффектов по шрамам** после пробуждения в перековке (бой, UI) — следующий шаг после платформы шрамов (§10 п.3); генерация перков древа по шрамам — в концепции (части II, IV–VI).
