# Канонический domain contract системы зачарований

## Назначение

Этот документ фиксирует доменную модель, на которую должна опираться внешняя разработка. Если код в отдельной среде разойдётся с этим документом, обратная интеграция в SwordCraft станет дорогой и рискованной.

## Бизнес-цели системы

Система зачарований решает четыре задачи:

1. Даёт игроку второй контур прогрессии поверх кузницы.
2. Превращает ненужное оружие в полезный магический ресурс.
3. Позволяет усиливать уже созданные предметы без полного перекрафта.
4. Создаёт будущую точку интеграции между крафтом, экспедициями, заказами и экономикой.

## Канонические сущности

### Enchantment

Источник: `src/data/enchantments.ts`

Обязательные поля:

- `id: string`
- `school: EnchantmentSchool`
- `tier: 1 | 2 | 3`
- `name: string`
- `description: string`
- `effect.type`
- `effect.value`
- `effect.duration?`
- `cost.soulEssence`
- `cost.gold`
- `requiredLevel`
- `requiredFame`
- `unlocked`

Смысл:

- определяет доступную игроку магическую модификацию;
- содержит все требования разблокировки;
- содержит только данные домена, а не runtime-поля конкретного оружия.

### EnchantmentSchool

Канонический набор на текущий момент:

- `fire`
- `frost`
- `life`
- `protection`
- `power`
- `speed`

Эти значения нужно считать каноническими до отдельного пересмотра дизайна.

### WeaponEnchantment

Источник: `src/types/shared/enchantment.ts`

Обязательные поля:

- `id: string`
- `enchantmentId: string`
- `appliedAt: number`

Смысл:

- это не definition чара, а runtime-запись о наложении чара на конкретное оружие;
- хранится внутри оружия, а не отдельно.

### SacrificeResult

Обязательные поля:

- `soulEssence: number`
- `bonusGold: number`

Смысл:

- единый результат жертвоприношения;
- используется UI, store и persistence-контуром только как вычисляемый runtime-результат;
- не хранится в save как отдельная сущность.

## Базовые инварианты системы

Эти правила должны быть неизменны в любой внешней реализации, если они не пересматриваются отдельно:

1. Зачарование применяется только к существующему оружию в инвентаре.
2. Наложить можно только уже разблокированное зачарование.
3. На оружии не должно быть дубля одного и того же зачарования.
4. На оружии не должно быть двух чар одной школы.
5. Жертвоприношение удаляет оружие из инвентаря.
6. Разблокировка должна проверять ресурсы и требования игрока.
7. Наложение чара должно обновлять статистику применения чар.
8. Жертвоприношение должно обновлять статистику пожертвованного оружия.

## Канонические правила совместимости

### Правило текущего канона

До отдельного redesign-а совместимость определяется просто:

- две чары одной школы на одном оружии запрещены.

Это правило соответствует текущему UI и `src/data/enchantments.ts`.

### Что пока не входит в канон

Логика через `incompatibleMap` из `src/lib/store-utils/enchantment-utils.ts` не должна считаться обязательным контрактом, пока она не станет реальной частью store и UI.

## Канонические правила лимитов

### Текущий канонический лимит

Для автономной разработки считать каноническим:

- `MAX_ENCHANTMENTS_PER_WEAPON = 2`

Причина:

- это значение используется в текущем UI;
- это значение импортируется боевой логикой из `src/data/enchantments.ts`.

### Будущее расширение

Поля `stats.enchantSlots` и `stats.enchantPower` должны рассматриваться как предусмотренные точки расширения, но не как уже действующие ограничения текущей altar-механики.

## Канонические правила разблокировки

Разблокировка чара должна проверять:

- наличие чара по `enchantmentId`;
- отсутствие id в `unlockedEnchantments`;
- `resources.soulEssence >= enchantment.cost.soulEssence`;
- `resources.gold >= enchantment.cost.gold`;
- `player.level >= enchantment.requiredLevel`;
- `player.fame >= enchantment.requiredFame`.

Успешная разблокировка должна:

- списывать `soulEssence`;
- списывать `gold`;
- добавлять `enchantmentId` в `unlockedEnchantments`.

Примечание:

- если текущая реализация проекта этого не делает, это баг текущей реализации, а не норма контракта.

## Канонические правила жертвоприношения

Жертвоприношение принимает оружие и вычисляет:

- количество `soulEssence`;
- бонусное золото.

Минимальные входные поля оружия:

- `quality`
- `tier`
- `warSoul`
- `epicMultiplier`

Успешное жертвоприношение должно:

- начислять `resources.soulEssence`;
- начислять `resources.gold`, если бонус больше нуля;
- удалять оружие из `weaponInventory`;
- увеличивать `statistics.weaponsSacrificed`.

## Канонические правила наложения зачарования

Наложение чара должно проверять:

- существует ли оружие;
- разблокирован ли `enchantmentId`;
- не достигнут ли лимит чар;
- нет ли уже этого `enchantmentId` на оружии;
- не нарушается ли правило совместимости школ.

Успешное наложение должно:

- создать `WeaponEnchantment`;
- добавить его в `weapon.enchantments`;
- увеличить `statistics.enchantmentsApplied`.

## Канонические правила снятия зачарования

Снятие чара должно:

- проверять наличие чара на оружии;
- удалять соответствующую запись из `weapon.enchantments`.

Что не зафиксировано как канон:

- возврат части ресурсов;
- штрафы за снятие;
- очистка оружия целиком;
- перерасчёт цены или боевых статов.

Если такие правила понадобятся, их лучше оформлять как отдельную версию доменного контракта.

## Канонический каталог эффектов

Текущий набор `effect.type`:

- `damage`
- `defense`
- `speed`
- `regen`
- `lifesteal`
- `burn`
- `slow`
- `crit`

Для внешней разработки важно разделять два уровня:

- `effect.type` как доменное описание;
- реальное применение эффекта в UI, экспедициях и карточке оружия как отдельный integration concern.

## Границы domain contract

Этот документ не определяет:

- форму Zustand slice;
- save/load структуру;
- UI layout;
- SQL shape;
- маршруты App Router.

Эти части вынесены в отдельные документы пакета и должны меняться синхронно с доменной моделью.
