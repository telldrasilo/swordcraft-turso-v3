# Технический аудит SwordCraft (swordcraft-turso-v3)

**Дата:** 31 марта 2026  
**Методология:** статический анализ репозитория, `npm run type-check`, `npm run lint`, `npm run build`, выборочный обзор критичных модулей (API сохранений, схема БД, store, крафт v1/v2, генераторы наград). **Обновление:** добавлены разделы по мёртвому коду, заглушкам и дуализму крафта.

## Краткое резюме

Проект функционально богатый (крафт v2, гильдия, экспедиции, облачные сохранения Turso), но **состояние «готовности к релизу» ослаблено** тремя кластерами проблем:

1. **Сборка и типизация:** `tsc --noEmit` фиксирует порядка **876** сообщений об ошибках; **`next build` падает** на этапе «Linting and checking validity of types». Это блокирует предсказуемый production-build без ослабления правил или массового исправления.
2. **Согласованность данных (БД ↔ API ↔ Prisma):** облачное API оперирует полем **`materialKnowledge`**, которого **нет** в DDL в `src/lib/db.ts` и в модели `GameSave` в `prisma/schema.prisma`. Это риск **ошибок SQLite при UPDATE** на «чистой» базе и расхождения с локальной Prisma-схемой.
3. **Безопасность и целостность сохранений:** идентификация игрока по заголовку без настоящей аутентификации; при `DELETE` для `save_history` используется **`playerId` вместо id записи сохранения**, что не соответствует смыслу поля `gameSaveId`.
4. **Две системы крафта и наслойка костылей:** в store и игровом цикле остаётся **ветка v1** (`activeCraft` в craft-slice, `weapon-recipes.ts`, `completeCraftWithExperience`), а в интерфейсе кузницы уже реализована **многоступенчатая линия** (`CraftPlanner` / `useCraftV2`, `@/data/recipes`: рецепт → части → материалы → техники → этапы). **Принятое направление развития:** закрепить эту UI-систему как единственную и **удалить legacy-крафт** (см. §10.5). Пока параллельно живут мёртвые файлы и заглушки — это коррелирует с ошибками типов (два определения рецептов/качества).

Ниже детализация по областям, с указанием приоритета и наблюдаемых симптомов.

## 1. Критические и высокие риски

### 1.1. Схема БД и API сохранений: рассинхрон

**Наблюдение:** маршрут `src/app/api/save/route.ts` читает/пишет колонку `materialKnowledge` (в том числе в `UPDATE` и `formatSaveData`).

**Факт:** в `src/lib/db.ts` таблица `game_saves` создаётся **без** столбца `materialKnowledge`. В `prisma/schema.prisma` у `GameSave` **нет** соответствующего поля.

**Последствия:**

- На новой БД, инициализированной только через `CREATE_TABLES_SQL`, запросы с `materialKnowledge` могут завершаться ошибкой вида «no such column».
- Prisma и «сырой» libSQL живут параллельно: ментальная модель «одна схема» нарушена; миграции Prisma не отражают реальную схему Turso.

**Рекомендация:** единый источник правды для списка колонок `game_saves`: добавить `materialKnowledge` в DDL + Prisma (или отказаться от Prisma для game_saves и явно задокументировать только libSQL-схему); прогнать миграцию/ALTER для существующих баз.

### 1.2. Ошибка в DELETE для `save_history`

Фрагмент логики (упрощённо): после удаления строки из `game_saves` выполняется удаление из `save_history` с условием `gameSaveId = ?`, в качестве аргумента передаётся **`playerId`**.

По `prisma/schema.prisma`, `SaveHistory.gameSaveId` — это внешний ключ на **id** записи `GameSave` (UUID/cuid), а не на `playerId`. Если при записи истории в `gameSaveId` попадает именно id сохранения, **строки истории не удалятся** при сбросе, либо семантика поля в коде заполнения истории должна быть явно иначе (тогда это несоответствие именованию и схеме Prisma).

**Рекомендация:** при удалении сохранения сначала получить `id` строки `game_saves` по `playerId`, затем `DELETE FROM save_history WHERE gameSaveId = ?` с этим `id`; либо изменить схему так, чтобы ключ был однозначным везде.

### 1.3. Безопасность облачного слоя

- Игрок определяется как `request.headers.get('x-player-id') || 'demo-player'`. Любой клиент может подставить чужой идентификатор: **нет привязки к сессии/OAuth/JWT**.
- Комментарий в коде прямо указывает демо-режим без авторизации — для публикации это **критический** пробел.
- Тело сохранения проходит через **`validateSaveData`**, фактически это частичное приведение типов без строгой схемы: возможны неполные/вредоносные полезные нагрузки (захламление JSON-полей, чрезмерные структуры), если не ограничивать размер запроса и глубину объектов.

**Рекомендация:** NextAuth (уже в зависимостях) или другой провайдер; сопоставление `playerId` с субъектом токена; лимиты размера тела; по возможности Zod/структурная валидация полного snapshot.

## 2. Состояние TypeScript и production-сборки

### 2.1. Массовые ошибки компилятора

Запуск `npm run type-check` даёт **876** строк с `error TS...` (файл лога агента). Характер ошибок смешанный:

| Код (примеры) | Интерпретация |
|---------------|----------------|
| **TS6133** | Неиспользуемые переменные/импорты — **~302** вхождения (существенная доля объёма) |
| **TS2322, TS2345, TS2339** | Несовместимость типов, отсутствующие свойства (несогласованность интерфейсов рецептов, ремонта, пропсов) |
| **TS18048** | Возможный `undefined` при строгих проверках |
| **TS2305** | Отсутствующие экспорты (напр. `MaterialNode`, `QualityGrade` в ожидаемых модулях) |

**Вывод:** включённые `noUnusedLocals` / `noUnusedParameters` делают неиспользуемые импорты **ошибками**, а не предупреждениями; репозиторий в текущем виде под это не подчищен.

### 2.2. Файлы-артефакты разработки в области компиляции

В отчёт компилятора попадают файлы вроде `src/components/guild/adventurer-card-v2.backup.tsx` с устаревшими типами и опечатками в именах полей (`refusalChanceBonus` vs `refuseChanceBonus`). **`tsconfig` не исключает шаблон `*.backup.*`**, только `src/backup` как каталог.

**Рекомендация:** удалить бэкапы из `src/components`, перенести в git-ветку/историю или добавить явный exclude.

### 2.3. Несогласованность доменных типов (примеры)

- Рецепты: в одном месте ожидается `tier: number`, в данных — **`string`** (`WeaponRecipe`) — ломает вызовы утилит.
- Качество (`QualityGrade`): сравнение с литералом `"masterwork"` при типах, с которыми пересечение пустое — признак **двух конкурирующих перечислений/строк**.
- Ремонт: UI опирается на поля результата (`durabilityRestored`, `experienceGained`), которых **нет** в типе `ExecuteRepairResult`; пропсы `RepairCard` не совпадают с использованием.

Это указывает на **не завершённые рефакторинги** между слоями данных, store и UI.

## 3. ESLint и React Compiler

`npm run lint` завершается с ненулевым кодом. Помимо предупреждений (`no-explicit-any`, `no-console` в примерах):

- **Ошибки** `react-hooks/rules-of-hooks` (хуки внутри колбэков) — нарушение правил React.
- Ошибки вида **«Cannot call impure function during render»**, **«Calling setState synchronously within an effect»**, **«Compilation Skipped: Existing memoization could not be preserved»** — характерны для **React Compiler** / строгих правил Next 15; они требуют точечной переработки компонентов.
- `no-require-imports` в коде под `docs/Random_gen` и др.
- `react/no-unescaped-entities` в JSX.

**Несимметрия с `tsc`:** в `eslint.config.mjs` каталог **`docs/` не в `ignores`**, поэтому экспериментальная документация и генераторы в `docs/Random_gen` **участвуют в общем lint** и раздувают отчёт. В `tsconfig.json` папка `docs` **в exclude**, поэтому TypeScript их не проверяет.

**Рекомендация:** добавить `docs/**` в `ignores` ESLint (или вынести прототипы в отдельный пакет/репозиторий); для исходников приложения — планомерно снимать ошибки компилятора React.

## 4. Архитектура и сопровождение

### 4.1. Монолитный composed store

Файл `src/store/game-store-composed.ts` содержит **~1539** строк. Комментарий в шапке описывает слой сборки как «**~300 строк**» — документация и реальность **расходятся**.

Побочные эффекты:

- Сложность code review и высокий риск конфликтов при merge.
- Множественные приведения `as any` при создании слайсов и merge persist — **слабая типобезопасность** в самом центре состояния.

**Рекомендация:** вынести cross-slice операции в именованные модули; сузить типы `StateCreator`/инференс Zustand без `any`; обновить `docs/03_STATE_MANAGEMENT.md` под фактический размер и границы ответственности.

### 4.2. Дублирование и размытые границы `src/` vs корень репозитория

Обнаружены параллельные артефакты:

- `data/knowledge-discoveries.ts` (корень), `docs/Random_gen/data/knowledge-discoveries.ts`, `src/backup/unused-root-files/knowledge-discoveries.ts`.
- `lib/loot-integration.ts` и `types/expedition-loot.types.ts` в **корне** (импорты вида `../types/...`), тогда как основной код живёт в `src/` с алиасом `@/*`.

`tsconfig` включает `**/*.ts` с корня проекта, поэтому дубликаты **могут попадать в общую компиляцию** и путать IDE/импорты.

**Рекомендация:** единая политика: весь исполняемый код в `src/`; корневые `lib/`, `data/`, `types/` — либо удалить, либо исключить из `include` и пометить как архив.

### 4.3. Prisma vs libSQL

В проекте есть Prisma (`sqlite` `file:./dev.db` в схеме) и отдельный клиент `@libsql/client` для Turso. Это **два канала** для похожих сущностей; без чёткой документации «что для чего» легко допустить рассинхрон (как с `materialKnowledge`).

### 4.4. Неполные подсистемы (явно помеченные в коде)

- `src/lib/expedition-reward-generator.ts` — заглушка, TODO по полноценным наградам и интеграции со store.
- `src/lib/expedition-calculator-v2.ts` — TODO по `enemyTypes`, `glory` из контекста и т.д.
- Корневой/документный `loot-integration` — интерфейсы без реализации.

Игровой дизайн может это допускать, но для QA важно отмечать **незавершённые ветки** как известные ограничения.

## 8. Мёртвый код и неиспользуемые экспорты

Поиск по импортам (а не только по экспорту из barrel) показывает, что часть «официальных» компонентов кузницы **нигде не монтируется** в экранах.

| Артефакт | Назначение | Статус |
|----------|------------|--------|
| `src/components/forge/active-craft-card.tsx` | Отображение **v1** `state.activeCraft` + данные из `weapon-recipes.ts` | Экспортируется из `forge/index.ts`, **нет импортов** в `src/` кроме самого себя и barrel — **мёртвый UI**. |
| `src/components/forge/recipe-card.tsx` | Карточка рецепта со **v1** `startCraft(recipe)` из store | **Не импортируется** нигде; в v2-планировщике используется **другой** `RecipeCard` из `craft-v2/planner/RecipeCard.tsx`. |
| `src/components/forge/repair-section.tsx` | Ремонт с привязкой к store (`executeWeaponRepair` и т.д.) | Только экспорт из barrel; **на экране кузницы не используется**. |
| `src/store/slices/craft-v2-slice.ts` | Альтернативная модель состояния крафта v2 | **Не подключается** в `game-store-composed.ts`; фактическое v2-состояние ведётся в **`useCraftV2`** + `craftV2Persisted` в composed store — **дублирующий неиспользуемый слайс**. |

**Итог:** репозиторий несёт полный «скелет» старого UX и запасной слайс, что увеличивает шум при типизации и путает при рефакторинге.

## 9. Заглушки, адаптеры и костыли

### 9.1. Явные заглушки в продуктовом UI

- **`RepairStub`** (`src/components/forge/repair-stub.tsx`) подключён в `forge-screen.tsx` с комментарием «Заглушка ремонта», хотя **`RepairSection`** уже существует и экспортируется — типичный случай «новый экран не дописали, старый не убрали».
- **`ForgeScreen` / `onWeaponCreated`:** в `CraftContainerV2` колбэк по смыслу внешней интеграции фактически сводится к `console.log` и комментарию «Интегрировано с game-store» — **обратная связь для родителя не реализована** (реальное добавление оружия делается внутри `handleCollectWeapon` через `addWeaponV2`).

### 9.2. Адаптеры совместимости

- **`src/data/materials/adapter.ts`** — явный слой «MaterialNode → Material» для **обратной совместимости** с типами `craft-v2`; признак того, что новая иерархия материалов и старый `Material` сосуществуют.
- **`src/data/weapon-recipes.ts`** реэкспортирует типы из `craft-slice`, но определяет **свой** интерфейс `WeaponRecipe` (в т.ч. `tier` как строковый тир), тогда как модульный крафт использует **`@/data/recipes`** и `WeaponRecipe` из **`@/types/craft-v2`** — два несовместимых контракта с одним именем.

### 9.3. Отладочные и сомнительные приёмы

- **`use-craft-v2.ts`** и **`use-cloud-save.ts`** содержат **`fetch` на `http://127.0.0.1:7756/ingest/...`** с телом JSON (помечено как `agent log`). Это **жёстко зашитая телеметрия/отладка** для локальной среды: лишние запросы в браузере у всех пользователей, шум в сети, потенциальная утечка структуры состояния.
- **`forge-screen.tsx`:** `getAvailableRecipes` вызывается через **`require('@/data/recipes')` внутри `useMemo`** — обход нормального ESM-импорта, усложняет статический анализ и бандлинг (костыль под циклические зависимости или быстрый фикс).

### 9.4. Прочие заглушки в логике

- `src/lib/expedition-reward-generator.ts`, корневой/`docs` **`loot-integration`**, фрагменты **`weapon-card.tsx`** помечены как заглушки или будущая логика — ожидаемо для прототипа, но для продукта это **скрытые no-op ветки**.

## 10. Craft v1 и Craft v2: как устроено и почему это запутывает

### 10.1. Два источника правды по рецептам

| Аспект | Craft v1 (legacy в store) | Craft v2 (UI кузницы) |
|--------|---------------------------|------------------------|
| Данные рецептов | `src/data/weapon-recipes.ts` — список вроде `iron_sword`, строковый `tier`, структура под старый калькулятор | `src/data/recipes/index.ts` — модульные рецепты (`basic_sword`, …), типы из `@/types/craft-v2`, части (`parts`), `combatPart` |
| Активный процесс | `craft-slice`: `ActiveCraft` (`progress`, `endTime`, один общий прогресс %) | `useCraftV2`: `ActiveCraftV2` с **поэтапными** `stages`, локальный React state + синк в `craftV2Persisted` |
| Запуск крафта | `startCraft` / `startCraftWithResources` (второй **не вызывается** из UI) | `startCraft()` внутри хука после выбора материалов/техник в `CraftPlanner` |
| Завершение | `completeCraftWithExperience` в store: поиск рецепта в **`weaponRecipes`**, **`createWeapon`**, `addWeapon` | Генерация **`CraftedWeaponV2`** в хуке, затем **`addWeaponV2`** из `CraftContainerV2` |

Игрок на экране кузницы **всегда** проходит по v2; пути v1 в основном **висят в store и `use-game-loop`**.

### 10.2. Игровой цикл всё ещё обслуживает v1

В `src/hooks/use-game-loop.ts` при наличии `activeCraft.recipeId` и `endTime` вызываются `updateCraftProgress` и **`completeCraft()`** (через `completeCraftWithExperience`). Пока ни один экран не выставляет эти поля через `startCraft`, ветка **не срабатывает**, но остаётся **скрытой зависимостью**: при любом будущем вызове `startCraftWithResources` возможны **два параллельных** мира прогресса (таймер в цикле vs стейджи в `useCraftV2`).

### 10.3. Смешение типов в одном интерфейсе

- **`craft-slice`** уже хранит инвентарь как **`CraftedWeaponV2[]`**, но **`ActiveCraft`** остаётся схемой **v1**.
- **`active-craft-card.tsx`** смешивает `activeCraft` (v1), **`weaponRecipes`** (legacy) и тип **`CraftedWeaponV2`** для уведомления о новом предмете — симптом **незавершённого слияния** моделей.

### 10.4. Решение по продукту (целевая модель крафта)

**Единственным источником правды по крафту остаётся та система, что уже отражена в интерфейсе игры:**

- выбор **рецепта** (`CraftPlanner`, `@/data/recipes`, типы `@/types/craft-v2`);
- назначение **материалов по частям оружия** (`PartMaterialSelector`, план материалов);
- выбор **техник** (`TechniqueSelector`);
- **многоступенчатый прогресс** по категориям этапов (`ActiveCraftV2`, `CraftProgress`, `useCraftV2`).

**Старую линию крафта (v1)** — отдельный `ActiveCraft` с одним прогресс-баром, `weapon-recipes.ts` как основной каталог рецептов для крафта, `startCraft` / `completeCraftWithExperience` / `createWeapon` под эту модель, ветку в `use-game-loop`, неиспользуемые UI-компоненты под v1 — нужно **убрать из кодовой базы** после переноса оставшихся зависимостей (заказы, баланс, миграции сохранений) на данные и типы **модульного** крафта.

## 11. Заключение

Проект демонстрирует зрелую **игровую** проработку, но **инженерный контур** (типы, линт, однозначная схема БД, безопасность API) находится в состоянии, **несовместимом с безболезненным production-release**: сборка падает на проверке типов, а облачный слой содержит расхождения схемы и слабую модель доверия к клиенту.

Дополнительно **дуализм крафта v1/v2**, мёртвые компоненты и заглушки (ремонт, колбэки, отладочный `fetch`) создают ощущение «слоя поверх слоя»: правки в одном месте легко не отражаются в другом, что усиливает число ошибок TypeScript и риск регрессий.

Устранение рассинхрона БД и ошибки удаления истории, **приведение крафта к одной многоступенчатой модели (как в текущем UI) с удалением legacy v1** по плану §10.5, чистка мёртвого кода и доведение TypeScript/ESLint до **зелёного** `npm run build` даст измеримый скачок предсказуемости и снизит цену дальнейших фич.
